# Agentic AI Management コンセプト

> 経営管理サイクル（予実管理 - 差異分析 - ネクストアクション定義 - 結果の反映）を AI が並走し、計画と現実の乖離を構造的・必然的に縮小させるプロダクトのコンセプトを定義する。

## 1. 課題定義

### 1.1 表層的な課題

スタートアップ経営の現場では、以下の課題が繰り返し発生している。

- **KPI の未達**: 目標数値に対して実績がショートしているが、現場からは「市況が悪い」「リソース不足」といった定性的な理由しか上がってこない
- **マイクロマネジメントの徒労**: CEO 自らが現場の Slack や Notion を漁り、一次情報を確認しないと実態が掴めない
- **意思決定の遅延**: 正しい判断を下すための「ファクト」が揃うまでに時間がかかり、対策を打つのが手遅れになる

### 1.2 根本的な課題

表層的な課題の背後には、2 つの構造的な問題が存在する。

**原子レベルの分解能力の欠如 (Lack of Atomic Decomposition)**

「売上未達」という事象を、「A-1（商談数）x A-2（受注率）」といった制御可能な変数まで分解できていない。そのため、どこに手を打てば解決するのかが論理的に特定できず、精神論（「頑張る」「気合い」）に逃げている。

**事実と解釈の混同 (Fact/Interpretation Mixing)**

報告プロセスにおいて、都合の悪い事実（Fact）が、報告者の主観や希望的観測（Interpretation）によって無意識に隠蔽・歪曲されている。組織内に「論理的整合性」よりも「その場の空気や人間関係」を優先する力学が働いている。

### 1.3 As-Is / To-Be / Gap

**現状（As-Is）: 人間によるマネジメントの限界**

- 人間は感情、疲労、政治的配慮（忖度）から逃れられない
- 厳しいフィードバック（Bad Cop）を継続することは精神的コストが高く、徐々に規律が緩む
- 認知リソースに限界があり、24 時間 365 日の監視は物理的に不可能

**理想（To-Be）: 計算可能で透明な経営（Management as Code）**

- 経営管理サイクルが、ソフトウェアのようにバグ（曖昧さ）なく回り続ける
- 「誰が言ったか」ではなく「データが何を示しているか」のみに基づく意思決定
- 未達の予兆が発生した瞬間に、要因が原子レベルで特定され、自動的に介入

**ギャップ**

- 人間関係を壊さずに論理的矛盾を突き続けられるシステムの不在
- 非構造データから構造的なファクトをリアルタイム抽出し KPI ツリーにマッピングする技術的仕組みの不在

## 2. 経営管理サイクル（Management Cycle）

### 2.1 定義

経営管理サイクルとは、CEO の意思決定ワークフローの中核をなす 4 工程の循環プロセスである。

```
┌──────────────────────────────────────────────────────────┐
│                 経営管理サイクル                            │
│                                                          │
│    ┌──────────┐     ┌──────────┐     ┌──────────────┐   │
│    │ 予実管理  │────→│ 差異分析  │────→│ ネクストアクション│   │
│    │ (Plan    │     │ (Analyze │     │ 定義 (Act)    │   │
│    │  vs Act) │     │  Gap)    │     │               │   │
│    └────▲─────┘     └──────────┘     └──────┬────────┘   │
│         │                                     │          │
│         └─────────────────────────────────────┘          │
│              結果の反映 (Reflect & Update)                 │
└──────────────────────────────────────────────────────────┘
```

### 2.2 各工程の内容

**予実管理（Plan vs Act）**

KPI の目標設定、実績入力、達成率の自動計算。計画と実績の乖離を定量的に把握する起点。

- KPI ツリーの定義と目標値の設定
- 実績データの取得（手動入力 / API 自動取得）
- 達成率の自動計算と異常検知（閾値超過、トレンド変化、急変）

**差異分析（Analyze Gap）**

計画と実績の乖離の原因を構造的に分析する。v1 の曖昧さ検知はこの工程に組み込まれる。

- 定量差異の自動検知
- 報告テキストの品質担保（曖昧さ 5 パターン検知）
- 因果分析の支援（なぜ乖離が生じたかの構造化）
- 差異の分解: 全体差異 = 市場要因 + 固有要因 + 一時要因

**ネクストアクション定義（Act）**

差異を埋めるための具体的な行動計画を生成する。

- 因果チェーンの末端（根本原因）に対するアクション案の生成
- Who/What/When/完了基準の構造化
- アクション間の優先度付けと依存関係の整理
- 過去の類似アクションの効果参照

**結果の反映（Reflect & Update）**

アクションの効果を測定し、次サイクルの計画にフィードバックする。

- アクション効果測定（予測 vs. 実際のインパクト）
- 因果モデルの精度検証と更新
- 曖昧さ検知の false positive 分析（dismiss 理由の学習）
- 予測モデルの改善

### 2.4 サイクルの駆動タイミング

経営管理サイクルは週次で駆動される。

```
【毎週月曜 朝】KPI 実績データを取得 → 異常検知
    ↓
  [異常あり?] ── No → サマリーのみ出力（平常運転）
    │ Yes
    ↓
【月曜-火曜】 マネージャーの週次報告取り込み → 品質チェック → 修正ループ
    ↓
【水曜】      差異分析（品質担保済みの報告 + KPI データ + 外部データ）
              アクション計画の生成 → Governor 承認
    ↓
【木曜】      統合レポート生成 → 関係者に配信
    ↓
【月末】      前月のアクション効果測定 → 因果モデル更新 → 検知精度改善
```

### 2.5 差異分解の考え方

差異分析において、乖離の原因を以下の 3 要因に分解する。

```
全体差異 = 市場要因 + 固有要因 + 一時要因

例: 売上 -8%
  市場要因: 業界全体 -3%（ai-exec-dashboard からのデータ）
  固有要因: 新規受注の遅延 -4%
  一時要因: 大型案件の期ずれ -1%
```

この分解により、「自社固有の問題」と「市場全体の動き」を切り分け、対策の焦点を明確化する。さらに固有要因は因果チェーンとして深掘りされる。

```
「新規受注の遅延」→ なぜ？
  → 「SDR のアポ獲得数が目標比 72%」→ なぜ？
    → 「新規リードの質が低下」+「架電効率の低下」
```

### 2.3 v1 からの進化

v1 は「報告テキストの曖昧さ検知」に焦点を当てていた。しかし報告品質の担保は経営管理サイクルの手段であり、目的ではない。

| 工程 | v1 のカバー範囲 | v2 のカバー範囲 |
|---|---|---|
| 予実管理 | KPI ツリーの表示のみ（モック） | KPI 定義、自動取得、達成率計算、異常検知 |
| 差異分析 | 報告テキストの曖昧さ検知（5 パターン） | 定量差異の自動検知 + 報告品質担保 + 因果分析支援 |
| ネクストアクション | なし（検知のみ） | アクション案の自動生成 + 構造化 |
| 結果の反映 | なし | アクション効果の追跡 + モデル更新 |

詳細設計は [docs/design-v2-management-cycle.md](docs/design-v2-management-cycle.md) を参照。

## 3. ソリューション仮説

### 3.1 プロダクト概要

**「経営管理サイクルの全工程を AI が並走し、計画と現実の乖離を必然的に縮小させるプロダクト」**

v1 のソリューション: AI ドリブンの定例資料作成アシスタント。週次定例・MBR で提出される資料の曖昧さを AI が検知し、ユーザーに質問・指示を出すことで解像度の高い資料を作成する。

```
v1 フロー:
マネージャー → AI が曖昧さを検知 → 質問/指示 → 回答 → 解像度の高い資料完成

v2 フロー:
KPI 実績データ取得 → 異常検知 → 週次報告品質チェック → 差異の構造化分析
→ 因果関係の特定 → アクション計画生成 → 効果測定 → 次サイクルへ
```

### 3.2 曖昧さ検知パターン（v1 コア）

Zero Ambiguity の原則に基づき、5 つのパターンを検知する。

| パターン | 説明 | 検知例 | 質問/指示例 |
|---|---|---|---|
| **P1: 事実/解釈混在** | Fact と Interpretation の混同 | 「顧客の反応は概ね良好」 | 「事実（受注率 XX%）と解釈を分離してください」 |
| **P2: 定量化不足** | 数値がない、定性的すぎる | 「概ね順調」「改善傾向」 | 「具体的な数値を記載してください（現在値/目標値）」 |
| **P3: アクション不明確** | Who/What/When が不明 | 「対応予定」「検討中」 | 「誰が、何を、いつまでに行うか明記してください」 |
| **P4: 浅い分析** | 原子レベルまで分解されていない | 「売上未達」で止まっている | 「未達の要因を A-1, A-2... に分解してください」 |
| **P5: カバレッジ不足** | MECE でない、抜け漏れ | 3 施策のうち 2 つしか報告なし | 「施策 C の進捗が記載されていません」 |

### 3.3 MVP 機能

1. **Google Slides / Docs / Notion 連携**: 資料のインポート/監視、曖昧さのハイライト
2. **曖昧さ検知エンジン**: 5 パターンの検知、検知箇所と理由の特定
3. **質問/指示生成**: 曖昧さに対する具体的な質問・指示の生成、Web アプリ内チャット
4. **タスク管理**: 未解決の曖昧さのタスク化、通知によるリマインド
5. **品質スコア**: 資料の曖昧さ排除度合いの可視化

### 3.4 将来機能（MVP 後）

- KPI データの自動連携（Salesforce, GA, freee 等）
- 差異分析エンジン（市場/固有/一時要因の分解、因果チェーン構築）
- アクション計画生成（Who/What/When/完了基準の構造化）
- Reflection（アクション効果測定、因果モデル更新）
- CEO 向けダッシュボード（Observability）
- 自動レポート生成

## 4. バリュープロポジション

### 4.1 一言で言うと

**「定例資料の曖昧さをゼロにし、経営会議の生産性を最大化する AI」**

週次定例・MBR で「なぜ？」「具体的には？」「誰がいつまで？」と指摘される前に、AI が事前に曖昧さを検知・排除。手戻りゼロの解像度の高い報告を実現する。

### 4.2 Pain Relievers

| Pain | Reliever |
|---|---|
| 毎回「なぜ？」と深掘りが必要 | 定例前に AI が深掘り完了、CEO の認知負荷ゼロ |
| 品質がマネージャーにより異なる | 一貫した基準で全員に同じ品質を要求 |
| 情報確認に会議時間を浪費 | 定例開始時点で情報は揃っている |
| 嫌われ役になる疲弊 | AI が Bad Cop、CEO は意思決定に集中 |
| 何を書けばいいかわからない | AI が具体的に質問・指示 |
| 定例で指摘されて手戻り | 事前に曖昧さ排除、手戻りゼロ |

### 4.3 競合ポジショニング

```
                    自律性（Autonomous）
                           |
                           |
              agentic-ai-management
                           |
        Human COO          |
                           |
    ───────────────────────┼───────────────────────→ 経営特化
        汎用                |                専門
                           |
            ChatGPT        |
            Notion AI      |
                           |
                    受動性（Reactive）
```

**独自ポジション**: 経営管理に特化 x 自律介入

| 観点 | 本プロダクト | Notion AI / ChatGPT | Human COO |
|---|---|---|---|
| 曖昧さ検知 | 5 パターン特化 | 汎用的 | 属人的 |
| 自律介入 | 許可不要 | 受動的 | 可能 |
| 一貫した基準 | 常に同じ | プロンプト次第 | 属人的 |
| 感情なき指摘 | 常に冷徹 | 常に冷徹 | 忖度あり |
| コスト | SaaS 価格 | 安価 | 1500 万-1 億 |
| 24/7 稼働 | 常時 | 常時 | 不可 |

### 4.4 システム構成概要

```
┌─────────────────────────────────────────────────────────┐
│              agentic-ai-management (Web アプリ)           │
│         統合作業環境・ダッシュボード                       │
│  ┌─────────────┬─────────────┬─────────────┐            │
│  │   チャット   │    通知     │   タスク    │            │
│  └─────────────┴─────────────┴─────────────┘            │
└─────────────────────────────────────────────────────────┘
          | 監視                    | 出力
┌─────────────────┐         ┌─────────────────────────┐
│   Data Sources  │         │      Output             │
│  Google Chat    │         │  Google Chat（通知）     │
│  Google Sheets  │         │  Google Docs（レポート） │
│  Google Slides  │         │  Google Slides（資料）   │
└─────────────────┘         └─────────────────────────┘
```

### 4.5 既存の代替手段との比較

| 代替手段 | 長所 | 短所 |
|---|---|---|
| **CEO 自身が指摘** | 精度が高い | 認知負荷が高い、定例の時間が浪費される |
| **Human COO** | 文脈理解、柔軟性 | 高コスト（1500 万-1 億）、忖度が入る |
| **テンプレート強制** | 一定の構造化 | 形式的になる、曖昧さは排除できない |
| **レビュー文化** | 相互チェック | 人間関係で遠慮、時間がかかる |

本プロダクトのアプローチの優位性:
1. **事前介入**: 定例の「前」に曖昧さを排除（手戻りゼロ）
2. **感情なき指摘**: 人間関係を気にせず論理的に指摘
3. **一貫した基準**: 誰が作成しても同じ品質基準を適用
4. **24/7 稼働**: 人間の COO と違い、常時監視・即座に介入
5. **学習効果**: 繰り返し指摘を受けることで、作成者自身のスキルが向上

## 5. 設計原則

### 5.1 コア原則

1. **Zero Ambiguity（曖昧性の完全排除）**: ゴール、課題、数値報告における曖昧さをバグとして扱う。未達の内訳が原子レベルまで分解されていない状態を許容しない
2. **Fact/Interpretation Separation（事実と解釈の分離）**: 「何が起きたか」と「どう思うか」を明確に区別する。混在した情報はノイズとして処理から除外
3. **Recursive Resolution（再帰的解決）**: 経営管理は KPI 設計 - 特定 - 課題 - 仮説 - 検証の無限ループ。このサイクルを止めるものを排除
4. **Risk/Outcome Isolation（リスクと成果の分離）**: リスクは管理対象、成果は必達目標。「リスクがあるから成果が出ない」を認めない
5. **Autonomous Intervention（自律介入）**: AI は「聞かれたら答えるツール」ではなく「聞かれる前に介入するパートナー」

### 5.2 トレードオフの指針

AI が判断に迷った際の優先基準。

- **Logic > Context**: 文脈より論理を優先。空気を読んで真実を隠すより、空気を壊して真実を提示
- **Fact > Opinion**: 意見より事実。ファクトの裏付けがなければ棄却
- **Structure > Speed**: 速度より構造。ただしユーザーが Speed を選択した場合はトレードオフを記録し、残存する曖昧さを明示
- **Correction > Harmony**: 調和より是正。間違った軌道の修正を優先

## 6. Neural Organization マッピング

### 6.1 5 層への対応

本プロダクトの各機能は、Neural Organization の 5 層アーキテクチャに以下のように対応する。

| Neural Org Layer | 本プロダクトでの実装 | 経営管理サイクルとの対応 |
|---|---|---|
| **L0: Perception（知覚）** | KPI 実績データの自動取得、週次報告の取り込み、外部データ連携 | 予実管理の入力 |
| **L1: Understanding（理解）** | KPI ツリー上の異常検知、曖昧さ 5 パターン検知、差異の構造化 | 差異分析の前半 |
| **L2: Reasoning（推論）** | 因果分析、要因の分解、アクション構成、優先度付け | 差異分析の後半 + ネクストアクション |
| **L3: Execution（実行）** | レポート自動生成、アクションシート生成、通知 | 出力・配信 |
| **L4: Reflection（内省）** | アクション効果測定、因果モデル更新、検知精度改善 | 結果の反映 |

### 6.2 横断的要素

| 要素 | 本プロダクトでの実装 |
|---|---|
| **Purpose** | 経営計画: KPI 目標 + 戦略方針。L2 に作用し行動計画を方向づける |
| **Governance** | Trust Score x HITL ゲート。信頼の蓄積に応じてゲートの閾値を動的調整 |
| **Memory** | 長期: 差異パターン辞書 + 因果モデル + アクション効果実績。作業: 今期の予実状態。評価: 予測精度推移 |
| **Orchestration** | 週次/月次のサイクル駆動。異常検知時の即時起動。定例報告フロー制御 |

### 6.3 人間の参与形態

Neural Organization では、人間は「仕事をする」のではなく「組織知能に参与する」。本プロダクトにおける各参与形態の具体的な役割は以下の通りである。

| 参与形態 | 本プロダクトでの役割 | 具体的な関与場面 |
|---|---|---|
| **Governor** | CEO | KPI 目標の承認、差異分析の検証、アクション計画の承認。経営管理サイクルの方向性を決定 |
| **Sensemaker** | CEO / マネージャー | データに表れない定性的要因の入力、因果分析の補完。「チームの士気が低下している」等の定性情報を提供 |
| **Creator** | CEO | 新たな KPI 設計、戦略方針の定義。事業の方向性を構想 |
| **Connector** | マネージャー | チーム・顧客との関係構築、定性情報の収集。人間関係から得られるインサイトをシステムに統合 |
| **Custodian** | CEO / 管理者 | レポート出力の承認、機密情報の管理。システムの行動が倫理・法規制と整合しているかを監視 |

### 6.4 Governance ゲート

経営管理サイクルの各工程には HITL（Human-in-the-Loop）ゲートが設けられる。Trust Score の蓄積に応じてゲートの閾値が動的に調整され、信頼が高まるほど自動化される範囲が拡大する。

| ゲート | 工程 | 承認者 |
|---|---|---|
| REPORT_QUALITY | 報告品質チェック | マネージャー（報告者本人） |
| VARIANCE_ANALYSIS | 差異分析 | Governor + Sensemaker |
| ACTION_PLAN | アクション計画 | Governor |
| REPORT_OUTPUT | レポート出力 | Custodian |

## 7. 課題の検証状況

| 仮説 | 検証方法 | 状態 |
|---|---|---|
| Fact/Interpretation の分離可能性（LLM 精度 >= 95%） | 過去の Slack ログ 100 件での抽出テスト | 未実施 |
| 再帰的分解の有効性 | プロトタイプによる尋問シミュレーション | 未実施 |
| CEO の受容性（Bad Cop Acceptance） | Wizard of Oz 法による定性評価 | 未実施 |
| 現場の行動変容 | 導入後の曖昧性解消数モニタリング | 未実施 |

## 8. ソリューションの前提条件とリスク

### 前提条件

- CEO が AI からの指摘を受け入れる（確認済み）
- LLM が Google Slides/Docs/Notion から曖昧さを検知できる精度（検証中）
- マネージャーが AI の質問・指示に従って回答する動機づけ（未検証）
- Google Workspace API との連携が技術的に可能（未検証）

### リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| 曖昧さ検知の精度が低い | 高 | 5 パターンに絞り、ルールベース + LLM のハイブリッドで精度向上 |
| マネージャーが指摘を無視 | 中 | CEO への可視化、タスク完了率のトラッキング |
| 過剰な指摘でユーザーが疲弊 | 中 | 重要度の優先順位付け、指摘の上限設定 |
| Google API 制限・コスト | 低 | 監視頻度の最適化、キャッシュ活用 |

## 9. エレベーターピッチ

> **Agentic AI Management** は、週次定例・MBR の資料から曖昧さを自動検知し、作成者に質問・指示を出す AI です。
>
> 「売上未達」で止まっている報告に「なぜ？どの要因？」と深掘りを促し、「対応予定」には「誰が、いつまでに、何を？」と具体化を求めます。
>
> 結果、CEO は定例で「なぜ？」と聞く必要がなくなり、意思決定に集中できます。マネージャーは手戻りなく報告を完了でき、経営視点も身につきます。

## 10. 用語定義

本プロダクトで使用する主要な用語の定義。

| 用語 | 定義 |
|---|---|
| **経営管理サイクル** | 予実管理 - 差異分析 - ネクストアクション定義 - 結果の反映の 4 工程循環プロセス |
| **曖昧さ（Ambiguity）** | ゴール、課題、数値報告における解釈の揺らぎ。本プロダクトではバグとして扱う |
| **原子レベル分解** | 課題の要因を制御可能な最小単位（A-1, A-2...）まで分解すること |
| **Bad Cop** | 冷徹に論理的矛盾を指摘する役割。本プロダクトでは AI が担当する |
| **Trust Score** | AI エージェントの信頼度を定量化するスコア。蓄積に応じて自律範囲が拡大する |
| **HITL ゲート** | Human-in-the-Loop ゲート。AI の判断・行動に人間の承認を要求するポイント |
| **因果チェーン** | 差異の原因を「なぜ？」で深掘りした構造化データ |
| **差異分解** | 全体差異を市場要因・固有要因・一時要因に分解する分析手法 |
