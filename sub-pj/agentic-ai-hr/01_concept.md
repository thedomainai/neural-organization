# HR Policy Advisor — コンセプト設計

> 本ドキュメントは HR Policy Advisor のコンセプト（問題構造・解決方針・思想）を定義する。プロジェクト概要は [00_overview.md](00_overview.md)、アーキテクチャは [02_architecture.md](02_architecture.md) を参照。
> 詳細な v2 設計は [docs/design-v2-neural-org-aligned.md](docs/design-v2-neural-org-aligned.md) を参照。

## 1. People Cycle — 人事制度のライフサイクル

### 1.1 People Cycle とは

People Cycle は、人事制度の設計から運用・進化までの全ライフサイクルを表すモデルである。HR Policy Advisor はこの People Cycle 全体を対象とし、AI と人間の協働によって各フェーズを支援する。

```
        ┌───────────────────────────────────┐
        │                                   │
        ▼                                   │
  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
  │  Define   │───▶│  Design  │───▶│  Deploy  │───▶│ Operate  │
  │  定義     │    │  設計    │    │  導入    │    │  運用    │
  └──────────┘    └──────────┘    └──────────┘    └──────────┘
        ▲                                              │
        │         ┌──────────┐                         │
        │         │  Evolve  │◀────────────────────────┘
        └─────────│  進化    │
                  └──────────┘
```

### 1.2 People Cycle の 5 フェーズ

| フェーズ | 内容 | 主要なアクティビティ |
|---|---|---|
| **Define（定義）** | 組織の存在目的・価値観・求める人材像を明確化する | Mission/Vision/Values の言語化、人材像の定義、「3 つの究極の問い」への回答 |
| **Design（設計）** | 定義に基づいて等級・評価・報酬制度を設計する | 等級制度設計、コンピテンシー定義、評価基準策定、報酬体系構築 |
| **Deploy（導入）** | 設計した制度を組織に導入する | 導入計画策定、説明会実施、初期格付け、激変緩和措置 |
| **Operate（運用）** | 制度を日常的に運用する | 期初ミッション設定、期中 1on1、期末評価、キャリブレーション、フィードバック |
| **Evolve（進化）** | 運用結果を分析し、制度を改善する | HITL 修正パターン分析、離職率・エンゲージメント評価、制度改定提案 |

### 1.3 People Cycle の各フェーズの詳細

**Define（定義）**

制度設計のすべての起点となるフェーズ。企業が「どのような組織でありたいか」「どのような人材を求めるか」を明確にする。v2 では Knowledge Layer の「3 つの究極の問い」を通じて、報酬哲学まで含めた包括的な定義を行う。

主なアクティビティ:
- Mission/Vision/Values の言語化と構造化
- 求める人材像（IdealTalentProfile）の定義
- 報酬哲学の決定（3 つの究極の問い）
- 業界ベンチマークとの比較による現状把握

**Design（設計）**

定義フェーズの出力を入力として、具体的な制度を設計するフェーズ。等級 → 評価 → 報酬の依存関係に従い、順次設計が進行する。各ステップで HITL 承認を挟むことで、人間の価値判断を反映する。

**Deploy（導入）**

設計された制度を組織に導入するフェーズ。4 フェーズ運用体系（F6）に基づき、準備 → 展開 → 導入・移行 → 運用サイクル確立の順で進行する。初期格付けシミュレーションや激変緩和措置を含む。

**Operate（運用）**

制度を日常的に運用するフェーズ。期初のミッション設定、期中の 1on1、期末の評価、キャリブレーション、フィードバックのサイクルを回す。運用データ（評価結果分布、離職率、エンゲージメント）は Evolve フェーズの入力となる。

**Evolve（進化）**

運用結果を分析し、制度を改善するフェーズ。ReflectionAgent が HITL 修正パターンと運用データを分析し、制度改定提案を自動生成する。承認された改定は Define フェーズに戻り、新たな People Cycle が始まる。

### 1.4 v1 と v2 の People Cycle カバレッジ

v1 は People Cycle の **Define** と **Design** フェーズのみをカバーしている。v2 では全フェーズをカバーし、Evolve から Define へのフィードバックループにより、制度が組織の現実から学習し自律的に進化し続ける状態を実現する。

| フェーズ | v1 | v2 |
|---|---|---|
| Define | 企業情報の手動入力 | 外部データ自動取得 + Knowledge Layer 参照 |
| Design | 5 エージェントの逐次設計 | フレームワークベース設計 + Purpose 駆動 |
| Deploy | 対象外 | PolicyDocumentAgent + DeploymentPlannerAgent |
| Operate | 対象外 | 運用テンプレート生成 + 運用データ自動取り込み |
| Evolve | 対象外 | ReflectionAgent による継続的学習と制度改定提案 |

### 1.5 People Cycle と組織の成熟

People Cycle は 1 回で完結するものではない。サイクルを重ねるごとに制度の精度が向上し、組織固有の学習が蓄積される。

| サイクル | 特徴 |
|---|---|
| 1 周目 | Knowledge Layer のフレームワークを基に初期制度を設計。企業コンテキストに合わせた粗いカスタマイズ |
| 2 周目 | 1 周目の運用結果と HITL 修正パターンを反映。組織の暗黙の価値観が学習され始める |
| 3 周目以降 | 組織固有の判断パターンが長期記憶に蓄積され、提案の初期品質が大幅に向上。Trust Score 上昇により HITL ゲートが自動パスされ始める |

## 2. agentic-ai-hr-system 依存関係

### 2.1 agentic-ai-hr-system とは

**agentic-ai-hr-system** は、HR 制度の定義レイヤーとして機能する別プロジェクトである。等級定義、コンピテンシー定義、評価基準を構造化データとして管理し、HR ドメインにおける **Single Source of Truth（SSOT）** を提供する。

### 2.2 役割分担

```
┌─────────────────────────────────┐
│  agentic-ai-hr-system           │
│  （定義レイヤー / SSOT）          │
│                                 │
│  - 等級定義（Grade Definitions） │
│  - コンピテンシー定義             │
│  - 評価基準（Evaluation Criteria）│
│  - 報酬テーブル雛形              │
└──────────────┬──────────────────┘
               │ 参照・消費
               ▼
┌─────────────────────────────────┐
│  agentic-ai-hr                  │
│  （知的支援レイヤー）              │
│                                 │
│  - 企業コンテキストの理解          │
│  - 定義を基にした制度設計提案      │
│  - HITL 承認ワークフロー          │
│  - Reflection による進化          │
└─────────────────────────────────┘
```

### 2.3 依存関係の詳細

| 本プロジェクトが消費するもの | 説明 |
|---|---|
| スコープベース等級モデル | 責任範囲（Scope of Impact）で等級を定義する 8 段階の参照モデル |
| コンピテンシー・トリアド | 3 領域 x 2 要素 = 6 コンピテンシーのメタ構造 |
| コンピテンシー進化マトリクス | 各コンピテンシーが等級ごとにどう進化するかの 8 段階記述 |
| 3 ゲート評価システム | 成果評価・行動評価・コンピテンシー評価の 3 段階ゲート構造 |
| 報酬設計の 3 つの究極の問い | 能力 vs 職務、蓄積 vs 洗い替え、階級 vs 機能 |

### 2.4 設計原則

本プロジェクトは agentic-ai-hr-system の定義をそのまま適用するのではなく、**企業のコンテキストに応じてカスタマイズして適用する**。agentic-ai-hr-system が提供するのは普遍的なフレームワーク（骨格）であり、本プロジェクトがそれを企業固有の制度に具体化する。

例:
- コンピテンシー・トリアドの「受信・傾聴」→ コンサル企業では「顧客課題の深掘り」に具体化
- スコープベース等級モデルの 8 段階 → 企業規模に応じて 4〜8 段階を選択
- 3 ゲート評価の重み → スタートアップはゲート 1（成果）重視、大企業はゲート 2（行動）重視

## 4. Knowledge Layer — 普遍的フレームワーク

### 4.1 フレームワーク一覧

agentic-ai-hr-system が提供する普遍的フレームワークを Knowledge Layer として本プロジェクトに組み込む。これらは企業固有の要素を除外した、あらゆる企業に適用可能な構造である。

| # | フレームワーク | 概要 | 利用エージェント |
|---|---|---|---|
| F1 | コンピテンシー・トリアド | 3 領域 x 2 要素の行動特性構造 | TalentModelingAgent |
| F2 | スコープベース等級モデル | 責任範囲で等級を定義する 8 段階モデル | GradingArchitectAgent |
| F3 | コンピテンシー進化マトリクス | 等級ごとの行動進化の 8 段階記述 | GradingArchitectAgent |
| F4 | 3 ゲート評価システム | 成果・行動・コンピテンシーの 3 段階ゲート | EvaluationArchitectAgent |
| F5 | 報酬設計の 3 つの究極の問い | 報酬方針の根本的選択 | CompensationArchitectAgent |
| F6 | 4 フェーズ運用体系 | 準備 → 展開 → 導入 → 運用サイクル | DeploymentPlannerAgent |

### 4.2 コンピテンシー・トリアド（F1）

```
思考・構想（Thinking）
  ├─ A. 課題設定（Issue Finding）
  └─ B. 構造化・計画（Structuring）

実行・行動（Action）
  ├─ C. スピード（Speed）
  └─ D. 品質（Quality）

影響・対人（Relation）
  ├─ E. 受信・傾聴（Listening）
  └─ F. 発信・伝達（Delivery）
```

3 領域は業種・企業規模を問わず不変の骨格として機能し、要素名・重みは企業の Mission/Vision/Values に応じてカスタマイズされる。

### 4.3 スコープベース等級モデル（F2）

| 段階 | スコープ | 概念 |
|---|---|---|
| 8 | 業界・社会 | 産業構造の再定義 |
| 7 | 全社・経営 | 経営資源の最適配分 |
| 6 | 本部・事業部 | 事業戦略・構造改革 |
| 5 | 部・機能 | 年単位の KPI 達成 |
| 4 | プロジェクト群 | 複数チーム横断支援 |
| 3 | チームリード | 単一チーム統括 |
| 2 | 自律・技術牽引 | 専門性 + リード支援 |
| 1 | タスク・定型業務 | 基本動作習得 |

v1 の J1/J2/S1/S2 命名は固定的であったが、v2 ではスコープに基づく本質的な設計に移行する。

### 4.4 コンピテンシー進化マトリクス（F3）

各コンピテンシーが等級ごとにどう進化するかを記述した 8 段階のパターンライブラリ。

```
例: 課題設定（Issue Finding）の進化
  G1: 違和感のアラート → 異常に気づいて報告できる
  G2: 自律的な解決策提示 → 複数案を持って上司に相談
  G3: チーム課題の発見 → 改善テーマを自ら設定
  G4: プロジェクト群の最適化 → 横断的課題を発見
  G5: 部門方針の策定 → 戦略を KPI に落とし込む
  G6: 構造的課題の特定 → 根本原因を見抜く
  G7: 事業創造と破壊 → ピボット判断をする
  G8: 産業構造の再定義 → 業界ルール自体を書き換える
```

GradingArchitectAgent は、企業固有のコンピテンシー名に対してこの進化パターンをテンプレートとして適用し、等級別の期待行動を生成する。

### 4.5 3 ゲート評価システム（F4）

昇格判定の 3 段階ゲート構造。

```
ゲート 1: 成果評価（チケット）      → 目標達成度が基準以上
ゲート 2: 行動評価（ゲートキーパー）  → バリュー体現度が基準以上
ゲート 3: コンピテンシー評価（卒業要件）→ 全領域で基準以上
```

5 段階採点: Outstanding(5) / Exceed(4) / Meet(3) / Below(2) / Unsatisfactory(1)

EvaluationArchitectAgent は、この構造を推奨デフォルトとして企業に提示し、企業の状況に応じてゲートの重みをカスタマイズする。

### 4.6 3 つの究極の問い（F5）

CompensationArchitectAgent が報酬制度を設計する前に、Governor（人間の意思決定者）に以下の 3 つの問いを提示し、方針を明確化する。

```
Q1: 評価ソース — Competency（能力）vs. Role（職務）?
Q2: 報酬ロジック — Accumulation（安定性）vs. Wash-replacement（市場競争力）?
Q3: 管理定義 — Status（階級）vs. Function（機能）?
```

これにより「Intent over Instruction」— 指示ではなく意志を受け取る設計を実現する。人間は報酬の具体的な数値を指示するのではなく、報酬哲学を表明する。

### 4.7 4 フェーズ運用体系（F6）

制度導入後の運用を 4 フェーズで体系化する。

```
Phase 0: 準備・合意（マネージャー先行説明会、ツール整備）
Phase 1: 展開（全社 Town Hall、FAQ 設置）
Phase 2: 導入・移行（初期格付け、キャリブレーション、激変緩和）
Phase 3: 運用サイクル（期初ミッションレター → 期中 1on1 → 期末評価 → フィードバック）
Phase 4: メンテナンス（制度サーベイ、マイナーアップデート）
```

v1 にはなかった「設計後のライフサイクル」をカバーする新エージェント群（PolicyDocumentAgent, DeploymentPlannerAgent）の基盤となる。

### 4.8 除外する企業固有要素

Knowledge Layer は普遍的なフレームワークのみを含み、以下の企業固有要素は除外する。これらはパラメータ化されたテンプレートとして保持し、企業ごとに Governor が値を設定する。

| 除外する要素 | 除外理由 |
|---|---|
| 具体的年俸テーブル（G1: 300 万〜G8: 2,200 万） | 企業の予算・市場・業種に依存 |
| 役職手当の金額 | 企業の報酬ポリシーに依存 |
| 評価サイクル日程 | 企業の会計年度に依存 |
| ボーナス廃止/存続方針 | 経営方針として個社判断 |

## 5. Neural Organization マッピング

### 5.1 設計原理の体現

HR Policy Advisor は Neural Organization の設計原理を人事制度ドメインで体現する。

| Neural Org 設計原理 | 本プロジェクトでの体現 |
|---|---|
| Intent over Instruction | 3 つの究極の問いで経営の「意志」を受け取り、具体的な制度を自律的に導出する |
| Governance, not Control | Governor は方針を設定し、詳細設計はシステムに委ねる。HITL ゲートは品質確認ではなく価値判断に集中する |
| Radical Transparency | 各エージェントが設計根拠を明示する。「なぜこの等級構造にしたか」の説明が常にアクセス可能 |
| Agency Preservation | 人間はいつでも制度案を覆せる。システムなしでも制度運用は継続可能 |
| Mutual Evolution | 人間の修正が学習データとなり、システムの提案品質が向上する。同時に、システムのインサイトが人間の判断を拡張する |

### 5.2 人間の参与形態の適用

Neural Organization の 5 つの参与形態を HR Policy Advisor の文脈に適用する。

| 参与形態 | HR 領域での役割 | 具体的な行動 |
|---|---|---|
| Governor | 経営者 / CHRO | Mission/Vision/Values の定義、3 つの究極の問いへの回答、制度改定の最終承認 |
| Sensemaker | 現場マネージャー | 「データに表れない組織文化」「従業員の本音」をシステムに伝える |
| Creator | 人事企画担当 | まだ存在しない制度の構想を提示し、システムがその実現方法を推論する |
| Connector | HRBP | 従業員との信頼関係構築、制度導入時のコミュニケーション |
| Custodian | 法務 / コンプライアンス | 制度が労基法・倫理基準に適合しているかを監視する |

### 5.3 Reflection の核心

Neural Organization の concept.md が述べる「人間の修正行為そのものが最も価値の高い学習データ」を、HR Policy Advisor は以下のように具体化する。

| 人間の行動 | システムの学習 |
|---|---|
| Governor が報酬テーブルを 2 回連続で上方修正した | 「この企業は市場の 75 パーセンタイルを目指す傾向がある」 |
| Custodian が評価基準の表現を 3 回修正して柔らかくした | 「この企業は評価の表現にケアを重視する文化」 |
| Governor がスタートアップ向けに等級数を 4 に減らした | 「小規模企業では等級を絞る傾向。50 名以下は 4 等級を推奨初期値に」 |

これらの学習は長期記憶に蓄積され、同一企業の次回設計や、類似企業への推奨値として活用される。

## 6. 対象ドメインの知識

### 6.1 等級制度の類型

| 類型 | 定義 | 特徴 | 適合企業 |
|---|---|---|---|
| 職能等級 | 従業員の能力・スキルに基づく | 日本企業で伝統的、年功的運用になりやすい | 製造業、長期育成型 |
| 職務等級 | 職務の内容・責任に基づく | 欧米型、同一労働同一賃金と親和性高い | 外資系、IT、スタートアップ |
| 役割等級 | 期待される役割・成果に基づく | 職能と職務のハイブリッド | 変化の激しい業界、成長企業 |

### 6.2 業種別の制度設計特性

**コンサルティング業**
- 等級制度: 役割等級が適合（柔軟な役割変更）
- 評価: 成果とプロセスのハイブリッド
- コンピテンシー: 論理的思考、顧客志向、知識習得

**SaaS 業**
- 等級制度: 職務等級または役割等級
- 評価: OKR 連動、360 度評価
- コンピテンシー: 自走力、協働、顧客成功志向

**軽貨物運送業**
- 等級制度: 職能等級（技能習熟）
- 評価: 安全・品質・効率のバランス
- コンピテンシー: 安全意識、顧客対応、業務改善

### 6.3 法令遵守事項

| 法令 | 関連する制度設計要件 |
|---|---|
| 労働基準法 | 最低賃金の遵守、割増賃金（時間外・休日・深夜）、賃金支払いの 5 原則 |
| 同一労働同一賃金 | 正規・非正規間の不合理な待遇差禁止（基本給、賞与、手当、福利厚生） |
| 個人情報保護法 | 評価データ・報酬データの適切な管理 |
