# HR Policy Advisor — プロジェクト概要

> 本ドキュメントは HR Policy Advisor プロジェクトの全体像を示す。コンセプト設計は [01_concept.md](01_concept.md)、アーキテクチャは [02_architecture.md](02_architecture.md)、実装詳細は [03_implementation.md](03_implementation.md) を参照。

## 1. WHY — なぜこのプロジェクトが必要か

### 1.1 背景

企業の人事制度（等級・評価・報酬）の設計は、組織運営における最も重要かつ困難な意思決定の一つである。制度は従業員の行動を方向づけ、組織文化を形成し、人材の獲得・定着・育成に直結する。

しかし、多くの企業がこの領域で以下の課題を抱えている。

| 課題 | 影響 |
|---|---|
| 専門知識の偏在 | 制度設計のノウハウは一部の人事コンサルタントに集中しており、中堅企業がアクセスしにくい |
| 高いコスト | 外部コンサルタントへの依頼は数百万〜数千万円のコストがかかる |
| 属人性 | 制度の設計理由や過去の議論が担当者の頭の中にしか存在せず、引き継ぎで失われる |
| 法令対応の難しさ | 労働基準法・同一労働同一賃金等の法令との整合性確認に専門知識が必要 |
| 運用との乖離 | 設計された制度が実際の運用で形骸化し、意図した効果を発揮しない |

### 1.2 ビジョン

HR Policy Advisor は、AI エージェントと人間の協働により、企業の人事制度策定を高品質かつ効率的に支援するシステムである。

人事コンサルタントの専門知識を AI で再現し、企業の特性（業種・規模・文化）に応じた制度を設計する。重要な意思決定には必ず人間の判断を介在させる Human-in-the-Loop（HITL）アーキテクチャにより、AI の効率性と人間の価値判断を両立する。

### 1.3 核となる価値提案

1. **専門知識の民主化**: 人事コンサルタントの知見を AI で再現し、中堅企業でもアクセス可能にする
2. **企業別最適化**: 業種・規模・文化に応じた制度設計を、汎用フレームワークのカスタマイズによって実現する
3. **コンプライアンス担保**: 労働法規との整合性を自動チェックし、法的リスクを未然に防止する
4. **Human-in-the-Loop**: 重要な意思決定は必ず人間が確認し、AI の効率性と人間の価値判断を両立する
5. **知識の永続化**: 制度設計の理由・判断履歴をシステムに蓄積し、属人性を排除する

### 1.4 目標

| # | 目標 | 成功指標 |
|---|---|---|
| G1 | 人事制度策定の工数を 50% 削減 | 従来の制度策定期間との比較 |
| G2 | 企業特性に応じたカスタマイズ提案 | 採用された提案の割合 |
| G3 | 法令・規制への準拠率 100% | コンプライアンスチェック合格率 |

## 2. WHO — 誰のためのシステムか

### 2.1 プライマリペルソナ: 人事部長（HR Director）

企業規模 100〜1,000 名の中堅企業に所属する人事責任者。制度設計の専門知識は限定的で、外部コンサルタントへの依頼コストに課題を感じている。従業員の可能性を最大化でき、かつ運用がシンプルな人事制度の構築を目指す。

**ターゲット業種**:
- コンサルティング業: プロジェクトベース、専門性重視
- SaaS 業: スピード重視、クロスファンクショナルなチーム構成
- 軽貨物運送業: 現場オペレーション中心、安全重視

### 2.2 セカンダリペルソナ: 人事コンサルタント

人事制度設計経験 5 年以上の専門家。案件数増加に伴う品質担保の困難さに課題を持ち、効率的に高品質な提案を行うためのツールとして活用する。

### 2.3 ペルソナ別の利用シナリオ

**人事部長の場合**:
- 課題: 従業員 200 名の SaaS 企業で、創業期の属人的な評価制度がスケールしなくなった
- 利用フロー: 企業情報登録 → AI が業界ベンチマークと比較した分析を提示 → コンピテンシーベースの等級制度案を承認/修正 → 評価・報酬制度を一貫設計 → 導入計画を受け取る
- 得られる価値: コンサルタント不要で、自社文化に合った制度を 2 週間で設計

**人事コンサルタントの場合**:
- 課題: 月 3-4 件の案件を並行で担当しており、各社への初期提案に時間がかかる
- 利用フロー: クライアントの企業情報を入力 → AI が初期設計案を生成 → コンサルタントが専門知識で修正・精緻化 → クライアントに提案
- 得られる価値: 初期設計の工数を 50% 削減し、カスタマイズと付加価値提案に集中

### 2.3 ユーザージャーニー（MVP）

```
1. 企業情報登録
   └ 業種、従業員数、現行制度の有無、経営理念、課題感をフォーム入力

2. 現状分析
   └ AI が業界ベンチマークと比較して分析
   └ [HITL] 分析結果の確認・修正

3. 制度設計（自動生成 → 承認）
   └ 求める人材像 → 等級制度 → 評価制度 → 報酬制度の順で提案
   └ [HITL] 各制度の承認・差し戻し

4. ドキュメント生成
   └ 規程文書、説明資料の自動生成
   └ [HITL] 最終確認

5. 運用支援
   └ 導入スケジュール、FAQ 生成
```

## 3. WHAT — 何を提供するか

### 3.1 対象とする人事制度

| 制度 | MVP | Phase 2 | Phase 3 |
|---|---|---|---|
| 等級制度（職能・職務・役割） | 対象 | - | - |
| 評価制度（MBO・コンピテンシー） | 対象 | - | - |
| 報酬制度（基本給・賞与） | 対象 | - | - |
| 教育研修制度 | - | 対象 | - |
| 退職金制度 | - | - | 対象 |
| 福利厚生制度 | - | - | 対象 |

### 3.2 コア機能

**マルチエージェント制度設計**
複数の専門エージェントが協調して制度を設計する。各エージェントは特定の領域（文脈収集、人材像設計、等級設計、評価設計、報酬設計）を担当し、前のステップの出力を次のステップの入力として利用する。

**Human-in-the-Loop 承認ワークフロー**
制度設計の各ステップに承認ゲート（7 ゲート）を配置する。人間は AI が生成した制度案を確認し、承認または差し戻しを行う。差し戻し時のフィードバックは次回の生成品質向上に活用される。

**コンプライアンスチェック**
労働基準法・同一労働同一賃金等の法令との整合性を自動検証する。重大なリスクが検出された場合は人間にエスカレーションする。

**業種別最適化**
業種固有の特性（重視するコンピテンシー、報酬水準、評価方式）を反映した制度設計を行う。

### 3.3 制度設計のフロー

制度設計は以下の依存関係に従って順次進行する。各ステップで HITL 承認が必要であり、差し戻しが発生した場合は該当ステップからやり直す。

```
企業情報登録
    |
    ▼
求める人材像（HITL-002）
    |  承認後
    ▼
等級制度（HITL-003）
    ├── 等級階層の定義
    ├── 3 つのコンピテンシー
    ├── コンピテンシー要素（各 2 つ、計 6 個）
    └── 等級別・コンピテンシー別の卒業要件
    |  承認後
    ▼
評価制度（HITL-004）
    |  承認後
    ▼
報酬制度（HITL-005）
    |  承認後
    ▼
コンプライアンスチェック（HITL-006）
    |
    ▼
ドキュメント生成（HITL-007）
```

### 3.4 MVP のスコープ外

- 給与計算システムとの連携
- 人事評価システムとの連携
- 多言語対応
- モバイルアプリ

## 4. プロジェクトステータス

### 4.1 現在の状態: バックエンド実装完了（v1）

v1 はバックエンド中心の実装が完了している。以下のコンポーネントが動作する状態にある。

| コンポーネント | 状態 | 概要 |
|---|---|---|
| API レイヤー | 実装済み | FastAPI ベースの REST API（Companies, Policies, Reviews） |
| オーケストレーター | 実装済み | 5 ステップのワークフロー管理（DAG ベース） |
| エージェント群 | 実装済み | 5 設計エージェント + 3 補助エージェント |
| HITL | 実装済み | 7 ゲートの承認ワークフロー |
| ドメインモデル | 実装済み | Company, GradingSystem, EvaluationSystem, CompensationSystem |
| インフラ | 実装済み | Docker Compose（Redis, RabbitMQ, PostgreSQL, Vault） |

**ソースコード構成**: 56 の Python ファイル（`src/` 配下）

### 4.2 v2 設計方針

v2 では Neural Organization の 5 層パイプラインに完全準拠した設計への進化を計画している。主な変更点は以下の通りである。

| 観点 | v1 | v2 |
|---|---|---|
| スコープ | 制度の設計 | 設計 → 導入 → 運用 → 進化の全ライフサイクル |
| 知識基盤 | LLM の汎用知識のみ | 構造化フレームワーク + 組織固有の学習 |
| 学習 | なし | Reflection による継続的学習 |
| 自律性 | 固定（全ゲート毎回承認） | Trust Score に応じた動的ゲート制御 |
| 記憶 | Redis TTL（揮発的） | 3 構造記憶（長期/作業/評価）の永続化 |

v2 の詳細設計は [docs/design-v2-neural-org-aligned.md](docs/design-v2-neural-org-aligned.md) を参照。

### 4.3 ロードマップ

| Phase | 期間 | 内容 |
|---|---|---|
| Phase 1 | 2-3 週 | Knowledge Layer の構築（HR フレームワーク YAML 化） |
| Phase 2 | 3-4 週 | 既存エージェントの Knowledge Layer 参照型への改修 |
| Phase 3 | 2-3 週 | Memory の永続化（PostgreSQL スキーマ実装） |
| Phase 4 | 4-5 週 | 新エージェント実装（PolicyDocument, DeploymentPlanner, Reflection） |
| Phase 5 | 2-3 週 | Trust Score と動的ゲート制御 |
| Phase 6 | 2-3 週 | 共有記憶基盤接続（neumann, ai-executive-dashboard） |

## 5. Neural Organization における位置づけ

HR Policy Advisor は Neural Organization エコシステムの一部であり、**人事制度ドメインにおける Agentic AI** として機能する。

### 5.1 他プロジェクトとの関係

| 関連プロジェクト | 関係性 |
|---|---|
| agentic-ai-hr-system | HR 定義レイヤー（等級・コンピテンシー・評価基準の Single Source of Truth）。本プロジェクトはこの定義を消費して制度設計を行う |
| neumann | 報告品質管理 AI。等級定義・コンピテンシー定義を参照し、等級に応じた報告品質基準を動的調整する |
| ai-executive-dashboard | 経営ダッシュボード。評価結果データを受け取り、組織能力の経時的変化をレポートに反映する |

### 5.2 共有記憶基盤

本プロジェクトは共有記憶基盤を通じて他プロジェクトとデータを交換する。

**提供するデータ**: 等級定義、コンピテンシー定義、評価結果データ、制度改定履歴
**取得するデータ**: 報告品質スコア（neumann）、市場トレンド（ai-executive-dashboard）
