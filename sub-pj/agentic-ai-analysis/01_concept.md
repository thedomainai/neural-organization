# Agentic AI Analysis - コンセプト設計

> 本ドキュメントは agentic-ai-analysis のコンセプト詳細を定義する。プロジェクトの WHY / WHO / WHAT は [00_overview.md](00_overview.md)、技術アーキテクチャは [02_architecture.md](02_architecture.md) を参照。

> **Project Status: Planning** — 本プロジェクトは設計段階にあり、実装はまだ存在しない。

## 1. Analysis Cycle（データ分析サイクル）

### 1.1 定義

**データ分析サイクル（Analysis Cycle）** とは、組織のデータ分析業務における「問い定義」から「効果測定」に至る全工程を循環的に最適化するプロセスである。単発のレポート作成ではなく、分析→意思決定→アクション→効果測定→次の問いという閉ループとして捉える。

agentic-ai-analysis は、この分析サイクルの全工程を AI が並走し、**「問い」から「アクション」への変換効率を継続的に改善する**プロダクトである。

```
┌──────────────────────────────────────────────────────────────┐
│                    データ分析サイクル                            │
│                                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐     │
│  │ 問い定義  │─→│ データ   │─→│ 探索的   │─→│ 仮説検証  │     │
│  │          │  │ 収集・統合│  │ 分析     │  │ モデリング│     │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘     │
│       ▲                                         │           │
│       │                                         ▼           │
│  ┌──────────┐  ┌──────────────┐  ┌──────────────────┐       │
│  │ 効果測定  │←─│ 意思決定支援  │←─│ インサイト抽出   │       │
│  │          │  │ アクション提案│  │ レポーティング    │       │
│  └──────────┘  └──────────────┘  └──────────────────┘       │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 分析サイクルが「サイクル」である理由

従来のデータ分析は「依頼→分析→レポート」の直線的プロセスとして扱われることが多い。このモデルの根本的な問題は、学習ループが存在しないことである。

**直線的モデルの問題点**:
- 分析結果が正しかったか検証されない
- 同じデータ品質問題が毎回再発する
- アナリストの暗黙知が組織に蓄積されない
- 「どの問いが高インパクトだったか」のフィードバックがない

**サイクルモデルの優位性**:
- 効果測定が次の問い定義にフィードバックされる（問いの質が改善される）
- データ品質ルールが蓄積・自動適用される（品質問題の再発が減少する）
- 分析手法の有効性が追跡される（手法選択の精度が改善される）
- 予測と実績の乖離が記録される（予測精度が改善される）

### 1.3 各工程の詳細

#### 工程 1: 問い定義

ビジネス課題を分析可能な「問い」に変換し、ビジネスインパクトに基づいて優先度を判定する。

**入力**: 分析リクエスト（人間からの依頼、または他 Agentic AI からの自動連携）

**処理の核心**:
- 曖昧な依頼（例:「最近の売上が伸び悩んでいる」）を構造化された問い（主問い + 副問い）に分解する
- 各問いにビジネスインパクト、緊急性、実現可能性のスコアを付与する
- 推奨する分析手法、必要データ、前提条件を整理する

**出力**: AnalysisDesign（構造化された問い + 優先度 + 分析設計）

```
元のリクエスト: 「最近の売上が伸び悩んでいる」

構造化された問い:
  主問い: 売上成長率の鈍化要因は何か
  副問い 1: 新規顧客の獲得ペースは変化したか
  副問い 2: 既存顧客の ARPU は変化したか
  副問い 3: 解約率は変化したか
  必要データ: CRM, 請求データ, 解約データ
  推奨手法: コホート分析 + 分解分析
```

#### 工程 2: データ収集・統合

分析に必要なデータを複数ソースから収集し、分析可能な状態に統合する。

**入力**: AnalysisDesign（必要なデータソースのリスト）

**処理の核心**:
- データソースカタログから該当テーブル/API を検索し、接続する
- 複数ソースの結合条件を推定し、統合パイプラインを構築する
- データリネージ（どのソースからどう変換されたか）を記録する

**出力**: IntegratedDataset（統合データセット + スキーマ情報 + リネージ）

#### 工程 3: データ品質確保

データの品質を検証し、既知のルールに基づくクレンジングを適用する。新規の品質問題を検出した場合はルールとして蓄積する。

**入力**: IntegratedDataset + データ品質ルール辞書（Memory）

**処理の核心**:
- プロファイリング（分布分析、外れ値検出、欠損パターン）
- 既知の品質ルール適用（過去に発見・確認済みのルール）
- 新規品質問題の検出と対処案の提案
- 新ルールとして Memory に登録

**出力**: CleanedDataset（クレンジング済みデータ + 品質レポート + 新規ルール提案）

品質ルールの例:

```
ルール: "revenue_null_to_zero"
条件: revenue カラムが NULL
対処: 0 に変換（過去の分析で確認済み）
適用回数: 12 回
最終確認: 2025-W40

ルール: "login_anomaly_cap"
条件: 日次ログイン数 > 10000
対処: 外れ値としてフラグ（Bot の可能性）
適用回数: 3 回
```

#### 工程 4: 探索的分析（EDA）

データからパターン・トレンド・異常を発見し、仮説を生成する。

**入力**: CleanedDataset + AnalysisDesign + 過去の EDA パターン（Memory）

**処理の核心**:
- 単変量・二変量・多変量分析による網羅的探索
- 統計的に有意な変化点・セグメント差の検出
- 発見に基づく仮説の生成（確信度と検証方法を付与）
- 可視化の自動生成

**出力**: EDAReport（発見事項 + 仮説リスト + 可視化セット + 推奨次ステップ）

仮説生成の例:

```
発見: 新規顧客の獲得数が Q3 から 20% 減少

仮説 A: マーケ施策の効果低下（根拠: CPA 上昇）
  必要な検証: チャネル別 CPA 推移の分析
  確信度: 0.65

仮説 B: 市場の縮小（根拠: 業界全体の傾向）
  必要な検証: 市場データとの照合
  確信度: 0.40

仮説 C: 競合の攻勢（根拠: 失注理由の変化）
  必要な検証: 競合価格・機能の比較
  確信度: 0.55
```

#### 工程 5: 仮説検証・モデリング

分析サイクルの中核工程。仮説を統計的に検証し、予測モデル・因果モデルを構築する。

**入力**: EDAReport + CleanedDataset + 過去のモデル精度実績（Memory）

**処理の核心**:
- 問いの性質に応じた手法の選択（差の検定、回帰分析、因果推論等）
- 複数モデルの構築と比較（ベースライン → 高度なモデル）
- 統計的検証（p 値、信頼区間、効果量、多重比較補正）
- 結果の解釈性確保（SHAP 値、部分依存プロット、反実仮想分析）

**出力**: ModelingResult（仮説別検証結果 + モデル精度 + 特徴量重要度 + 再現コード）

手法選択の基準:

| 問いの性質 | 推奨手法 |
|-----------|---------|
| 差があるか？ | t 検定, ANOVA |
| 関係があるか？ | 回帰分析, 相関分析 |
| 将来どうなるか？ | 時系列予測, ML モデル |
| 原因は何か？ | 因果推論, DAG |
| グループはあるか？ | クラスタリング |
| 異常か？ | 異常検知モデル |

#### 工程 6: インサイト抽出

分析結果を「So What?（だから何か）」→「Now What?（何をすべきか）」に変換する。

**入力**: ModelingResult + AnalysisDesign + Purpose + 過去のインサイト→効果実績（Memory）

**処理の核心**:
- 分析結果をビジネスにとっての意味（So What）に変換する
- 具体的で実行可能なアクション提案（Now What）を導出する
- 期待効果を定量的に見積もる
- 受け手（経営層/分析チーム/実行部門）に応じた粒度を調整する

**出力**: InsightPackage（So What + Now What + 受け手別サマリー + リスク）

インサイト変換の例:

```
分析結果:
  新規顧客獲得数が Q3 から 20% 減少
  主要因: CPA の上昇（+35%）、特に Web 広告

So What:
  現在のマーケ投資効率では、年間売上目標に対して
  ¥120M のギャップが発生する見込み

Now What:
  1. Web 広告の ROI 低下チャネルを特定・停止
     期待効果: CPA -15%、節約額 ¥8M/四半期
  2. 高 ROI チャネル（セミナー + 紹介）への再配分
     期待効果: 新規獲得数 +12%
  3. 既存顧客のアップセル強化（代替の売上源）
     期待効果: ARPU +8%
```

#### 工程 7: レポーティング

分析結果を受け手に最適な形式で報告する。

**入力**: InsightPackage + EDAReport + ModelingResult

**処理の核心**:
- 受け手に応じたレポート形式の選択（経営レポート/詳細分析/ダッシュボード/プレゼン）
- データの性質に応じた最適な可視化
- 品質チェック（数値の正確性、個人情報マスキング、ブランド準拠）
- 配信（Slack / メール / BI ツール）

**出力**: AnalysisReport（レポート本体 + 可視化 + 再現コード + 補足資料）

#### 工程 8: 意思決定支援

インサイトに基づく具体的なアクション提案を意思決定者に提示し、承認を得る。

#### 工程 9: 効果測定

アクションの効果を定量追跡し、予測と実績の乖離を記録する。この情報が次のサイクルの問い定義にフィードバックされる。

### 1.4 Neural Organization とのマッピング

本プロジェクトは Neural Organization の 5 層アーキテクチャに以下のようにマッピングされる。

| レイヤー | データ分析サイクルにおける役割 |
|---------|---------------------------|
| L0: Perception（知覚） | 分析リクエストの受付、データソースの接続・取得、データ品質シグナルの検知 |
| L1: Understanding（理解） | データプロファイリング、探索的分析、メタデータの構造化 |
| L2: Reasoning（推論） | 問いの優先度判定、分析設計、仮説検証、インサイト抽出、アクション提案 |
| L3: Execution（実行） | SQL/Python コード生成・実行、可視化・レポート生成、配信 |
| L4: Reflection（内省） | 分析精度の追跡、手法有効性の評価、品質ルール改善、問いの質の評価 |

詳細なマッピングは [02_architecture.md](02_architecture.md) を参照。

## 2. 問題構造

### 2.1 構造的問題の因果関係

データ分析業務の 5 つの構造的問題は独立ではなく、因果関係で繋がっている。

```
「問い」が曖昧なまま分析が始まる
       │
       ├─→ 分析が意思決定に繋がらない
       │          │
       │          └─→ 分析の「やりっぱなし」（効果が測定されない）
       │                      │
       │                      └─→ 分析手法の有効性がわからない
       │                                 │
       │                                 └─→ 分析の属人化が進む
       │
       └─→ 分析リクエストの洪水（優先度が判断できない）

クレンジングルールが記憶されない
       │
       └─→ データ品質問題が繰り返される
                  │
                  └─→ 分析の信頼性が低下 → 意思決定に使われない
```

### 2.2 問題の根本原因

5 つの問題はすべて「**分析サイクルに学習ループ（Reflection）がない**」ことに帰結する。

| 問題 | Reflection があれば |
|------|------------------|
| 分析が意思決定に繋がらない | 過去の「問い→アクション」転換率を学習し、問い定義を改善できる |
| 分析の属人化 | 分析手法の選択理由と有効性を記録し、組織知として蓄積できる |
| データ品質問題の再発 | 品質ルールを辞書化し、自動適用できる |
| 分析のやりっぱなし | 予測と実績の乖離を追跡し、次の分析にフィードバックできる |
| 分析リクエストの洪水 | 過去のインパクト実績に基づいて、問いの優先度を判断できる |

したがって、本プロジェクトの最も重要なエージェントは **Agent 8: AnalysisCycleReflectionAgent**（内省エージェント）である。

### 2.3 「問い」の品質が全工程を支配する

分析サイクルにおいて最もレバレッジが高いのは「問い定義」工程である。問いが曖昧であれば、後続のすべての工程が無駄になる。

良い問いの条件:
- **具体的**: 「売上が伸び悩んでいる」ではなく「Q3 の新規顧客獲得数が前年比 20% 減少した要因は何か」
- **検証可能**: データによって答えが出せる問い
- **アクショナブル**: 結果から具体的なアクションが導ける問い
- **インパクトが定量化できる**: ビジネスへの影響が金額やKPIで表現できる問い

## 3. 設計原則

### 3.1 問い駆動（Question-Driven）

すべての分析は構造化された「問い」から始まる。問いが定義されない分析は実行しない。

**理由**: 問いが曖昧なまま分析を始めると、データの海で漂流し、意思決定に繋がらないレポートが生産される。Agent 1（QuestionArchitectAgent）が問いを構造化し、ビジネスインパクトに基づく優先度を付与することで、分析リソースを最も価値のある問いに集中させる。

### 3.2 品質ルールの蓄積と自動適用

データ品質問題への対処をルールとして蓄積し、次回以降は自動適用する。

**理由**: データ品質問題は組織固有のパターンがある（特定カラムの NULL、特定期間のデータ欠損、特定ソースの異常値）。これらを毎回ゼロから対処するのは非効率である。品質ルール辞書（Memory）に蓄積し、確信度が高いルールは自動適用、新規ルールは人間の承認を経て辞書に追加する。

### 3.3 So What → Now What 変換

分析結果を「だから何か（So What）」→「何をすべきか（Now What）」に必ず変換する。

**理由**: 分析結果が技術的な数値の羅列にとどまると、意思決定者はアクションに結びつけられない。Agent 6（InsightGeneratorAgent）が分析結果をビジネスインパクトで表現し（So What）、具体的で実行可能なアクション提案（Now What）に変換することで、分析→意思決定の断絶を解消する。

### 3.4 予測→実績の追跡（Closed-Loop Analysis）

すべてのアクション提案に期待効果を定量的に付与し、実績との乖離を追跡する。

**理由**: 分析の価値は「正しかったかどうか」によってのみ測定できる。予測精度の追跡は、分析手法の有効性評価、問いの優先度モデルの改善、アナリストの暗黙知の形式知化に不可欠である。Agent 8（AnalysisCycleReflectionAgent）がこの追跡を担う。

### 3.5 受け手最適化

同じ分析結果を受け手に応じた粒度で提供する。

| 受け手 | 関心事 | 提供内容 |
|-------|-------|---------|
| 経営層（Governor） | ビジネスインパクト | So What + Now What + 期待効果の金額 |
| 分析チーム | 手法の妥当性 | 統計的根拠 + 前提条件 + 再現コード |
| 実行部門 | 何をすればいいか | 具体的なアクションステップ + タイムライン |

### 3.6 自動連携による分析リクエスト起動

他の Agentic AI からの自動連携により、分析サイクルが自律的に起動する。

**主要な連携パターン**:

```
neumann → agentic-ai-analysis（自動起動）:
  neumann が KPI 異常を検知
  → agentic-ai-analysis に分析リクエストを自動発行
  → 問いを構造化、優先度を判定
  → 分析サイクルが自律的に回る
  → インサイトを neumann に返却

agentic-ai-sales → agentic-ai-analysis（深掘り依頼）:
  agentic-ai-sales が失注パターンを検出
  → 因果推論モデルで影響度を定量化
  → 結果を agentic-ai-sales に返却
```

## 4. ドメインモデル

### 4.1 中核エンティティ

本プロジェクトの中核となるエンティティと、それらの関連を示す。

```
AnalysisRequest（分析リクエスト）
  │ 1   依頼者、ソース（人間 or 他 AI）、元のリクエスト文
  │
  ▼ 1
AnalysisDesign（分析設計）
  │     構造化された問い、優先度、推奨手法、必要データ
  │
  ├─→ IntegratedDataset → CleanedDataset
  │     データ統合            品質チェック済み
  │
  ├─→ EDAReport
  │     発見事項、仮説リスト、可視化
  │
  ├─→ ModelingResult
  │     仮説検証結果、モデル精度、特徴量重要度
  │
  ├─→ InsightPackage
  │     So What、Now What（ActionProposal[]）
  │
  └─→ AnalysisReport
        レポート本体、可視化、再現コード
```

### 4.2 ActionProposal（アクション提案）

分析サイクルの最終成果物であり、効果測定の追跡対象でもある。

```
ActionProposal:
  description: アクションの内容
  expectedEffect:
    metric: 影響する指標
    predictedImpact: 予測される効果量
    confidence: 確信度
    timeToEffect: 効果発現までの期間
  priority: 優先度
  status: proposed → approved → in_progress → completed
  actualEffect: 実績（Reflection で記録）
```

### 4.3 DataQualityRule（品質ルール）

品質ルールは蓄積と自動適用の対象であり、それ自体が継続的に改善される。

```
DataQualityRule:
  name: ルール名
  condition: 適用条件
  action: 対処方法（修正 / フラグ / 除外）
  confidence: 確信度（0-1）
  applicationCount: 適用回数
  falsePositiveRate: 誤検知率
  lastValidated: 最終確認日
```

## 5. Memory 設計

### 5.1 3 構造

| 構造 | 内容 | 用途 |
|------|------|------|
| 長期記憶 | 分析パターンライブラリ、データ品質ルール辞書、データソースカタログ、インサイト→アクション→効果の因果モデル | 分析手法の選択、品質ルールの自動適用、過去の実績に基づく判断 |
| 作業記憶 | 進行中の分析コンテキスト、生成済みの中間データセット、HITL 待ち状態 | 分析セッションの状態管理 |
| 評価記憶 | 予測精度推移、分析→アクション転換率、問い優先度モデルの精度推移、品質ルールの precision/recall | 分析能力の継続的改善 |

### 5.2 分析パターンライブラリ

## 6. Governance 設計

### 6.1 HITL ゲート一覧

分析サイクルの各工程に人間の承認ポイント（HITL ゲート）を設置する。

| Gate ID | ゲート名 | エージェント | 承認者 | トリガー |
|---------|---------|------------|-------|---------|
| DA-001 | ANALYSIS_REQUEST | QuestionArchitectAgent | Governor | 全分析リクエスト |
| DA-002 | DATA_QUALITY | DataQualityAgent | Custodian | 新規ルール追加時のみ |
| DA-003 | MODEL_VALIDATION | StatisticalModelingAgent | Governor + Sensemaker | 仮説検証完了時 |
| DA-004 | INSIGHT_REVIEW | InsightGeneratorAgent | Governor | アクション提案を含む場合 |
| DA-005 | REPORT_OUTPUT | ReportGeneratorAgent | Custodian | 対外共有時 |
| DA-006 | REFLECTION_REVIEW | AnalysisCycleReflectionAgent | Governor | 四半期レビュー時 |

### 6.2 Trust Score による動的ゲート制御

Trust Score（信頼スコア）の上昇に伴い、自動パスされるゲートが増える。

| Trust Score | 自動化されるゲート | 承認が必要なゲート |
|-------------|-----------------|-----------------|
| Cold Start (0.00-0.39) | なし | 全ゲート |
| Learning (0.40-0.69) | DA-002, DA-005 | DA-001, DA-003, DA-004, DA-006 |
| Trusted (0.70-0.89) | DA-001, DA-002, DA-003, DA-005 | DA-004, DA-006 |
| Highly Trusted (0.90+) | ほぼ全自動 | DA-004（高インパクト時のみ） |

### 6.3 Trust Score の測定因子

| 因子 | 測定方法 |
|------|---------|
| 成功率 | 分析結果が HITL で修正なしに承認された割合 |
| 結果品質 | 予測/インサイトの的中率（予測効果 vs 実際の効果） |
| 整合性 | 分析が Purpose（経営計画の優先事項）と整合している割合 |
| リスク管理 | データ品質問題・個人情報リスクを適切にエスカレーションした割合 |

## 7. フィードバックループ

### 7.1 内省エージェント（AnalysisCycleReflectionAgent）の学習対象

| 学習対象 | 具体的な学習内容 | フィードバック先 |
|---------|---------------|-------------|
| 分析精度 | 予測効果 vs 実績効果の乖離率 | 全エージェント（予測の校正） |
| 手法有効性 | 「手法×ドメイン×問いの種類」ごとの精度追跡 | Agent 4, 5（手法選択の改善） |
| 品質ルール精度 | false positive / false negative の分析 | Agent 3（閾値調整） |
| 問いの質 | 高インパクトだった問いの特徴 | Agent 1（優先度判定の改善） |
| HITL 修正パターン | 人間がどのような修正を加えたか | 該当エージェント（出力の改善） |

### 7.2 HITL 修正パターンの学習例

```
パターン: Governor がインサイトの表現を毎回ビジネスインパクト寄りに修正する
学習: インサイトは技術的表現ではなくビジネスインパクトで表現すべき
反映先: Agent 6（InsightGeneratorAgent）の出力スタイル

パターン: Sensemaker が毎回定性的要因を追加する
学習: この組織では定量分析だけでは不十分、定性要因の考慮が重要
反映先: Agent 4, 5（探索と検証時に定性要因を考慮）
```
