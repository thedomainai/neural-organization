# Agentic AI Analysis - プロジェクト概要

> 本ドキュメントは agentic-ai-analysis プロジェクトの WHY / WHO / WHAT を定義する。コンセプトの詳細は [01_concept.md](01_concept.md)、技術アーキテクチャは [02_architecture.md](02_architecture.md) を参照。

> **Project Status: Planning** — 本プロジェクトは設計段階にあり、実装はまだ存在しない。本ドキュメント群は設計ドキュメント [docs/design-v2-analysis-cycle.md](docs/design-v2-analysis-cycle.md) に基づく計画を整理したものである。

## 0. WHY - 存在意義

### 0.1 解決する課題

多くの組織のデータ分析業務は、以下の構造的問題を抱えている。

| 課題 | 現状の痛み | 本質的原因 |
|------|-----------|-----------|
| 分析が意思決定に繋がらない | レポートは作られるが、アクションに結びつかない | 「問い」が曖昧なまま分析が始まる。分析結果とアクションの間に断絶がある |
| 分析の属人化 | 特定のアナリストに依存。退職すると再現不可能 | 分析の知見（なぜこの手法を選んだか、どの前提が重要か）が暗黙知のまま |
| データ品質の問題が繰り返される | 同じデータ品質問題が毎回発生 | クレンジングルールが記憶されず、毎回ゼロから対処 |
| 分析の「やりっぱなし」 | 分析→レポート→放置。施策の効果が測定されない | 分析サイクルに Reflection がない |
| 分析リクエストの洪水 | 経営層・各部門からの分析依頼が優先度なく積み上がる | 「どの問いが最もインパクトが大きいか」を判断する仕組みがない |

### 0.2 提供価値（バリュープロポジション）

1. **問い→アクション変換の効率化**: ビジネス課題を構造化された分析問いに変換し、統計的に検証されたインサイトから具体的なアクション提案までを一気通貫で支援する
2. **分析ナレッジの組織化**: 分析手法の選択理由、データ品質ルール、インサイトの有効性実績をすべて構造化された記憶として蓄積し、属人化を解消する
3. **分析サイクルの完結**: 問い定義から効果測定までの全工程を回し、予測と実績の乖離を追跡することで、分析精度を継続的に改善する
4. **分析リソースの最適配分**: ビジネスインパクトと実現可能性に基づく問いの優先度判定により、限られた分析リソースを最も効果的な問いに集中させる

### 0.3 現状分析

| 評価軸 | 現状 | 本プロジェクトの目指す姿 |
|--------|------|----------------------|
| 問い定義 | 依頼者の曖昧なリクエストがそのまま分析に入る | 構造化された問い + ビジネスインパクトに基づく優先度判定 |
| データ準備 | アナリストが毎回手動でデータを取得・結合 | データソースカタログ + 自動パイプライン構築 |
| 品質管理 | 品質問題を毎回ゼロから対処 | 品質ルール辞書による自動適用 + 継続的改善 |
| 分析手法 | アナリスト個人の経験に依存 | 分析パターンライブラリによる手法推奨 |
| インサイト抽出 | 分析結果の技術的な数値報告にとどまる | So What → Now What の自動変換 |
| 効果測定 | 分析後の追跡がほぼ行われない | 予測→実績の自動追跡 + フィードバックループ |

### 0.4 Neural Organization における位置づけ

agentic-ai-analysis は Neural Organization のサブプロジェクトとして、**データ分析サイクル（Analysis Cycle）** の最適化を担う。データ分析サイクルとは、問い定義からデータ収集・分析・インサイト抽出・意思決定支援・効果測定に至る分析業務の全工程を指す。

```
データ分析サイクル概要:

  問い定義 → データ収集・統合 → 探索的分析 → 仮説検証
       ▲                                       │
       │                                       ▼
  効果測定 ← 意思決定支援 ← インサイト抽出・レポーティング

  スコープ: 全工程の最適化（8 エージェント体制）
```

### 0.4 他プロジェクトとの連携

agentic-ai-analysis は他の Neural Organization サブプロジェクトと双方向でデータを交換する。

| 連携先 | 提供するデータ | 受け取るデータ |
|--------|-------------|-------------|
| neumann | 分析結果・インサイト（差異分析の精度向上） | KPI 異常検知（分析リクエストの自動起動） |
| agentic-ai-sales | 予測モデルの出力（リードスコアリング精緻化） | Win/Loss データ（受注/失注の要因分析入力） |
| ai-executive-dashboard | データ品質レポート（データ基盤の健全性） | 市場トレンド（外部要因の分離に活用） |
| agentic-ai-hr | セグメンテーション結果（制度効果分析） | 等級・評価データ（人材分析入力） |

## 1. WHO - ユーザー定義

### 1.1 プライマリユーザー

| ペルソナ | 特徴 | ゴール | ペイン |
|---------|------|-------|-------|
| データアナリスト | 分析チームの中核メンバー。SQL/Python を使いこなす | 分析の品質と速度を両立したい | 定型的なデータ準備と品質チェックに時間を取られる |
| ビジネスアナリスト | 事業部門に所属。ビジネス知識は深いがコーディングは限定的 | データに基づく意思決定を推進したい | 分析の技術的なハードルが高い |
| 経営企画担当 | 経営戦略の立案・評価を担当 | 経営判断に必要な分析を迅速に入手したい | 分析チームのキャパシティがボトルネック |

### 1.2 セカンダリユーザー

| ペルソナ | 役割 |
|---------|------|
| 分析マネージャー | 分析チームの管理。分析品質の監督、リソース配分の判断 |
| データエンジニア | データパイプラインの構築・運用。品質ルールの技術的実装 |
| 経営層（Governor） | 分析結果に基づく意思決定。インサイトとアクション提案の承認 |

### 1.3 自動連携ユーザー（他 Agentic AI）

| 連携元 | トリガー | 分析リクエストの例 |
|--------|---------|-----------------|
| neumann | KPI 異常検知 | 「売上が前月比 -24%。要因を分析せよ」 |
| agentic-ai-sales | 失注パターン検出 | 「Q3 の失注率が上昇。競合要因を統計分析せよ」 |
| agentic-ai-hr | 組織課題シグナル | 「特定部門の離職率が閾値を超過。要因を分析せよ」 |

## 2. WHAT - データ分析サイクルの定義

### 2.1 分析サイクルの 9 工程

| # | 工程 | 最適化目標 | KPI |
|---|------|-----------|-----|
| 1 | 問い定義 | ビジネスインパクトが最大の問いを選定 | 問い→アクション転換率、インパクト見積もり精度 |
| 2 | データ収集・統合 | 必要なデータを最短で取得・結合 | データ準備時間、ソースカバレッジ率 |
| 3 | データ品質確保 | 品質問題の再発防止 | 品質問題の再発率、クレンジング自動化率 |
| 4 | 探索的分析 | 見落としのない網羅的探索 | パターン発見数、false discovery rate |
| 5 | 仮説検証・モデリング | 統計的に妥当な結論 | モデル精度、予測の的中率 |
| 6 | インサイト抽出 | 意思決定者に伝わるインサイト | インサイト→アクション採択率 |
| 7 | レポーティング | 受け手に最適化された報告 | レポート品質スコア、修正率 |
| 8 | 意思決定支援 | 具体的で実行可能なアクション提案 | 提案採択率、アクション完了率 |
| 9 | 効果測定 | 施策の効果を定量追跡 → 次の問いに反映 | 予測効果 vs 実績の乖離率 |

### 2.2 分析の種類（手法分類）

| 分析手法 | 用途 | 典型的な問い |
|---------|------|------------|
| 記述統計（Descriptive） | 現状把握 | 「今の状況はどうなっているか？」 |
| 原因分析（Diagnostic） | 要因特定 | 「なぜそうなったのか？」 |
| 予測（Predictive） | 将来推定 | 「今後どうなるか？」 |
| 処方的分析（Prescriptive） | アクション提案 | 「何をすべきか？」 |
| 因果推論（Causal Inference） | 因果関係の特定 | 「X が Y の原因か？」 |
| セグメンテーション（Clustering） | グループ分け | 「どのようなグループがあるか？」 |
| 異常検知（Anomaly Detection） | 逸脱検出 | 「通常と異なる動きはあるか？」 |

### 2.3 エージェント構成（概要）

8 つの専門エージェントが分析サイクルの各工程を担当する。

```
【問い定義 + データ準備】
  Agent 1: QuestionArchitectAgent      分析リクエスト→構造化された問い
  Agent 2: DataIntegrationAgent        データ収集・統合・パイプライン構築

【探索 + 分析】
  Agent 3: DataQualityAgent            データ品質チェック・クレンジング
  Agent 4: ExploratoryAnalysisAgent    EDA・パターン発見・仮説生成
  Agent 5: StatisticalModelingAgent    統計モデリング・予測・因果推論

【インサイト + 実行】
  Agent 6: InsightGeneratorAgent       分析結果→インサイト→アクション提案
  Agent 7: ReportGeneratorAgent        レポート・ダッシュボード・可視化

【内省】
  Agent 8: AnalysisCycleReflectionAgent  分析精度追跡 + 手法改善 + 効果測定
```

詳細は [01_concept.md](01_concept.md) および [02_architecture.md](02_architecture.md) を参照。

### 2.4 想定ユースケース

#### ユースケース 1: 経営層からの売上分析依頼

```
トリガー: 経営層が「最近の売上が伸び悩んでいる」と依頼
  │
  ▼ Agent 1: 問い構造化
  主問い: 売上成長率の鈍化要因は何か
  副問い: 新規獲得/ARPU/解約の分解
  │
  ▼ Agent 2-5: データ収集→品質チェック→EDA→モデリング
  発見: 新規顧客獲得数が Q3 から 20% 減少、主要因は CPA 上昇
  │
  ▼ Agent 6: インサイト抽出
  So What: 年間売上目標に対して ¥120M のギャップが発生する見込み
  Now What: ROI 低下チャネルの停止、高 ROI チャネルへの再配分
  │
  ▼ Agent 7: レポート生成 → 経営層向けサマリー
  │
  ▼ Agent 8: 効果測定（四半期後）
  予測: 新規獲得 +12% → 実績: +8% → 学習: 紹介チャネルは効果発現に 2 四半期要
```

#### ユースケース 2: neumann からの自動連携

```
トリガー: neumann が KPI 異常を検知（売上 -24%）
  → agentic-ai-analysis に分析リクエストを自動発行
  → 分析サイクルが自律的に回る
  → インサイトを neumann に返却
  → neumann の差異分析レポートに統計的根拠が追加される
```

#### ユースケース 3: agentic-ai-sales からの深掘り依頼

```
トリガー: agentic-ai-sales が Q3 の失注率上昇パターンを検出
  → 「価格」「機能」「競合」の影響度を因果推論モデルで定量化
  → 「競合 X の新機能リリースが失注率 +12% の主要因」と特定
  → 結果を agentic-ai-sales の商談戦略エージェントに返却
```

## 3. Project Status

### 3.1 現在のフェーズ

**Planning** — 設計ドキュメントの策定段階。実装は開始されていない。

### 3.2 ロードマップ

| フェーズ | 期間（見込み） | 内容 | 状態 |
|---------|-------------|------|------|
| Phase 1 | 3-4 週 | 分析エンジンのコア（問い定義 + EDA + レポート生成） | 計画中 |
| Phase 2 | 3-4 週 | データ品質 + 統計モデリング | 計画中 |
| Phase 3 | 2-3 週 | インサイト生成 + 意思決定支援 | 計画中 |
| Phase 4 | 2-3 週 | Reflection（分析精度追跡 + 手法改善） | 計画中 |
| Phase 5 | 2 週 | 共有記憶基盤接続（neumann, agentic-ai-sales 等） | 計画中 |

### 3.3 成功指標（計画値）

**分析品質メトリクス**

| 指標 | 測定方法 | 目標値 |
|------|---------|-------|
| 問い→アクション転換率 | 分析完了後にアクションが実行された割合 | > 70% |
| 予測精度 | 予測効果 vs 実績の一致度 | > 60% |
| インサイト採択率 | 提案されたアクションが承認された割合 | > 50% |

**効率メトリクス**

| 指標 | 測定方法 | 目標値 |
|------|---------|-------|
| データ準備時間 | リクエストからデータ準備完了まで | 現状の 1/3 |
| 品質問題再発率 | 既知の品質問題が再発した割合 | < 10% |
| 分析サイクル完了時間 | リクエストからレポート完了まで | 現状の 1/2 |
