# Agentic AI HR System -- コンセプト設計

> 本ドキュメントは Agentic AI HR System の設計コンセプトを記述する。プロジェクト概要は [00_overview.md](00_overview.md) を参照。

## 1. 目的: 人事制度の Single Source of Truth

### 1.1 SSOT とは何か

Single Source of Truth（SSOT）とは、ある情報について「唯一の正しい参照元」を確立する設計パターンである。本プロジェクトでは、人事制度に関する以下の情報について SSOT を提供する。

| 情報 | 従来の状態 | SSOT 化後 |
|---|---|---|
| 等級の定義と期待行動 | 人事担当者の暗黙知、散在する Word/Excel | `definition/grades/G*.md` |
| コンピテンシーの体系と詳細 | 評価シートに埋め込まれた断片的な記述 | `definition/competency_matrix_v3.md` |
| 評価の判定ロジック | マネージャーごとの独自解釈 | `definition/evaluation_system_v1.md` |
| 報酬テーブルと運用ルール | 経理部門のスプレッドシート | `definition/compensation_system_v1.md` |

### 1.2 SSOT が解決する問題

**問題 1: 情報の不整合**

等級定義が複数の場所に存在し、それぞれが微妙に異なる記述を持つ場合、「どれが正しいのか」を判断するコストが発生する。本プロジェクトに定義を一元化することで、コンシューマプロジェクトは常に最新かつ正しい定義を参照できる。

**問題 2: AI エージェントの誤解釈**

AI エージェントが人事制度を参照する際、情報源が複数あると、矛盾する情報をどう扱うかという判断が必要になる。SSOT により、AI エージェントは単一の参照元を信頼すればよい。

**問題 3: 制度改定の追跡不能**

制度が改定された際、「何を」「なぜ」「いつ」変えたかの記録がないと、改定の意図が失われる。Git ベースの管理により、全ての変更がコミット履歴として永続化される。

## 2. 定義レイヤーと実装レイヤー

### 2.1 二層分離の設計思想

本プロジェクトは意図的に「定義」と「実装」を分離している。

| 観点 | 定義レイヤー（本プロジェクト） | 実装レイヤー（agentic-ai-hr） |
|---|---|---|
| 責務 | 「制度はこうあるべき」を定義する | 定義を消費し、クライアント企業向けの制度設計を AI で支援する |
| 成果物 | 構造化された Markdown ドキュメント | Python アプリケーション（FastAPI, Gemini, PostgreSQL 等） |
| 変更頻度 | 低い（半期〜年次の制度レビュー時） | 高い（機能開発・バグ修正の日常的な変更） |
| 編集者 | 人事マネージャー、システムアーキテクト | ソフトウェアエンジニア |
| バージョニング | ファイル名にバージョン番号（例: `v3`） | Git タグ・リリース |

### 2.2 分離の利点

**1. 変更の独立性**

定義の変更と実装の変更を独立に管理できる。等級定義の文言修正のために、アプリケーションのリリースサイクルに巻き込まれる必要がない。

**2. 複数コンシューマへの対応**

定義が独立していることで、agentic-ai-hr だけでなく neumann や ai-executive-dashboard からも同一の定義を参照できる。実装レイヤーに定義が埋め込まれていた場合、他プロジェクトからの参照が困難になる。

**3. 専門性の分離**

人事マネージャーは Markdown の編集に集中でき、Python のコードベースに触れる必要がない。逆に、エンジニアは定義の妥当性を判断する必要がない。

### 2.3 コンシューマとの接続

コンシューマプロジェクトは本プロジェクトの定義ファイルを以下の方法で参照する。

```
参照パターン:
  1. 同一リポジトリ内のパス参照（Neural Organization monorepo）
     sub-pj/agentic-ai-hr → sub-pj/agentic-ai-hr-system/definition/

  2. AI エージェントによるドキュメント読み込み
     エージェントが definition/ 配下のファイルを読み込み、
     コンテキストとして利用する
```

## 3. 定義の構造

### 3.1 ディレクトリ構成

```
definition/
  grades/              # 等級別の詳細定義
    G1_member.md       # 各ファイルが 1 等級の全コンピテンシーを定義
    G2_leader_support.md
    ...
    G8_director.md
  competency_matrix_v3.md   # 全等級 x 全コンピテンシーの俯瞰マトリクス
  evaluation_system_v1.md   # 評価の判定ロジックと運用サイクル
  compensation_system_v1.md # 報酬テーブルと運用ルール

operation/
  operation_manual_v1.md    # 制度の導入・運用の実務指針

skills/
  hr-system-architect.md    # AI スキル定義
```

### 3.2 定義ファイルの設計原則

定義ファイルは以下の原則に従って設計されている。

**原則 1: 人間と AI の双方が読める**

定義ファイルは Markdown で記述する。テーブル形式を積極的に活用し、構造化された情報を提供する。自然言語の長文ではなく、簡潔な行動記述を用いる。

**原則 2: 冗長性よりも参照の明確さ**

同一の情報が複数のファイルに存在する場合がある（例: 等級定義は grades/ の個別ファイルにも competency_matrix にも存在する）。これは意図的な設計であり、参照コンテキストに応じた最適なビューを提供するためである。

- `grades/G3_leader.md`: G3 リーダーの全コンピテンシーを一覧する場合に使用
- `competency_matrix_v3.md`: 全等級を横断して特定のコンピテンシーを比較する場合に使用

**原則 3: バージョンをファイル名に含める**

ファイル名にバージョン番号を含めることで、参照元が「どのバージョンの定義を使っているか」を明示的に管理できる。

### 3.3 等級定義の構造

各等級ファイル（`grades/G*.md`）は統一された構造を持つ。

```markdown
# {等級名} -- コンピテンシー定義

> **Scope of Impact**: {責任範囲}
>
> {等級のミッション}

## 1. 思考・構想 (Thinking)
### A. 課題設定 -- {サブタイトル}
{行動記述}

### B. 構造化・計画 -- {サブタイトル}
{行動記述}

## 2. 実行・行動 (Action)
### C. スピード -- {サブタイトル}
{行動記述}

### D. 品質 -- {サブタイトル}
{行動記述}

## 3. 影響・対人 (Relation)
### E. 受信・傾聴 -- {サブタイトル}
{行動記述}

### F. 発信・伝達 -- {サブタイトル}
{行動記述}
```

この構造の特徴は以下の通りである。

| 要素 | 役割 |
|---|---|
| Scope of Impact | 等級の責任範囲を一文で定義。等級間の差異を最も端的に表現する |
| ミッション | Scope of Impact を補足する 1-2 文の説明 |
| 3 領域 x 2 要素 | 全等級で統一された 6 コンピテンシーの枠組み |
| サブタイトル | 各等級のコンピテンシーの核心を 1 フレーズで表現（例: G1-A は「違和感のアラート」、G5-A は「部門方針の策定」） |
| 行動記述 | 具体的な期待行動を自然言語で記述 |

### 3.4 コンピテンシーマトリクスの構造

`competency_matrix_v3.md` は、全等級を横断した俯瞰ビューを提供する。

**セクション 1: 制度概要**
- コンピテンシー体系（3 領域 x 6 要素の定義）
- グレードとスコープの対応表

**セクション 2: グレード別コンピテンシー定義詳細**
- 領域ごとにテーブル形式で全等級のコンピテンシーを一覧
- 2.1 思考・構想: A. 課題設定 / B. 構造化・計画
- 2.2 実行・行動: C. スピード / D. 品質
- 2.3 影響・対人: E. 受信・傾聴 / F. 発信・伝達

このマトリクス形式により、以下のユースケースに対応する。

- 「G3 と G4 の課題設定能力はどう違うのか」を横比較できる
- 「スピード（C）は等級が上がるとどう変化するか」を縦断的に確認できる

### 3.5 評価制度の構造

`evaluation_system_v1.md` は、昇格判定のロジックと運用サイクルを定義する。

**卒業要件方式の設計思想**

本制度は「相対評価」ではなく「絶対評価」を採用している。他者との比較ではなく、現グレードの要件を完全に満たしているかで判定する。「卒業」というメタファーは、「一定の基準を達成した者が次のステージに進む」ことを意味する。

**3 ステップゲート**

昇格には 3 つの独立したゲートをすべて通過する必要がある。

| ゲート | 名称 | 測定対象 | メタファー |
|---|---|---|---|
| Step 1 | コンピテンシー評価 | 能力の発揮度（5 段階） | 卒業要件: 実力があるか |
| Step 2 | 成果評価 | 目標の達成度（○/△/×） | チケット: 結果を出しているか |
| Step 3 | バリュー評価 | 行動規範の体現度（○/△/×） | ゲートキーパー: 組織人としてふさわしいか |

**採点基準**

| 点数 | 判定 | 意味 |
|---|---|---|
| 5 | Outstanding | 上位グレードの要件すら一部満たしている |
| 4 | Exceed | 卒業レベル。現グレードの期待を常に上回る |
| 3 | Meet | 充足。現グレードの期待を概ね満たしている |
| 2 | Below | 不足。周囲のサポートを頻繁に必要とする |
| 1 | Unsatisfactory | 未達。改善の兆しが見られない |

### 3.6 報酬制度の構造

`compensation_system_v1.md` は、報酬テーブルと運用ルールを定義する。

**3 つのポリシー**

| ポリシー | 内容 |
|---|---|
| 完全等級連動型 | 年収はグレードと役職で機械的に決定。属人的調整を排除 |
| 年俸制 | 賞与なし。約束された年収を 12 分割で支給 |
| 洗い替え型 | 毎回の評価でステップが変動。過去ではなく「今の貢献度」で決定 |

**報酬テーブル（抜粋）**

各グレードに 5 ステップ（S1-S5）を設定し、レンジを明確にする。

| グレード | S1 (Min) | S3 (Mid) | S5 (Max) |
|---|---|---|---|
| G8 取締役級 | 1,800万 | 2,000万 | 2,200万 |
| G5 部長級 | 850万 | 920万 | 990万 |
| G3 リーダー | 550万 | 610万 | 670万 |
| G1 メンバー | 300万 | 320万 | 340万 |

**昇給・降給ルール**

| 評価 | ステップ変動 |
|---|---|
| Outstanding (5) | +2 ステップ |
| Exceed (4) | +1 ステップ |
| Meet (3) | ステイ（維持） |
| Below (2) 以下 | -1 ステップ（または降格検討） |

## 4. コンシューマによる利用パターン

### 4.1 agentic-ai-hr（制度設計 AI）

agentic-ai-hr は本プロジェクトの最も重要なコンシューマである。以下のように定義を利用する。

**参照する定義**:
- 等級定義（G1-G8）: リファレンスフレームワークとして、クライアント企業の等級制度設計の基盤にする
- コンピテンシーマトリクス: クライアント企業の業種・文化に応じてカスタマイズする際のベースラインとする
- 評価制度: 卒業要件方式のロジックをクライアント企業に提案する際のテンプレートとする
- 報酬制度: 報酬テーブルの構造（5 ステップ制、等級連動型）を設計の参考にする

**利用フロー**:
```
1. クライアント企業の情報収集
2. 本プロジェクトの定義を読み込み、コンテキストに設定
3. クライアント企業の特性に応じて定義をカスタマイズ
4. カスタマイズした制度案を Human-in-the-Loop で承認
5. 最終的な制度ドキュメントを生成
```

### 4.2 neumann（報告品質管理 AI）

neumann は等級定義を参照し、レポートの品質基準を動的に調整する。

**参照する定義**:
- 等級定義（特に「構造化・計画」「発信・伝達」のコンピテンシー）
- コンピテンシーマトリクス（等級間の能力差異）

**利用例**:
- G3 リーダーのレポートには「チーム課題の発見」「実行プランの具体化」レベルの構造化を期待
- G5 部長のレポートには「部門方針の策定」「リソース戦略」レベルの戦略性を期待

### 4.3 ai-executive-dashboard（経営ダッシュボード）

ai-executive-dashboard は評価基準を参照し、組織全体の能力分布を可視化する。

**参照する定義**:
- 評価制度（採点基準、昇格要件）
- 等級体系（G1-G8 の構造）

**利用例**:
- 組織全体の等級分布と、各等級のコンピテンシー充足率をレポートに反映
- 「卒業要件」の達成率推移から、組織の成長トレンドを分析

## 5. 設計原則

### 5.1 等級設計の原則

**原則 1: Scope of Impact による定義**

等級を「能力の高さ」ではなく「責任範囲の広さ」で定義する。これにより、等級間の差異が直感的に理解でき、評価の曖昧さを減らせる。

| 等級 | Scope | キーワード |
|---|---|---|
| G1 | タスク | 指示された業務を完遂する |
| G2 | 自律 | 自走し、技術面でリーダーを支える |
| G3 | チーム | チームの成果に責任を持つ |
| G4 | チーム間 | 複数チームを横断して調整する |
| G5 | 部門 | 部門の戦略と KPI に責任を持つ |
| G6 | 事業部 | 事業全体の戦略と組織力に責任を持つ |
| G7 | 全社 | 全社の経営資源と組織文化に責任を持つ |
| G8 | 業界・社会 | 業界構造や社会的プレゼンスに影響を与える |

**原則 2: コンピテンシーの普遍性**

6 つのコンピテンシー（A-F）は全等級で共通である。等級が上がっても「新しいコンピテンシー」が追加されるのではなく、同じコンピテンシーが「より広い Scope」で発揮されることを求める。

例: 「A. 課題設定」の等級間比較

| 等級 | 課題設定の内容 |
|---|---|
| G1 | 違和感を報告する |
| G3 | チーム課題を自ら発見する |
| G5 | 部門方針を策定する |
| G7 | 事業の創造と破壊を決断する |

**原則 3: 役職と等級の分離**

等級は「能力の認定」であり、役職は「役割の任命」である。G5 の能力を持つ者が必ずしも部長職に就くわけではなく、逆もまた然りである。報酬は「等級（基本年俸） + 役職（手当）」の二層で構成される。

### 5.2 評価設計の原則

**原則 1: 絶対評価（卒業要件方式）**

相対評価（同僚との比較）ではなく、絶対評価（基準への到達度）を採用する。「同期の中で上位 20%」ではなく「G3 の卒業要件を満たしているか」で判定する。

**原則 2: Fact ベースの評価**

「頑張った」「やる気がある」といった印象ではなく、記録された事実（Fact）に基づいて評価する。月次 1on1 でのリアルタイムフィードバックと Fact の蓄積が、期末評価の精度を担保する。

**原則 3: サプライズ禁止**

期末評価の結果が被評価者にとって「サプライズ」にならないよう、中間レビュー（3 ヶ月時点）で着地見込みを仮伝えする。

### 5.3 報酬設計の原則

**原則 1: 透明性**

報酬テーブルを全社員に公開する。「自分が今いくらもらっていて、次の等級に上がればいくらになるか」が誰でもわかる。

**原則 2: シンプルさ**

報酬の決定要素を「等級」と「役職」の 2 つに限定する。入社経緯、前職給与、勤続年数などの属人的要素を排除する。

**原則 3: 洗い替え**

過去の実績ではなく「今の貢献度」で報酬を決定する。評価が下がればステップも下がる。これにより、「昔は活躍していたが今は...」という問題を構造的に防ぐ。

## 6. 運用マニュアルの位置づけ

### 6.1 制度設計と運用の関係

制度をいくら精緻に設計しても、運用が形骸化すれば意味がない。[運用マニュアル](operation/operation_manual_v1.md) は、制度を「作る」フェーズから「使い続ける」フェーズへの橋渡しを行う。

### 6.2 運用の 5 フェーズ

| Phase | 名称 | 目的 |
|---|---|---|
| Phase 0 | 準備・合意 | マネージャーを「制度の推進者」に変える |
| Phase 1 | 展開 | 全社員への透明な情報公開 |
| Phase 2 | 導入・移行 | 納得感のある初期格付けと激変緩和 |
| Phase 3 | 運用サイクル | 評価を「日常のマネジメント」にする |
| Phase 4 | メンテナンス | 制度の形骸化を防ぎ、アップデートし続ける |

### 6.3 重要なポイント

- **マネージャーのアンバサダー化**: 一般社員への公開前に、マネージャーを「制度の理解者・推進者」に変える
- **初期格付けのキャリブレーション**: 全部門のマネージャーが集まり、横比較で基準を統一する
- **激変緩和措置**: 制度移行で年収が下がる社員には調整給を設け、1-2 年の猶予期間を設ける
- **異議申し立て**: 納得がいかない場合に再審査を要求できるプロセスを設ける

## 7. AI スキルの位置づけ

### 7.1 hr-system-architect スキル

[skills/hr-system-architect.md](skills/hr-system-architect.md) は、AI エージェントが人事制度の設計・レビューを支援する際のスキル定義である。

**スキルが提供するフレームワーク**:

| フレームワーク | 内容 |
|---|---|
| 3 つの究極の問い | 評価源泉・報酬ロジック・マネジメント定義を決定する |
| コンピテンシーの 3 軸 | Thinking / Action / Relation の体系 |
| Scope ベースの等級定義 | 責任範囲による等級の定義方法 |
| 3 ステップゲート | 卒業要件方式の評価ロジック |
| 報酬ロジック | 等級連動・5 ステップ・洗い替えの設計 |
| 運用エクセレンス | ミッションレター・キャリブレーション・Appeal プロセス |

### 7.2 スキルと定義の関係

hr-system-architect スキルは、本プロジェクトの定義をベストプラクティスの実例として参照する。スキルが「等級制度はこう設計すべき」というフレームワークを提供し、definition/ 配下のファイルがそのフレームワークの具体的な実装例となる。

```
hr-system-architect スキル          definition/ 配下のファイル
（設計方法論・フレームワーク）        （フレームワークの具体的実装）
┌─────────────────────────┐      ┌──────────────────────────┐
│ "Scope ベースで等級を    │      │ G1: タスク               │
│  定義すべき"             │─────▶│ G3: チームリード          │
```

## 8. 定義の変更管理

### 8.1 バージョニング方針

| 変更種別 | 例 | バージョン変更 | コンシューマへの影響 |
|---|---|---|---|
| 文言の明確化 | 行動記述の表現改善 | なし | なし（非破壊的） |
| 定義の追加 | 新しい等級 G9 の追加 | メジャー | 要対応（参照範囲の拡大） |
| 構造の変更 | コンピテンシーの要素追加 | メジャー | 要対応（パース処理の変更） |
| テーブルの更新 | 報酬テーブルの金額改定 | マイナー | 通知のみ（構造は不変） |

### 8.2 変更プロセス

```
1. 変更の起案（人事マネージャーまたはシステムアーキテクト）
2. コンシューマへの影響分析
3. 破壊的変更の場合: コンシューマプロジェクトとの事前調整
4. 定義ファイルの更新
5. バージョン番号の更新（該当する場合）
6. Git コミット（変更理由をコミットメッセージに記録）
7. コンシューマプロジェクトへの通知
```

### 8.3 制度レビューサイクル

定義の見直しは以下のタイミングで実施する。

| タイミング | 内容 |
|---|---|
| 半期（評価終了後） | 評価運用で発覚した定義の曖昧さ・矛盾を微修正 |
| 年次 | 報酬テーブルの市場整合性確認、コンピテンシー定義の妥当性レビュー |
| 随時 | コンシューマプロジェクトからのフィードバックに基づく改善 |
