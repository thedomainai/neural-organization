# Agentic AI HR System -- プロジェクト概要

> 本ドキュメントは Agentic AI HR System プロジェクトの全体像を示す。設計コンセプトは [01_concept.md](01_concept.md) を参照。本プロジェクトは定義レイヤーのため、02_architecture.md および 03_implementation.md は存在しない。

## 1. WHY -- なぜこのプロジェクトが必要か

### 1.1 背景

人事制度（等級・評価・報酬）は組織運営の根幹である。しかし、制度の設計根拠や判断基準は多くの場合、以下の問題を抱えている。

| 問題 | 影響 |
|---|---|
| 暗黙知への依存 | 制度の設計意図や判断基準が特定の担当者の頭の中にしか存在しない |
| 断片的な管理 | 等級定義、評価基準、報酬テーブルが別々のファイル・ツールに散在し、整合性が保てない |
| 人間専用の設計 | 制度定義が自然言語の長文で書かれており、AI エージェントが正確に参照・解釈できない |
| バージョン管理の欠如 | 制度改定の履歴が追跡できず、「なぜこう変わったか」が失われる |

### 1.2 ビジョン

Agentic AI HR System は、人事制度のマスターデータを構造化されたドキュメントとして形式知化し、人間と AI エージェントの双方が正確に参照できる **Single Source of Truth（SSOT）** を提供する。

本プロジェクトは実装コードを持たない **定義レイヤー** である。制度の「定義」に特化することで、以下を実現する。

- 人事制度の全定義を一箇所に集約し、整合性を担保する
- 構造化された Markdown により、人間が読みやすく AI が解釈しやすいフォーマットを提供する
- Git によるバージョン管理で、制度改定の全履歴を追跡可能にする

### 1.3 核となる価値提案

1. **形式知化**: 暗黙知として担当者の頭の中にあった制度設計の根拠・判断基準を、構造化ドキュメントとして永続化する
2. **SSOT の確立**: 等級・コンピテンシー・評価・報酬の全定義を一箇所に集約し、複数のコンシューマプロジェクトが同一の定義を参照する
3. **AI 可読性**: テーブル形式・一貫したセクション構造により、AI エージェントが定義を正確にパースし、業務判断に活用できる
4. **変更追跡**: Git ベースのバージョン管理により、制度改定の「何を」「なぜ」「いつ」変えたかを完全に記録する

### 1.4 定義レイヤーと実装レイヤーの関係

本プロジェクトは Neural Organization エコシステムにおいて、**定義レイヤー** として位置づけられる。

```
定義レイヤー（本プロジェクト）        実装レイヤー
┌─────────────────────────┐     ┌──────────────────────┐
│ agentic-ai-hr-system    │     │ agentic-ai-hr        │
│                         │     │                      │
│ - 等級定義 (G1-G8)      │────▶│ マルチエージェント    │
│ - コンピテンシー体系     │     │ 制度設計システム      │
│ - 評価制度              │     │                      │
│ - 報酬制度              │     │ （FastAPI, Gemini,   │
│ - 運用マニュアル         │     │  PostgreSQL 等）     │
└─────────────────────────┘     └──────────────────────┘
         SSOT                     コンシューマ
```

- **定義レイヤー**: 「制度はこうあるべき」を定義する。コードは持たない
- **実装レイヤー**: 定義を消費し、クライアント企業向けの制度設計を AI で支援するシステム

## 2. WHO -- 誰のためのプロジェクトか

### 2.1 コンシューマ（定義の利用者）

本プロジェクトの定義を参照するシステム・プロジェクト。

| コンシューマ | 利用目的 |
|---|---|
| **agentic-ai-hr** | 等級・コンピテンシー・評価・報酬の全定義を参照し、クライアント企業向けの制度設計に活用する。最も重要なコンシューマ |
| **neumann** | 等級定義・コンピテンシー基準を参照し、報告品質の評価基準を等級に応じて動的に調整する |
| **ai-executive-dashboard** | 評価基準を参照し、組織能力の経時的変化をレポートに反映する |

### 2.2 エディタ（定義の編集者）

本プロジェクトの定義を作成・更新する人間。

| エディタ | 役割 |
|---|---|
| **人事マネージャー** | 等級定義・評価基準・報酬テーブルの実質的な決定権者。制度の設計意図を理解し、定義の妥当性を判断する |
| **システムアーキテクト** | 定義のフォーマット・構造を設計し、AI エージェントからの可読性を担保する。コンシューマプロジェクトとの接続仕様を管理する |
| **hr-system-architect スキル** | AI スキルとして、人事制度設計のベストプラクティスに基づいた定義の作成・レビューを支援する |

### 2.3 利用パターン

**コンシューマが定義を参照するケース**:
- agentic-ai-hr のエージェントがクライアント企業の等級制度を設計する際、本プロジェクトのコンピテンシーマトリクスをリファレンスフレームワークとして参照する
- neumann がレポートの品質基準を設定する際、対象者の等級に応じたコンピテンシー定義を参照する

**エディタが定義を更新するケース**:
- 半期ごとの制度レビューで評価基準を微修正する場合、`definition/evaluation_system_v1.md` を更新し、バージョンを v2 に上げる
- 新たなグレード（例: G9）を追加する場合、`definition/grades/` に新ファイルを作成し、`competency_matrix` を更新する

## 3. WHAT -- 何を定義しているか

### 3.1 定義ドメイン一覧

本プロジェクトは以下の 4 つのドメインにわたる人事制度定義を管理する。

| # | ドメイン | 概要 | ファイル |
|---|---|---|---|
| 1 | 等級制度 | G1（メンバー）から G8（取締役級）までの 8 等級。各等級の責任範囲（Scope of Impact）と期待行動を定義 | `definition/grades/G*.md` |
| 2 | コンピテンシー体系 | 3 領域（思考・実行・対人）x 2 要素の計 6 コンピテンシー。全 8 等級 x 6 要素 = 48 のコンピテンシー定義 | `definition/competency_matrix_v3.md` |
| 3 | 評価制度 | 卒業要件方式による昇格判定。コンピテンシー評価（5 段階）、成果評価、バリュー評価の 3 ステップゲート | `definition/evaluation_system_v1.md` |
| 4 | 報酬制度 | 完全等級連動型の年俸制。各等級 5 ステップ（S1-S5）の報酬テーブルと役職手当 | `definition/compensation_system_v1.md` |

### 3.2 等級体系（G1-G8）

等級は「役職」ではなく「責任範囲（Scope of Impact）」で定義される。

| 等級 | 役割名称 | 責任範囲 | 基本年俸レンジ |
|---|---|---|---|
| G8 | 取締役級 | 業界・社会 | 1,800万 - 2,200万 |
| G7 | 執行役員級 | 全社・経営 | 1,400万 - 1,700万 |
| G6 | 本部長級 | 本部・事業部 | 1,100万 - 1,300万 |
| G5 | 部長級 | 部・機能 | 850万 - 990万 |
| G4 | マネージャーサポート | プロジェクト群・チーム間 | 700万 - 820万 |
| G3 | リーダー | チームリード | 550万 - 670万 |
| G2 | リーダーサポート | 自律・技術牽引 | 360万 - 460万 |
| G1 | メンバー | 担当タスク・定型業務 | 300万 - 340万 |

### 3.3 コンピテンシー体系

6 つのコンピテンシー要素が 3 領域に分類される。

| 領域 | コンピテンシー | 定義 |
|---|---|---|
| 思考・構想 (Thinking) | A. 課題設定 (Issue Finding) | 状況を観察し、「何を解決すべきか」を定義する力 |
| | B. 構造化・計画 (Structuring) | 複雑な事象を整理し、解決までの道筋を描く力 |
| 実行・行動 (Action) | C. スピード (Speed) | 意思決定から完了までのサイクルを回す速さ |
| | D. 品質 (Quality) | アウトプットの正確性、完成度、持続可能性 |
| 影響・対人 (Relation) | E. 受信・傾聴 (Listening) | 他者の発言・感情・文脈を深く正確に読み取る力 |
| | F. 発信・伝達 (Delivery) | 意図やビジョンを届け、納得と行動を引き出す力 |

各コンピテンシーは等級ごとに具体的な行動記述が定義されている（全 48 定義）。

### 3.4 評価制度

昇格は「卒業要件方式」で判定される。以下の 3 ゲートをすべてクリアする必要がある。

| ステップ | 名称 | 判定基準 | 昇格要件 |
|---|---|---|---|
| Step 1 | コンピテンシー評価（卒業要件） | 1-5 点の 5 段階 | 3 領域すべて 4 点（Exceed）以上 |
| Step 2 | 成果評価（チケット） | ○/△/× | ○（達成）以上 |
| Step 3 | バリュー評価（ゲートキーパー） | ○/△/× | ○（体現）以上 |

運用サイクル: 月次 1on1 → 中間レビュー（3 ヶ月） → 期末評価（年 2 回: 6 月、12 月）

### 3.5 報酬制度

完全等級連動型の年俸制。属人的な調整を排除し、透明性を最大化する。

- **構成**: 基本年俸（等級連動） + 役職手当（ポジション連動）
- **ステップ**: 各等級に S1-S5 の 5 段階。評価に基づき毎期洗い替え
- **役職手当**: 本部長 +10 万/月、部長 +7 万/月、リーダー +3 万/月

### 3.6 補助ドキュメント

| ドキュメント | 概要 |
|---|---|
| [運用マニュアル](operation/operation_manual_v1.md) | 制度導入の 5 フェーズ（準備→展開→移行→運用→メンテナンス）の実務指針 |
| [HR System Architect スキル](skills/hr-system-architect.md) | AI スキル定義。人事制度設計のフレームワーク・プロセス・トーンを規定 |

### 4.2 更新方針

- 定義ファイルはファイル名にバージョン番号を含む（例: `competency_matrix_v3.md`）
- 破壊的変更（等級構造の変更、コンピテンシー要素の追加/削除等）は、全コンシューマプロジェクトとの事前調整が必要
- 非破壊的変更（行動記述の明確化、文言修正等）は随時実施可能

## 5. Neural Organization における位置づけ

### 5.1 レイヤーの役割

本プロジェクトは Neural Organization エコシステムにおいて、人事ドメインの **定義レイヤー** として機能する。実装レイヤー（agentic-ai-hr）やその他のプロジェクトに対して、人事制度のマスターデータを提供する。

### 5.2 他プロジェクトとの関係

| 関連プロジェクト | 関係性 |
|---|---|
| **agentic-ai-hr** | 最も重要なコンシューマ。本プロジェクトの定義を参照し、クライアント企業向けの制度設計を AI で支援する |
| **neumann** | 等級定義・コンピテンシー基準を参照し、報告品質の評価基準を等級に応じて動的調整する |
| **ai-executive-dashboard** | 評価基準を参照し、組織能力の経時的変化をレポートに反映する |

### 5.3 データフロー

```
本プロジェクト（定義レイヤー）
    │
    ├──▶ agentic-ai-hr: 等級定義・コンピテンシー・評価基準・報酬テーブル
    ├──▶ neumann: 等級定義・コンピテンシー基準
    └──▶ ai-executive-dashboard: 評価基準
```
