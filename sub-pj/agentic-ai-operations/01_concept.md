# Agentic AI Operations: コンセプト

> ステータス: **PoC（Proof of Concept）**
> 最終更新: 2026-02

## 1. Operations Cycle の定義

### 1.1 Operations Cycle とは

Operations Cycle は、ビジネス運営上の情報を**収集・分類・統合・配信**し、意思決定を継続的に支援するサイクルである。

従来の「レポート作成」が人間の手作業に依存する一方向的なプロセスであるのに対し、Operations Cycle は AI を用いて自律的に回り続けるフィードバックループとして設計される。

```
 Collect          Classify           Synthesize         Deliver
 (収集)            (分類)              (統合)              (配信)
                                                          │
 RSS/Blog    →   AI Classification  →  Trend Analysis  → Report
 Feeds            - Category            - Executive       - Dashboard
 Scrapers         - Impact Level          Summary         - Weekly
                  - Relevance           - Category          Briefing
                    Score                 Insights        - Actionable
                                        - Action Items      Insights
                                                          │
                                                          ▼
                                                       Feedback
                                                       (フィードバック)
                                                          │
                                         人間のレビュー・修正・評価
                                                          │
                                                          ▼
                                            分類精度・レポート品質の改善
                                                    (将来実装)
```

### 1.2 Operations Cycle の 4 フェーズ

#### Phase 1: Collect（収集）

外部の情報源から記事・ニュースを自動的に取得する。

**現在の実装:**
- RSS フィードのパース（rss-parser ライブラリ）
- ブログページの HTML スクレイピング（cheerio ライブラリ）
- ソース管理（有効/無効切り替え、フェッチ間隔設定）
- 重複排除（URL ベースのユニーク制約）
- 障害カウンタ（連続失敗時の自動検出）

**設計上の判断:**
- 最小フェッチ間隔は 4 時間（Gemini API のレート制限を考慮）
- ブログスクレイピングは汎用セレクタ（`article`, `.post`, `.blog-post`）を使用
- 1 ブログあたり最大 10 記事を取得（コスト制御）

#### Phase 2: Classify（分類）

収集した記事を AI が自動分類する。

**分類の軸:**

| 軸 | 値域 | 目的 |
|---|---|---|
| カテゴリ | 7 カテゴリ（後述） | 記事のテーマによるグルーピング |
| インパクトレベル | High / Medium / Low | 経営判断への影響度 |
| 関連度スコア | 0-100 | AI エグゼクティブにとっての関連性 |
| サマリー | 自由文（2-3 文） | エグゼクティブ向け要約 |

**7 カテゴリの定義:**

| カテゴリ | 対象領域 | 経営的意味 |
|---|---|---|
| Foundation Models | 基盤モデルの新リリース・ベンチマーク | AI 戦略の根幹に影響する技術変化 |
| Orchestration & Agents | エージェント基盤・オーケストレーション | 自動化・自律化の技術基盤 |
| For Developers | 開発者向けツール・API・SDK | 開発生産性と技術選定への影響 |
| Image Generation | 画像生成モデル・サービス | クリエイティブ・マーケティング領域 |
| Video Production | 動画生成・編集技術 | コンテンツ制作パイプラインへの影響 |
| Audio Technology | 音声合成・認識技術 | 音声インターフェース・アクセシビリティ |
| Vertical AI Agents | 業界特化型 AI エージェント | 特定業界での導入事例・ROI |

**インパクトレベルの判定基準:**

| レベル | 基準 | 例 |
|---|---|---|
| High | 大規模発表、市場の重要な変化、新たな能力の登場 | GPT-5 のリリース、主要 API の価格改定 |
| Medium | 注目すべきアップデート、パートナーシップ、研究成果 | エージェントフレームワークのメジャーバージョン |
| Low | マイナーアップデート、意見記事、一般的なニュース | ベンチマーク比較記事、業界予測 |

**関連度スコアの判定基準:**

| スコア範囲 | 意味 |
|---|---|
| 80-100 | AI 戦略の判断に直接影響する |
| 50-79 | エグゼクティブにとって有用な文脈情報 |
| 0-49 | 一般的な技術ニュース、経営的関連性は低い |

#### Phase 3: Synthesize（統合）

分類された記事群を横断的に分析し、統合的なインサイトを導出する。

**統合のプロセス:**

1. **カテゴリ別トレンド分析**: 各カテゴリの記事群からトレンドを抽出
   - サマリー（2-3 文のトレンド分析）
   - 主要な動向（箇条書き 3 点）

2. **エグゼクティブサマリー**: カテゴリ別の分析を横断的に統合
   - 200-300 語の概要
   - 業界全体のトレンドの方向性

3. **アクショナブルインサイト**: 具体的な行動提案を導出
   - 3 つの戦略的推奨事項
   - 経営層が取るべきアクション

**設計上の判断:**
- 統合分析には Gemini Pro（高品質モデル）を使用
- 分類には Gemini Flash（高速モデル）を使用
- 関連度スコア 60 以上の記事のみをレポートに含める
- レポートあたり最大 100 記事を分析対象とする

#### Phase 4: Deliver（配信）

生成されたレポートを適切な形式で配信する。

**現在の実装:**
- ダッシュボード UI でのレポート表示
- Markdown 形式でのレポート保存
- レポート一覧と詳細の API

**将来的な配信チャネル:**
- メール配信
- Slack 通知
- PDF エクスポート

### 1.3 Operations Cycle の周期

| 周期 | 処理内容 | 現在の実装状態 |
|---|---|---|
| 4 時間ごと | Collect: ソースからの記事取得と分類 | API 経由で手動実行 |
| 週次 | Synthesize + Deliver: 週次レポートの生成 | API 経由で手動実行 |
| リアルタイム | Dashboard UI でのライブフィード表示 | モックデータで実装済み |

## 2. 設計原則

### 2.1 Executive-First（経営者優先）

すべての出力は経営層の意思決定を支援することを最優先に設計する。

**具体的な意味:**
- 記事の分類基準は「経営者にとっての関連性」で定義する
- サマリーは技術詳細ではなくビジネスインパクトを記述する
- レポートは「知識の提供」ではなく「行動の提案」を目的とする
- UI はシンプルで情報密度が高く、操作を最小化する

**アンチパターン:**
- 技術的な詳細に深入りしたサマリー
- 行動に結びつかない情報の羅列
- 操作が複雑なダッシュボード

### 2.2 AI-Native Classification（AI ネイティブ分類）

情報の分類をルールベースではなく AI ネイティブに行う。

**具体的な意味:**
- カテゴリ分類はプロンプトで定義し、ルールエンジンを使用しない
- インパクトレベルと関連度は AI が文脈を理解して判定する
- 分類基準はプロンプトの更新で柔軟に変更できる

**トレードオフ:**
- AI 分類は確率的であり、100% の一貫性は保証されない
- API コストが発生する（1 記事あたり Gemini Flash 1 リクエスト）
- レイテンシはルールベースより高い

**このトレードオフを許容する理由:**
- ルールベースでは捕捉できない文脈的なニュアンスを拾える
- カテゴリの追加・変更が容易（プロンプト更新のみ）
- PoC フェーズではコスト・レイテンシより柔軟性を優先する

### 2.3 Separation of Speed（速度の分離）

処理の速度要件に応じて AI モデルを使い分ける。

| 処理 | モデル | 理由 |
|---|---|---|
| 記事分類（大量処理） | Gemini Flash | 高速・低コスト。1 記事あたりの分類に十分な品質 |
| トレンド統合（高品質） | Gemini Pro | 横断分析・エグゼクティブサマリーには深い推論が必要 |

この分離により、大量の記事を低コストで分類しつつ、統合分析の品質を維持する。

### 2.4 Progressive Automation（段階的自動化）

自動化を段階的に進め、各段階で品質を検証する。

| 段階 | 自動化レベル | 現在のステータス |
|---|---|---|
| 手動トリガー | API 呼び出しで手動実行 | 実装済み（PoC） |
| 定期実行 | cron ジョブによる自動実行 | 未実装 |
| 適応的実行 | ニュース量やトレンド変化に応じた動的スケジューリング | 構想段階 |
| 自律的運用 | ソースの自動発見・分類基準の自動調整 | 構想段階 |

PoC では手動トリガーに留め、パイプラインの正確性を確認した上で自動化を進める。

### 2.5 Structured Output（構造化出力）

AI の出力を構造化データとして扱う。

**具体的な実装:**
- 分類結果は JSON 形式で取得（プロンプトで JSON 出力を指定）
- レスポンスから正規表現で JSON を抽出
- 値の範囲バリデーション（スコア 0-100、既知カテゴリへのフォールバック）
- 構造化されたデータを PostgreSQL に永続化

**この設計の目的:**
- AI の出力を後続処理（統合・レポート生成）で機械的に利用可能にする
- 不正な出力をバリデーションで検出・補正する
- DB に格納することで集計・フィルタリング・トレンド分析を可能にする

## 3. Neural Organization マッピング

### 3.1 Operations Cycle と 5 レイヤーの対応

Operations Cycle の各フェーズは Neural Organization の 5 レイヤーアーキテクチャに対応する。

```
 Operations Cycle              Neural Organization Layer
 ────────────────              ─────────────────────────

 Collect (収集)          ←→    L0: Perception (検知)
 ├ RSS パース                   ├ 受動的接続: 外部データの取得
 ├ ブログスクレイピング          ├ 正規化: HTML → 構造化テキスト
 └ 重複排除                     └ シグナル検出: 新着記事の特定

 Classify (分類)         ←→    L1: Understanding (解釈)
 ├ カテゴリ分類                  ├ エンティティモデル: 記事の属性構造化
 ├ インパクト評価                ├ 因果モデル: 技術変化の影響評価
 └ 関連度スコアリング            └ 世界モデル: AI 業界の状態マップ

 Synthesize (統合)       ←→    L2: Reasoning (推論)
 ├ カテゴリ別トレンド分析        ├ ギャップ分析: 現状と理想の差分
 ├ エグゼクティブサマリー        ├ 機会の特定: 注目すべき変化
 └ アクショナブルインサイト      └ 判断: 推奨アクションの導出

 Deliver (配信)          ←→    L3: Execution (実行)
 ├ レポート生成                  ├ アーティファクト生成: Markdown レポート
 ├ ダッシュボード表示            ├ デリバリー: UI での表示
 └ (将来) メール・Slack 配信     └ 品質保証: レポートの一貫性検証

 Feedback (フィードバック) ←→   L4: Reflection (内省)
 ├ (将来) レポート評価            ├ Perception の改善: ソースの追加・除外
 ├ (将来) 分類修正               ├ Understanding の改善: 分類精度の向上
 └ (将来) インサイト検証          └ Reasoning の改善: 分析の深化
```

### 3.2 横断的要素のマッピング

| 横断的要素 | Operations における現在の実装 |
|---|---|
| Purpose（目的） | 「AI 業界の戦略的動向を経営層に届ける」というドメイン目的 |
| Governance（統治） | Admin API Key による管理操作の認証。PoC では最小限 |
| Memory（記憶） | PostgreSQL によるデータ永続化（Source, Article, Report） |
| Orchestration（統率） | API エンドポイントによる手動フロー制御 |

### 3.3 Phase 1（Awakening）への位置づけ

本プロジェクトは Neural Organization の変容プロセスにおいて、Phase 1: Awakening（覚醒）の実証である。

Phase 1 の核心は「システムが組織の現実を知覚し、理解を構築する」ことにある。Operations Cycle の Collect と Classify がまさにこれに該当する。

- **知覚の確立**: RSS/ブログへの接続により、AI 業界の情報フローを確立する
- **理解の構築**: AI 分類により、断片的なニュースが構造化された「世界モデル」に変換される
- **鏡としての機能**: ダッシュボードが「業界の現状」を可視化し、経営層に新たな視点を提供する

Phase 2（Symbiosis）への移行は、フィードバックループの実装（人間の修正 → 分類精度の改善）によって実現される。

## 4. BI 集約ワークフロー

### 4.1 データフロー全体像

```
外部データソース                                    内部処理
──────────────                                    ──────────

┌─────────────┐     ┌───────────┐     ┌──────────────┐
│  RSS Feed   │────▶│  Fetcher  │────▶│  Classifier  │
│  (rss-parser)│     │  Service  │     │  (Gemini     │
└─────────────┘     │           │     │   Flash)     │
                    │           │     └──────┬───────┘
┌─────────────┐     │           │            │
│  Blog Page  │────▶│           │            ▼
│  (cheerio)  │     └───────────┘     ┌──────────────┐
└─────────────┘                       │  PostgreSQL  │
                                      │  (Articles)  │
                                      └──────┬───────┘
                                             │
                                             ▼
                                      ┌──────────────┐
                                      │  Report      │
                                      │  Generator   │
                                      │  (Gemini Pro)│
                                      └──────┬───────┘
                                             │
                                             ▼
                                      ┌──────────────┐     ┌───────────┐
                                      │  PostgreSQL  │────▶│ Dashboard │
                                      │  (Reports)   │     │  UI       │
                                      └──────────────┘     └───────────┘
```

### 4.2 記事処理パイプライン

1. **ソース取得**: Fetcher が有効なソースを一括取得
2. **記事抽出**: ソース種別（RSS/Blog）に応じたパーサーで記事を抽出
3. **重複チェック**: sourceUrl の一意制約で重複を排除
4. **AI 分類**: Gemini Flash で各記事を分類（カテゴリ・インパクト・関連度・サマリー）
5. **永続化**: 分類結果と共に Article テーブルに保存
6. **ソース更新**: lastFetchedAt と failureCount を更新

### 4.3 レポート生成パイプライン

1. **対象記事の選定**: 指定週の記事で関連度 60 以上のものを取得（最大 100 件）
2. **カテゴリ別グルーピング**: 7 カテゴリ別に記事を分類
3. **カテゴリサマリー生成**: 各カテゴリの上位 10 記事で Gemini Pro がトレンド分析
4. **エグゼクティブサマリー生成**: カテゴリサマリーを統合し、全体概要を生成
5. **Markdown レポート構築**: Executive Summary → Key Trends → Actionable Insights → Sources
6. **レポート永続化**: Report テーブルに保存、ReportArticle で記事との関連を記録

## 5. データモデル

### 5.1 エンティティ関係

```
Source (情報源)
  │ 1:N
  ▼
Article (記事)
  │ N:M (through ReportArticle)
  ▼
Report (レポート)
```

### 5.2 エンティティの役割

| エンティティ | 役割 | Neural Organization との対応 |
|---|---|---|
| Source | 情報源の管理。接続先と取得状態を記録 | L0: Perception の接続レジストリ |
| Article | 分類済み記事の蓄積。AI 分類結果を含む | L1: Understanding の世界モデルの一部 |
| Report | 統合分析結果の保存。週次ブリーフィング | L3: Execution のアーティファクト |
| ReportArticle | レポートと記事の関連。トレーサビリティ | Memory: 判断の根拠の記録 |

## 6. 分類設計

### 6.1 プロンプト設計の方針

AI 分類プロンプトは以下の方針で設計する。

**ペルソナの明示**: 「AI news analyst for executives」としてモデルに役割を与え、経営者視点での分析を引き出す

**出力形式の厳密な指定**: JSON 形式での出力を指定し、後続処理での構造的なパースを可能にする

**判定基準のインライン記述**: スコアリングガイドをプロンプト内に記述し、一貫した判定基準を維持する

**コンテンツの切り詰め**: 入力記事を 3,000 文字に制限し、トークンコストを制御する

### 6.2 分類の品質保証

| 手法 | 説明 | 実装状態 |
|---|---|---|
| カテゴリバリデーション | 未知カテゴリが返された場合のデフォルトフォールバック | 実装済み |
| スコア範囲クリッピング | 0-100 の範囲に強制制限 | 実装済み |
| インパクトレベル検証 | 不正な値の場合 Medium にフォールバック | 実装済み |
| JSON パースエラーハンドリング | JSON 抽出失敗時の例外処理 | 実装済み |
| 人間によるレビュー | 分類結果の妥当性検証 | 未実装 |
| A/B テスト | プロンプト変更の効果測定 | 未実装 |

## 7. PoC 段階での制約と意図

### 7.1 意図的なスコープ制限

PoC では以下を意図的にスコープ外としている。

| スコープ外の項目 | 理由 |
|---|---|
| 自動スケジューリング | パイプラインの精度を手動で検証する段階 |
| 認証・認可の本格実装 | API キーで十分（シングルユーザー前提） |
| テストコード | パイプラインの設計が固まってからテストを書く |
| デプロイ | ローカル開発環境での検証が優先 |
| フィードバックループ | まず分類・レポートの基本品質を確立する |

### 7.2 PoC で検証する仮説

| 仮説 | 検証方法 | 成功基準 |
|---|---|---|
| RSS/ブログから十分な記事が収集できる | 主要 AI 系 RSS の接続テスト | 週 50 記事以上 |
| Gemini Flash の分類精度が実用に耐える | 50 記事のサンプル分類と人間レビュー | 80% 以上の分類精度 |
| 週次レポートが経営判断に有用である | 生成レポートの人間評価 | 「有用」以上の評価 |
| BI パイプライン全体が技術的に成立する | E2E での動作確認 | エラーなく完走 |

## 8. 用語集

| 用語 | 定義 |
|---|---|
| Operations Cycle | 収集 → 分類 → 統合 → 配信 → フィードバックの継続サイクル |
| Source（情報源） | RSS フィードまたはブログページの URL |
| Article（記事） | AI 分類済みの個別ニュース・記事 |
| Report（レポート） | 週次の統合分析レポート |
| Impact Level | 経営判断への影響度（High / Medium / Low） |
| Relevance Score | AI エグゼクティブにとっての関連度（0-100） |
| Executive Summary | カテゴリ横断の全体概要 |
| Actionable Insights | 具体的な行動提案 |
| Deep Research Console | ダッシュボード上の対話型調査インターフェース |
| Ingest（インジェスト） | 外部ソースからの記事の取得・分類・保存プロセス |
