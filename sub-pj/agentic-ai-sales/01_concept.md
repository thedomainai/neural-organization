# Sales Presentation Automator - コンセプト設計

> 本ドキュメントは agentic-ai-sales のコンセプト詳細を定義する。プロジェクトの WHY / WHO / WHAT は [00_overview.md](00_overview.md)、技術アーキテクチャは [02_architecture.md](02_architecture.md) を参照。

## 1. Revenue Cycle（レベニューサイクル）

### 1.1 定義

**レベニューサイクル**とは、リード獲得から受注、顧客成功、アップセル/クロスセルに至る売上創出の全工程を循環的に最適化するプロセスである。単なる営業プロセスではなく、プロダクトフィードバックまでを含む閉ループとして捉える。

```
┌───────────────────────────────────────────────────────────────────┐
│                      レベニューサイクル                              │
│                                                                   │
│  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐                  │
│  │ リード  │─→│ 商談化 │─→│  提案  │─→│  受注  │                  │
│  │ 獲得   │  │        │  │        │  │        │                  │
│  └────────┘  └────────┘  └────────┘  └────────┘                  │
│       ▲                                    │                      │
│       │                                    ▼                      │
│  ┌────────────┐  ┌────────────┐  ┌────────────────┐              │
│  │ プロダクト  │←─│ アップセル  │←─│      CS        │              │
│  │ フィードバック│  │ クロスセル │  │（顧客成功）     │              │
│  └────────────┘  └────────────┘  └────────────────┘              │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘
```

### 1.2 各工程の最適化目標

| # | 工程 | 最適化目標 | KPI |
|---|------|-----------|-----|
| 1 | リード獲得 | LTV/CAC の最大化 | CAC、リード品質スコア、チャネル別 ROAS |
| 2 | 商談化 | 商談化率の最大化 | リード→商談転換率、商談化リードタイム |
| 3 | 提案 | 再現性の確保（誰でも・入社後すぐ） | 提案品質スコア、準備時間、新人の初受注までの日数 |
| 4 | 受注 | 受注率の最大化 | 受注率、平均受注単価、リードタイム |
| 5 | CS | 契約期間の長期化 | 解約率、NPS、顧客健全性スコア |
| 6 | アップセル/クロスセル | 適切なタイミング × 商材 × 価格 × 表現 | エクスパンション率、ARPU成長率 |
| 7 | プロダクトフィードバック | 適切なプロダクトへの還元 | 機能要望の反映率、フィードバック起因の受注数 |

### 1.3 v1 の位置づけ

v1（本プロジェクトの現在のスコープ）は**工程3: 提案**にフォーカスしている。SPIN フレームワークに基づくプレゼン資料の自動生成を通じて、提案の品質と再現性を確保する。

v2 ではレベニューサイクル全体を Agentic AI で最適化する設計へ拡張する。詳細は [docs/design-v2-revenue-cycle.md](docs/design-v2-revenue-cycle.md) を参照。

### 1.4 Neural Organization とのマッピング

本プロジェクトは Neural Organization の 5 層アーキテクチャに以下のようにマッピングされる。

| レイヤー | レベニューサイクルにおける役割 |
|---------|---------------------------|
| L0: Perception（知覚） | リードデータ、CRMパイプライン、プロダクト利用データ、顧客の声、市場データの取得 |
| L1: Understanding（理解） | リードスコアリング、顧客ジャーニーマッピング、解約リスク予測 |
| L2: Reasoning（推論） | 商談戦略立案、提案戦略（フレームワーク選択）、価格戦略、アップセル機会特定 |
| L3: Execution（実行） | プレゼン資料生成（v1のSPIN生成）、提案書/見積書生成、メール生成 |
| L4: Reflection（内省） | Win/Loss分析、提案効果測定、勝ちパターン/負けパターンの自動抽出 |

詳細なマッピングは [02_architecture.md](02_architecture.md) を参照。

## 2. インプット定義

### 2.1 必須インプット

```
必須インプット
├── 顧客情報
│   ├── 基本情報（会社名、業界、規模、URL）
│   ├── 商談情報（フェーズ、想定金額、タイムライン）
│   ├── キーパーソン（役職、関心事、過去のやり取り）
│   └── 課題仮説（ユーザー入力 or 自動推定）
│
├── 商談コンテキスト
│   ├── プレゼン種類（シナリオ選択）
│   ├── 目的（何を達成したいか）
│   ├── 制約（時間、フォーマット）
│   └── 差別化ポイント（競合情報）
│
└── 過去のやり取り
    ├── 議事録
    ├── メール履歴
    └── 前回提案資料
```

### 2.2 補助インプット（自動取得可能）

```
自動取得データ
├── 企業情報
│   ├── 公開情報（IR、プレスリリース、ニュース）
│   ├── 財務情報（売上、利益、従業員数推移）
│   ├── 組織情報（部署、キーパーソン）
│   └── 技術スタック（利用ツール推定）
│
├── 業界情報
│   ├── 市場動向
│   ├── 競合動向
│   └── 規制動向
│
└── 自社情報
    ├── 製品情報
    ├── 価格情報
    ├── 導入事例
    └── 差別化ポイント
```

### 2.3 データソース連携

| カテゴリ | ソース | 取得データ | 優先度 |
|---------|--------|-----------|-------|
| CRM | Salesforce, HubSpot | 商談情報、顧客情報、活動履歴 | 必須 |
| 企業DB | SPEEDA, 帝国DB | 企業情報、財務情報 | 高 |
| Web | 企業サイト、ニュース | 最新情報、IR | 高 |
| 社内KB | Notion, Confluence | 製品情報、事例、FAQ | 必須 |
| コミュニケーション | Slack, Teams | 過去のやり取り | 中 |
| ドキュメント | Google Drive, Box | 過去の提案資料 | 高 |

### 2.4 データソース分類

データソースは取得方法別に 4 種類に分類される。

| 分類 | タイミング | 更新頻度 | 例 |
|------|-----------|---------|-----|
| PRESET（プリセット） | 導入時に1回設定 | 年1-2回 | ブランド設定、テンプレート、制約条件、ストーリーフレームワーク |
| MASTER（マスタ） | 継続的更新 | 週次/月次 | 製品情報、事例、競合情報、勝ち/負けパターン |
| AUTO（自動取得） | 資料生成時 | リアルタイム | CRM連携、企業公開情報、業界情報 |
| USER（ユーザー入力） | 都度 | 商談ごと | 会社名、業界、プレゼン種類、ゴール、課題仮説 |

## 3. 生成ロジック

### 3.1 教師データ学習

```
教師データ
├── 勝ちパターン
│   ├── 受注案件の提案資料
│   ├── 成功要因のタグ付け
│   │   ├── 刺さったメッセージ
│   │   ├── 効果的だった構成
│   │   ├── 決め手になったスライド
│   │   └── 顧客属性との相関
│   └── ベストプラクティス抽出
│
├── 負けパターン
│   ├── 失注案件の提案資料
│   ├── 失敗要因のタグ付け
│   └── アンチパターン抽出
│
├── 業界別ベンチマーク
│   ├── 業界特有の成功パターン
│   └── 業界別のNGワード/表現
│
└── 個社別学習
    ├── 過去の提案履歴
    ├── 顧客の反応履歴
    └── 担当者の好み
```

### 3.2 生成パイプライン

```
生成フロー
│
├── Phase 1: 情報収集・分析
│   ├── インプット取得
│   ├── データソース参照
│   ├── 顧客分析
│   └── 競合分析
│
├── Phase 2: 戦略立案
│   ├── ストーリーフレームワーク選択
│   ├── キーメッセージ決定
│   ├── 差別化ポイント整理
│   └── 想定Q&A準備
│
├── Phase 3: 構成設計
│   ├── スライド構成決定
│   ├── 各スライドの目的定義
│   ├── 必要データ/図表特定
│   └── 時間配分設計
│
├── Phase 4: コンテンツ生成
│   ├── テキスト生成
│   ├── データ可視化
│   ├── 図解生成
│   └── 画像選定/生成
│
├── Phase 5: デザイン適用
│   ├── テンプレート適用
│   ├── ブランドガイドライン適用
│   ├── レイアウト最適化
│   └── 一貫性チェック
│
└── Phase 6: 品質保証
    ├── 制約チェック
    ├── ファクトチェック
    ├── トーン&マナーチェック
    └── 最終レビュー
```

### 3.3 コンテンツ生成ロジック

| 要素 | 生成方法 | モデル/ツール |
|------|---------|-------------|
| キーメッセージ | 顧客課題×自社強み×競合差分 | LLM |
| 本文テキスト | テンプレート×コンテキスト | LLM |
| 数値/グラフ | データソース→可視化 | Plotly, Chart.js |
| 表 | 構造化データ→テーブル | Template |
| 図解 | 概念→Mermaid/SVG | LLM→レンダリング |
| 画像 | キーワード→検索/生成 | Stock/DALL-E |

## 4. アウトプット定義

### 4.1 出力形式

| 形式 | ユースケース | 優先度 |
|------|------------|-------|
| PowerPoint (.pptx) | 社内標準、編集必要時 | 必須 |
| Google Slides | コラボレーション | 高 |

| Keynote | Mac環境 | 中 |

### 4.2 付随出力

| 出力 | 内容 | 用途 |
|------|------|------|
| スピーカーノート | 話すべきポイント | プレゼン準備 |
| 想定Q&A | 質問と回答例 | 商談準備 |
| 競合比較表 | 差別化ポイント整理 | 質問対応 |
| 参考資料リンク | 詳細情報へのリンク | フォローアップ |
| 議事録テンプレ | 商談後の記録用 | 活動記録 |

### 4.3 プレゼンテーション構成要素

```
スライド構成要素
├── 構造的要素
│   ├── タイトルスライド
│   ├── アジェンダ
│   ├── セクション区切り
│   ├── サマリー/まとめ
│   └── Next Steps / CTA
│
├── コンテンツ要素
│   ├── テキスト（見出し、本文、箇条書き）
│   ├── 図表（グラフ、表、マトリクス、フレームワーク図）
│   ├── 図解（フロー図、アーキテクチャ図、組織図、タイムライン）
│   ├── ビジュアル（画像、アイコン、イラスト）

│
└── メタ要素
    ├── スピーカーノート
    ├── アニメーション指示
    └── 配布資料用補足
```

## 5. 制約

### 5.1 してはいけないこと（Negative Constraints）

```
禁止事項
├── コンテンツ制約
│   ├── 虚偽の情報（ないものをあると言わない）
│   ├── 未確認の数値（検証されていないROIなど）
│   ├── 競合の誹謗中傷
│   ├── 顧客の機密情報の不適切な使用
│   ├── 古いデータの使用
│   └── 出典のない統計データ
│
├── 表現制約
│   ├── 絶対表現（「絶対」「必ず」「100%」）
│   ├── 過度な約束（保証できない効果）
│   ├── 専門用語の過剰使用
│   └── ネガティブワード（問題、困難、できない）
│
├── ブランド制約
│   ├── ブランドガイドライン違反
│   ├── 非承認のビジュアル使用
│   ├── トーン&マナーの逸脱
│   └── 古いロゴ/デザインの使用
│
└── 法的制約
    ├── 景品表示法違反（誇大広告）
    ├── 著作権侵害
    ├── 個人情報の不適切な開示
    └── 業界固有規制違反
```

### 5.2 すべきこと（Positive Constraints）

```
必須事項
├── コンテンツ必須
│   ├── 顧客名/ロゴの正確な表記
│   ├── 数値の出典明記
│   ├── 最新情報の使用
│   └── 顧客業界に適した事例
│
├── 構成必須
│   ├── アジェンダスライド
│   ├── Next Stepsスライド
│   ├── 会社概要（初回のみ）
│   └── 問い合わせ先
│
└── スタイル必須
    ├── 一貫したフォント使用
    ├── ブランドカラーの適用
    ├── 余白の確保
    └── 1スライド1メッセージ
```

## 6. フィードバックループ

### 6.1 フィードバック収集

```
フィードバックポイント
├── 生成直後
│   ├── ユーザーの編集箇所
│   ├── 削除されたスライド
│   ├── 追加されたスライド
│   └── 修正されたテキスト
│
├── 商談後
│   ├── 商談結果（受注/失注/継続）
│   ├── 顧客の反応（ヒアリング）
│   ├── 効果的だったスライド
│   └── 改善が必要なスライド
│
└── 定期レビュー
    ├── 勝率分析
    ├── パターン分析
    └── ユーザーフィードバック
```

### 6.2 学習サイクル

```
学習プロセス
│
├── データ収集
│   └── フィードバック蓄積
│
├── 分析
│   ├── 成功/失敗パターン抽出
│   ├── 相関分析（顧客属性×成功率）
│   └── トレンド分析
│
├── モデル更新
│   ├── プロンプト改善
│   ├── テンプレート更新
│   ├── スコアリング調整
│   └── 制約ルール追加
│
└── 検証
    ├── A/Bテスト
    └── 効果測定
```
