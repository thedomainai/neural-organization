# Sales Presentation Automator - プロジェクト概要

> 本ドキュメントは agentic-ai-sales プロジェクトの WHY / WHO / WHAT を定義する。コンセプトの詳細は [01_concept.md](01_concept.md)、技術アーキテクチャは [02_architecture.md](02_architecture.md)、実装ガイドは [03_implementation.md](03_implementation.md) を参照。

## 0. WHY - 存在意義

### 0.1 解決する課題

| 課題 | 現状の痛み | 解決後の姿 |
|------|-----------|-----------|
| 資料作成時間 | 1提案あたり4-8時間 | 30分以内 |
| 品質のばらつき | 担当者スキル依存 | 勝ちパターン標準化 |
| ナレッジ属人化 | ベテランの暗黙知 | 組織の形式知化 |
| 顧客理解不足 | 汎用的な提案 | 個社最適化された提案 |

### 0.2 提供価値（バリュープロポジション）

1. **時間創出**: 資料作成から顧客対話への時間シフト。営業担当者が本来注力すべき顧客との関係構築・ヒアリングに時間を割けるようにする
2. **勝率向上**: 成功パターンの組織展開。トップ営業のノウハウをシステムに埋め込み、組織全体の提案品質を底上げする
3. **オンボーディング加速**: 新人の即戦力化。入社直後でも SPIN フレームワークに基づく質の高い提案資料を生成でき、スピーカーノートで「何を話すべきか」まで支援する
4. **一貫したブランド体験**: 顧客接点の品質統一。ブランドガイドライン準拠、禁止表現チェック、ファクトチェックを自動化し、対外的なコミュニケーション品質を担保する

### 0.3 現状分析

| 評価軸 | 良い点 | 不足点 |
|--------|-------|--------|
| 設計 | 営業シナリオの基本分類がある | WHY/WHO の深堀りが不十分だった |
| アーキテクチャ | AIエージェントアーキテクチャを意識している | 5層マッピングが未整理だった |
| 実装 | SPIN フレームワークの基本実装あり | 複数フレームワーク未対応 |
| 品質管理 | デザインシステム・制約を考慮 | 自動品質チェックが未実装 |

### 0.3 Neural Organization における位置づけ

agentic-ai-sales は Neural Organization のサブプロジェクトとして、**レベニューサイクル（Revenue Cycle）** の最適化を担う。レベニューサイクルとは、リード獲得から受注、顧客成功、アップセル/クロスセルに至る売上創出の全工程を指す。

v1 はプレゼン資料の自動生成（提案工程）にフォーカスしているが、v2 ではレベニューサイクル全体を Agentic AI で最適化する設計へ拡張する予定である。詳細は [docs/design-v2-revenue-cycle.md](docs/design-v2-revenue-cycle.md) を参照。

```
レベニューサイクル概要:

  リード獲得 → 商談化 → 提案 → 受注
       ▲                         │
       │                         ▼
  プロダクト ← アップセル ← 顧客成功(CS)
  フィードバック   クロスセル

  v1 のスコープ: 「提案」工程（SPIN プレゼン資料生成）
  v2 のスコープ: 全工程の最適化（10 エージェント体制）
```

## 1. WHO - ユーザー定義

| ペルソナ | 特徴 | ゴール | ペイン |
|---------|------|-------|-------|

| カスタマーサクセス | 既存顧客担当 | 継続・拡大提案 | データ整理が大変 |

|---------|------|

| プリセールス/SE | 技術詳細の補完 |

### 1.3 ステークホルダー（プレゼン相手）

| 層 | 関心事 | 必要なコンテンツ |

## 2. WHAT - プレゼンテーション定義

### 2.1 種類（シナリオ別）

```
1. 新規営業
   ├── 1.1 リード獲得フェーズ
   │   ├── コールドアプローチ（課題喚起型）
   │   ├── ウェビナーフォローアップ
   │   └── 紹介・リファラル
   │
   ├── 1.2 商談フェーズ
   │   ├── 1次商談（課題仮説 + ヒアリング + 概要説明）
   │   ├── 2次商談（詳細提案 + デモ）
   │   ├── 3次商談（見積 + 契約条件）
   │   └── 最終提案（役員プレゼン）
   │
   └── 1.3 コンペ・RFP対応
       ├── RFP回答書
       └── プレゼンテーション審査

2. 既存営業
   ├── 2.1 定期報告（週次/月次/四半期）
   ├── 2.2 拡大提案（アップセル/クロスセル/追加ライセンス）
   └── 2.3 更新・継続（契約更新/解約防止）

3. パートナー向け（開拓/共同提案）

4. 社内向け（案件相談/失注分析/成功事例共有）
```

### 2.2 業界別バリエーション

| 業界 | 特有の関心事 | 必須コンテンツ |
|------|------------|---------------|
| 金融 | セキュリティ、コンプライアンス | 認証取得状況、監査対応 |
| 製造 | 生産性、品質管理 | 現場での導入事例 |
| 小売 | 顧客体験、売上向上 | POS連携、オムニチャネル |
| IT・SaaS | 技術スタック、API | アーキテクチャ図、連携実績 |
| 医療 | 規制対応、患者安全 | 薬機法対応、導入実績 |
| 公共 | 調達要件、実績 | 自治体導入事例 |

### 2.3 MVP の出力（SPIN プレゼン資料）

Phase 1（MVP）では、新規1次商談向けに SPIN フレームワークで 7 枚構成のプレゼン資料を生成する。

| # | スライド | SPIN 要素 | 目的 |
|---|---------|-----------|------|
| 1 | タイトル | - | 会社名、日付、プレゼンタイトル |
| 2 | アジェンダ | - | 商談の流れを提示 |
| 3 | 現状認識 | Situation | 顧客の業界・規模・現状を確認 |
| 4 | 課題提起 | Problem | 顧客が直面する課題を言語化 |
| 5 | 影響・インパクト | Implication | 課題を放置した場合のビジネスインパクト |
| 6 | 解決価値・提案 | Need-payoff | 自社ソリューションによる解決価値 |
| 7 | Next Steps | - | 次回アクションの提示 |

### 2.4 ストーリーフレームワーク

| フレームワーク | 用途 | 構造 |
|---------------|------|------|
| SPIN | 1次商談 | Situation → Problem → Implication → Need-payoff |
| BANT | 予算確認 | Budget → Authority → Need → Timeline |
| MEDDIC | 大型案件 | Metrics → Economic Buyer → Decision Criteria → Decision Process → Identify Pain → Champion |
| 問題解決型 | 詳細提案 | 課題 → 原因 → 解決策 → 効果 → 実装計画 |
| Before/After | 事例紹介 | 導入前 → 導入後 → 成果 |
| Why/What/How | 概要説明 | なぜ必要か → 何ができるか → どう導入するか |

## 3. Project Status

### 3.1 現在のフェーズ

**Phase 1（MVP）**: 基本生成 — SPIN フレームワークによる新規1次商談向けプレゼン資料の自動生成

### 3.2 ロードマップ

**v1 ロードマップ**

| フェーズ | 期間 | 内容 | 状態 |
|---------|------|------|------|
| Phase 1 | 2週間 | 基本CLI + SPINフレームワーク + PPTX出力 | 実装中 |
| Phase 2 | 2週間 | 複数シナリオ（4種類）+ フレームワーク選択 + スピーカーノート | 計画中 |
| Phase 3 | 2週間 | 勝ちパターン登録 + フィードバック収集 + パターン適用 | 計画中 |
| Phase 4 | 2週間 | 企業情報自動取得 + 社内KB連携 + 複数出力形式 | 計画中 |

**v2 ロードマップ（レベニューサイクル全体最適化）**

| フェーズ | 期間 | 内容 | 状態 |
|---------|------|------|------|
| Phase 1 | 3-4週 | 提案エンジン完成（複数フレームワーク + 品質チェック） | 構想中 |
| Phase 2 | 3-4週 | 商談戦略 + Win/Loss 分析 + パターン学習 | 構想中 |
| Phase 3 | 3-4週 | リード + 商談化（スコアリング + CRM連携） | 構想中 |
| Phase 4 | 3-4週 | CS + エクスパンション（健全性監視 + アップセル） | 構想中 |
| Phase 5 | 2-3週 | プロダクトフィードバック + 共有記憶基盤接続 | 構想中 |

### 3.3 成功指標

**システムメトリクス**

| 指標 | 測定方法 | 目標値 |
|------|---------|-------|
| 生成時間 | 開始から完了まで | < 3分 |
| 編集率 | 編集スライド数/全体 | < 20% |
| リテンション | 継続利用率 | > 80% |
| 利用頻度 | 週あたり生成数 | > 3回/user |

**ビジネスメトリクス**

| 指標 | 測定方法 | 目標値 |
|------|---------|-------|
| 勝率向上 | 導入前後比較 | +10% |
| 資料作成時間 | 導入前後比較 | -70% |
| NPS | ユーザー調査 | > 50 |
| 商談準備時間 | 導入前後比較 | -50% |

**品質メトリクス**

| 指標 | 測定方法 | 目標値 |
|------|---------|-------|
| 制約違反率 | 自動チェック | < 1% |
| ファクトエラー率 | レビュー検出 | < 0.5% |
| ブランド準拠率 | 自動チェック | > 99% |
| ユーザー満足度 | 5段階評価 | > 4.0 |
