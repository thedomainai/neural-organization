# Solution Hypothesis

ソリューション仮説を定義する。

## ソリューション概要

**「AIドリブンの定例資料作成アシスタント」**

週次定例・MBRなどで提出される資料（Google Slides）の作成をサポート。
AIが曖昧さを検知し、ユーザーに質問・指示を出すことで、受動的に回答するだけで解像度の高い定例資料が完成する。

```
従来のフロー:
マネージャー → 曖昧な資料作成 → 定例で指摘される → 手戻り → 修正

neumannのフロー:
マネージャー → neumann が曖昧さを検知 → 質問/指示 → 回答 → 解像度の高い資料完成
                        ↑
              定例前に曖昧さが排除され、手戻りゼロ
```

## 曖昧さの検知パターン

Zero Ambiguityの原則に基づき、以下の4パターンを検知する。

| パターン | 説明 | 検知例 | 質問/指示例 |
|----------|------|--------|-------------|
| **深掘り不足** | 原子レベルまで分解されていない | 「売上未達」で止まっている | 「未達の要因をA-1, A-2...に分解してください」 |
| **網羅性不足** | MECEでない、抜け漏れがある | 3つの施策のうち2つしか報告がない | 「施策Cの進捗が記載されていません」 |
| **定量性不足** | 数値がない、定性的すぎる | 「概ね順調」「改善傾向」 | 「具体的な数値を記載してください（現在値/目標値）」 |
| **行動不明確** | Who/What/Whenが不明 | 「対応予定」「検討中」 | 「誰が、何を、いつまでに行うか明記してください」 |

## なぜこのソリューションか

### 既存の代替手段

| 代替手段 | 長所 | 短所 |
|----------|------|------|
| **CEO自身が指摘** | 精度が高い | 認知負荷が高い、定例の時間が浪費される |
| **Human COO** | 文脈理解、柔軟性 | 高コスト（1500万〜1億）、忖度が入る |
| **テンプレート強制** | 一定の構造化 | 形式的になる、曖昧さは排除できない |
| **レビュー文化** | 相互チェック | 人間関係で遠慮、時間がかかる |

### 我々のアプローチの優位性

1. **事前介入**: 定例の「前」に曖昧さを排除（手戻りゼロ）
2. **感情なき指摘**: 人間関係を気にせず論理的に指摘（Correction > Harmony）
3. **一貫した基準**: 誰が作成しても同じ品質基準を適用
4. **24/7稼働**: 人間のCOOと違い、常時監視・即座に介入
5. **学習効果**: 繰り返し指摘を受けることで、作成者自身のスキルが向上

## ソリューションの構成要素

### MVP機能

1. **Google Slides / Google Docs / Notion連携**
   - 資料をインポート/監視
   - 曖昧さを検知してハイライト

2. **曖昧さ検知エンジン**
   - 4パターン（深掘り/網羅性/定量性/行動）を検知
   - 検知箇所と理由を特定

3. **質問/指示生成**
   - 検知した曖昧さに対して具体的な質問・指示を生成
   - Webアプリ内チャットで対話

4. **タスク管理**
   - 未解決の曖昧さをタスクとして管理
   - 通知で進捗を促す

### 将来機能（MVP後）

- Google Sheets（KPIデータ）との自動連携
- Google Chat監視によるリアルタイム介入
- CEO向けダッシュボード（Observability）
- 自動レポート生成（Google Docs）

## 前提条件

このソリューションが成立するために必要な前提。

- [x] CEOがAIからの指摘を受け入れる（DEC-003で確認済）
- [ ] LLMがGoogle Slides / Google Docs / Notionのコンテンツから曖昧さを検知できる精度
- [ ] マネージャーがAIの質問/指示に従って回答する動機づけ
- [ ] Google Workspace APIとの連携が技術的に可能

## リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| 曖昧さ検知の精度が低い | 高 | 4パターンに絞り、ルールベース+LLMのハイブリッドで精度向上 |
| マネージャーが指摘を無視する | 中 | CEOへの可視化、タスク完了率のトラッキング |
| 過剰な指摘でユーザーが疲弊 | 中 | 重要度の優先順位付け、指摘の上限設定 |
| Google API制限・コスト | 低 | 監視頻度の最適化、キャッシュ活用 |
