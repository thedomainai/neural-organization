# HITL 抽出パターン設計

> 本ドキュメントは `information-requirements.md` で定義した情報抽出要件に対して、**人間から情報を引き出すための具体的な質問パターン・入力の足場・完全性検証プロトコル**を設計する。

## 1. 抽出パターン・カタログ

10 のパターンを定義する。各ゲートの抽出プロトコルは、これらの組み合わせで構成される。

### P1: 空白補完（Gap Filling）

```
用途: F（未観測の事実）の抽出
原理: システムが知っていることを明示し、「見えていない空白」を人間に気づかせる

構造:
  システム: 「以下が把握している状況です」
            [把握済み事項のリスト — カテゴリ別]
            「この中に含まれていない重要な事項はありますか？」
  足場:     カテゴリ別チェックリストを提示し、各カテゴリに「把握なし」を明示

効果: 人間は「何か追加は？」には「特にないです」と答えるが、
      具体的なカテゴリ一覧を見ると「あ、これは言っていなかった」と気づく
```

### P2: 仮説検証（Hypothesis Validation）

```
用途: C（因果の機序）、O（判断の補正）の抽出
原理: システムの仮説を明示的に提示し、人間の同意/修正/否定を求める

構造:
  システム: 「以下のように分析しました」
            [仮説: A が原因で B が発生した（確信度 72%）]
            「この因果関係は現場の実感と合いますか？」
            [合っている] [部分的に合っている（→どこが違うか）] [合っていない（→本当の原因は？）]

効果: 「合っていますか？」だけだと Yes で終わるが、
      確信度と3択を提示することで、人間が修正しやすくなる
```

### P3: 反実仮想（Counterfactual）

```
用途: C（因果の機序）の深掘り
原理: 「もし X がなかったら？」と問うことで、真の因果を切り分ける

構造:
  システム: 「仮に [原因候補] が発生していなかったとしても、
            同じ結果になっていたと思いますか？」
            [同じ結果になっていた → この要因は主因ではない]
            [異なる結果になっていた → この要因は主因の可能性が高い]
            [分からない]

効果: 人間が漠然と持っている因果の直感を、具体的な判断として表出させる
```

### P4: 二項対立選択（Binary Spectrum）

```
用途: V（価値の優先順位）の抽出
原理: トレードオフを二極のスペクトラムとして提示し、立場を表明させる

構造:
  システム: 「以下の軸について、御社はどちらに寄せたいですか？」
            [極 A] ←──●──────→ [極 B]
            現在の設計は ● の位置です

効果: 「どうしたいですか？」では答えられないが、
      二極を示すと「もう少し A 寄りにして」と表現できる
      HR-005 の「3 つの究極の問い」がこのパターンの最良の実装例
```

### P5: 時間アンカー（Temporal Anchoring）

```
用途: F（未観測の事実）、T（変化の予兆）の抽出
原理: 具体的な時間範囲を指定することで、記憶の検索範囲を狭め、想起を促す

構造:
  [過去の抽出]
  システム: 「先週月曜日から今日までの間で、
            データに反映されていない出来事はありましたか？」
            足場: カテゴリ別（顧客/競合/社内/市場/その他）

  [未来の抽出]
  システム: 「今後 [3ヶ月/6ヶ月/1年] で予定・予見される変化はありますか？」
            足場: 月別タイムライン + カテゴリ

効果: 「何か変わったことは？」には「特にない」と答えるが、
      「先週水曜日以降で」と限定すると具体的なエピソードを想起する
```

### P6: ステークホルダーマップ（Stakeholder Mapping）

```
用途: I（関係者の意図）の抽出
原理: 構造化されたテンプレートで、関係者の意図と力学を引き出す

構造:
  システム: 「この [意思決定/プロジェクト/商談] に関わるキーパーソンについて
            教えてください」
  足場:
    | 名前 | 役割 | 本音の関心事 | 姿勢 [推進/中立/抵抗] | 影響力 [高/中/低] |
    |------|------|------------|---------------------|----------------|
    |      |      |            |                     |                |

効果: 「誰が関係しますか？」には名前だけが返るが、
      構造化テンプレートにより姿勢・影響力まで引き出せる
```

### P7: 強制ランキング（Forced Ranking）

```
用途: V（価値の優先順位）の具体化
原理: 「全部大事」を許さず、優先順位の表明を強制する

構造:
  システム: 「リソースが [1つ/2つ] 分しかないとしたら、
            以下のうちどれを選びますか？」
            [選択肢 A]
            [選択肢 B]
            [選択肢 C]
            ...

効果: 「全部やりたい」は情報量ゼロ。
      強制的に選ばせることで、暗黙の優先順位が表出する
```

### P8: 事前検死（Pre-mortem）

```
用途: I（関係者の意図）、T（変化の予兆）の抽出
原理: 成功前提の思考を反転させ、失敗シナリオから情報を引き出す

構造:
  システム: 「6 ヶ月後にこの [計画/制度/施策] が失敗しているとします。
            最もありそうな原因は何ですか？」
            足場: 失敗原因のカテゴリ
              □ 実行リソースの不足
              □ 関係者の抵抗
              □ 前提条件の変化
              □ 想定外の外部要因
              □ その他: [自由記述]

効果: 「リスクは？」には「特にない」と答える人でも、
      「失敗するとしたら？」には具体的なシナリオを語れる
```

### P9: 確信度キャリブレーション（Confidence Calibration）

```
用途: O（判断の補正）の抽出
原理: 各結論への納得度を数値化し、「違和感」の所在を特定する

構造:
  システム: 「以下の各結論について、納得度を教えてください」
            結論 1: [内容]   納得度: [1──────10]
            結論 2: [内容]   納得度: [1──────10]
            結論 3: [内容]   納得度: [1──────10]
            「7 未満の項目について、何が引っかかりますか？」

効果: 「全体的にどうですか？」には「いいと思います」で終わるが、
      項目別に数値を求めると「ここだけ納得感が低い」が可視化される
```

### P10: 具体例要求（Concrete Example）

```
用途: F（未観測の事実）、C（因果の機序）の抽出
原理: 抽象的な質問を具体的な事例に落とし、暗黙知を引き出す

構造:
  システム: 「[この問題/この成功パターン] について、
            最近の具体例を 1 つ教えていただけますか？」
            「いつ、誰が関わり、何が起き、どうなりましたか？」

効果: 「どんな問題がありますか？」には抽象的な回答が返るが、
      「最近の 1 例を」と求めると、ディテールのある情報が引き出される
```

## 2. 盲点 × パターン マトリクス

| 盲点 | 第一選択パターン | 第二選択パターン |
|---|---|---|
| **F** 未観測の事実 | P1 空白補完 + P5 時間アンカー | P10 具体例要求 |
| **C** 因果の機序 | P2 仮説検証 + P3 反実仮想 | P10 具体例要求 |
| **I** 関係者の意図 | P6 ステークホルダーマップ | P8 事前検死 |
| **V** 価値の優先順位 | P4 二項対立選択 | P7 強制ランキング |
| **T** 変化の予兆 | P5 時間アンカー（未来） | P8 事前検死 |
| **O** 判断の補正 | P9 確信度キャリブレーション | P2 仮説検証 |

## 3. Trust Score に応じた抽出深度の適応

人間の入力負荷を管理するため、Trust Score に応じて抽出の深さを動的に調整する。

| Trust Level | 抽出深度 | 設計の根拠 |
|---|---|---|
| **Cold Start** (0.00-0.39) | 全抽出ステップ + 完全性検証 | システムが最も学習を必要とする段階。最大限の情報を収集する |
| **Learning** (0.40-0.69) | センターピン + 最重要カテゴリ 1 つ | 基礎的な学習が完了し、最も価値の高い情報に絞る |
| **Trusted** (0.70-0.89) | センターピンのみ（例外時は深掘り） | 定型的な判断は学習済み。新規性の高い情報のみ収集 |
| **Highly Trusted** (0.90+) | ゲート自動パス。Reflection 時に一括抽出 | 日常の負荷をゼロにし、四半期 Reflection で深い対話に集中 |

**注意**: 以降の設計は Cold Start（全抽出）を基本形として記述する。Learning 以上では、各ゲートの Step 1（センターピン対応）のみを実行する。

### 4.1 agentic-ai-hr（HR ライフサイクル）

#### HR-001: COMPANY_CONTEXT

EnrichedCompanyContext を提示した上で:

| Step | パターン | 盲点 | 抽出の問い | 足場 |
|---|---|---|---|---|
| 1 | P1 空白補完 | F | 「以下が把握している御社の状況です。[要約] この中に含まれていない、制度設計に影響する状況はありますか？」 | 5 カテゴリのチェックリスト: □組織変動（リストラ/再編/M&A） □人事（キーパーソン異動/退職リスク） □事業（新規事業/撤退/ピボット） □財務（資金調達/コスト削減要請） □外部（法規制/業界再編/労働市場） |
| 2 | P6 ステークホルダーマップ | I | 「この制度改革に対する主要な関係者の姿勢を教えてください」 | テンプレート: \| 名前 \| 役割 \| 姿勢[推進/中立/抵抗] \| 影響力[高/中/低] \| 最大の懸念 \| — 最低 3 名の記入を求める |
| 3 | P5 時間アンカー（未来） | T | 「今後 6-12 ヶ月で予定されている、以下のカテゴリの変化はありますか？」 | 月別タイムライン + カテゴリ: M&A/組織再編/大規模採用/事業計画/IPO 準備/法規制対応 |
| 4 | P4 二項対立 | V | 「制度設計の方向性について、以下 3 軸のどちらに寄せますか？」 | 3 つのスペクトラム: ① 変革速度: 急進的 ←→ 漸進的 ② 公平性: 結果の平等 ←→ 機会の平等 ③ 文化: 実力主義 ←→ 調和重視 |

**完全性検証**: Step 2 のマップが 3 名未満 → 「他に影響力のある方はいませんか？」。Step 3 が全カテゴリ空欄 → 「今後 1 年、御社の事業環境に変化がない見通しということでよろしいですか？」

TalentModel（6 コンピテンシー）を提示した上で:

| Step | パターン | 盲点 | 抽出の問い | 足場 |
|---|---|---|---|---|
| 1 | P10 具体例 | F | 「御社で最も高く評価されている人を 1-2 名思い浮かべてください。その方が持っていて、この 6 つのコンピテンシーに含まれていない特徴は何ですか？」 | 自由記述。記入がない場合: 「逆に、コンピテンシーは高いが実際には評価されていない人はいますか？何が足りないですか？」 |

## 5. 完全性検証プロトコル

各ゲートの抽出が完了した後、以下の 3 つの検証を実行する。

### 5.1 量的検証

```
判定基準:
  - 全 Step に何らかの入力があるか
  - ステークホルダーマップが含まれるゲートで、最低人数の記入があるか

不足時の対応:
  空欄の Step について:
  「[カテゴリ] について特に情報がないということでよろしいですか？
   もし思いつくことがあれば、後からでも追加できます」
```

### 5.2 具体性検証

```
判定基準:
  - 自由記述の入力が 1 文以上あるか（「特にない」のみは不合格）
  - 固有名詞（人名、企業名、数値）が 1 つ以上含まれるか

不足時の対応:
  P10（具体例要求）にフォールバック:
  「もう少し具体的にお聞きしたいのですが、
   最近の具体例を 1 つ教えていただけますか？」
```

### 5.3 新規性検証

```
判定基準:
  - 人間の入力に、システムが既に把握している情報と異なる内容が含まれるか
  - 全入力がシステムの既知情報と一致する場合、抽出が機能していない可能性

不足時の対応:
  P3（反実仮想）にフォールバック:
  「データ上の分析結果と現場の実感に、ギャップを感じる部分はありませんか？
   もし『ここは違う気がする』という箇所があれば教えてください」
```

```
差異分析レポート:
  売上 KPI: 計画比 -18%
  分解: 市場要因 -3% / 固有要因 -14% / 一時要因 -1%
  因果チェーン: 固有要因 → 新規商談数の減少 → リード品質の低下（確信度 68%）
────────────────────────────────────────────────────

[Step 1: 時間アンカー（F）]
システム: 先週月曜から今日までで、以下のカテゴリで
         データに反映されていない出来事はありましたか？

         □ 顧客の動き: ___
         □ 競合の動き: ___
         □ 社内の変化: ___
         □ 市場全体:   ___

Governor: 競合の動き → B社が先週金曜にエンタープライズ向けの
         大幅値下げを発表した。まだ影響は出ていないが、
         商談中の 3 社から問い合わせが来ている

────→ システム: 重要な情報です。因果チェーンに「競合の価格攻勢」を
         追加要因として組み込みます

[Step 2: 仮説検証（C）]
システム: リード品質の低下が主因と分析しました（確信度 68%）。
         これは現場の実感と合いますか？

         [合っている]
         [部分的に合っている → どこが違う？]
         [合っていない → 実際は？]

Governor: [部分的に合っている]
         → リード品質だけでなく、営業チームの稼働が採用面接の対応で
           先週は 30% 減っていた。これが大きい

────→ システム: 「営業チーム稼働率の低下」を因果チェーンに追加し、
         影響度を推定します。先ほどの競合値下げと合わせて、
         固有要因 -14% の分解を修正します

[Step 3: スキップ（Governorが Step 1 で競合情報を提供済み）]

[Step 4: 時間アンカー・未来（T）]
システム: 来月、この KPI に影響しそうな変化は？

         □ 大型案件の動向
         □ 季節要因
         □ キャンペーン/施策の開始・終了
         □ 人員の増減
         □ 特になし

Governor: 人員の増減 → 来週から営業 2 名が入社。
         採用面接の負荷はなくなる見込み

────→ システム: 「稼働率低下は一時要因」と分類を修正。
         因果モデルの更新: 「採用活動→稼働率低下→商談数減少」は
         一時的因果リンクとして記録
```

**この対話で抽出された情報**:
1. 競合の値下げ（F: 未観測の事実）→ 因果チェーンの修正
2. 採用面接による稼働率低下（C: 因果の機序）→ 主因の修正
3. 来週の採用完了（T: 変化の予兆）→ 一時要因への再分類

**もし抽出しなかった場合**: リード品質の低下が主因として固定され、リード獲得施策にリソースが投入される。実際の原因（稼働率低下）は対処されず、競合の値下げへの対策も遅れる。

## 7. ../agentic-ai-framework.md への統合

本ドキュメントの設計パターンは、`../agentic-ai-framework.md` のステップ 6「Governance 設計」に統合されるべきである。具体的には、フレームワークの HITL ゲート設計テンプレートに以下を追加する:

```
各 HITL ゲートに対して、以下を定義すること:

1. 承認/却下の判断基準（← 現設計で存在）
2. 情報抽出要件 — 6 つの盲点のうち、どれが該当するか
   （information-requirements.md 参照）
3. 抽出プロトコル — 10 パターンのうち、どれを使うか
   （本ドキュメント参照）
4. 足場（Scaffolding）— 人間の入力を助ける構造
5. 完全性検証 — 抽出が十分かを検証する基準
```
