# HITL 情報抽出要件定義

> 本ドキュメントは、各 HITL ゲートで「人間から何の情報を引き出すべきか」を定義する。現状の HITL 設計は「承認/却下」の判断を求める構造に偏っており、**人間が持つ暗黙知を能動的に抽出する設計**が欠落している。組織知能の学習品質は、HITL で人間から引き出す情報の質に直結する。

## 1. 前提 — なぜ「承認型」では不十分か

現状の全 27 ゲートの大半は以下の構造である。

```
システム: 「これを承認しますか？」
人間:     [承認] [条件付き承認] [差し戻し]
```

この構造は 3 つの誤った前提に基づいている。

| 誤った前提 | 現実 | 結果 |
|---|---|---|
| 人間は出力を丁寧に読む | 長い出力は読み飛ばされ「良さそう」で承認される | システムの誤りが学習データとして定着する |
| 人間は不足文脈を自発的に補完する | **聞かれなければ言わない。** 人間は自分が何を知っているかを知らない | システムの盲点が永続化する |
| 「修正」が最大の学習入力である | 修正は「何が間違っていたか」しか教えない。「何が見えていないか」は教えない | 学習が表層的になる |

**本ドキュメントの目的**: 各ゲートで「承認/却下」の判断に加え、**システムの盲点を埋める情報を能動的に抽出する要件**を定義する。

## 2. 抽出タクソノミー — システムの 6 つの盲点

AI システムには、データとモデルでは埋められない 6 種類の構造的盲点がある。HITL ゲートで人間から引き出すべき情報は、この 6 分類に帰着する。

| # | 盲点の分類 | 定義 | 人間が持つ理由 |
|---|---|---|---|
| **F** | **未観測の事実**（Unobserved Facts） | 現実に存在するがデータに反映されていない事象・状態 | 人間は会話・観察・体感で、システムの知覚範囲外の現実に触れている |
| **C** | **因果の機序**（Causal Mechanism） | 事象間の「なぜ」（相関ではなくメカニズム） | 人間はドメイン経験から因果の直感を持つ。データは相関しか示さない |
| **I** | **関係者の意図**（Stakeholder Intent） | 顧客・競合・社内の人間の計画・動機・政治力学 | 人間は他の人間の意図を読む能力を持つ。データは行動結果しか記録しない |
| **V** | **価値の優先順位**（Value Priority） | 複数の価値が衝突したときの重みづけ | トレードオフの判断はデータからは導出できない。意志の表明が必要 |
| **T** | **変化の予兆**（Upcoming Context） | 近い将来に発生する変化で、現在の判断に影響するもの | 人間は予定・計画・兆候を知っている。データは過去しか持たない |
| **O** | **判断の補正**（Judgment Override） | システムの結論に対する「違和感」とその根拠 | 人間の直感は言語化が難しいが、システムの過信/過小評価を検知できる |

### 2.1 盲点間の関係

```
未観測の事実（F）──→ 因果の機序（C）が見えない
       │                        │
       │                        ▼
       │              判断の補正（O）が必要になる
       │
       ▼
関係者の意図（I）──→ 変化の予兆（T）が見えない
                              │
                              ▼
                    価値の優先順位（V）が変わる
```

**F（事実）が抽出されないと C（因果）が歪む。C が歪むと O（補正）が多発する。** 上流の盲点を埋めるほど、下流のゲートの負荷が軽くなる。

## 3. 認知パイプライン位置と抽出優先度

ゲートが認知パイプラインのどの層に位置するかによって、抽出すべき盲点の優先順位が構造的に決まる。

| パイプライン層 | 主な盲点 | 理由 |
|---|---|---|
| **L0 Perception**（知覚） | F 未観測の事実 | 世界モデルの入力品質を決める。ここで欠落した事実は以降全層に波及する |
| **L1 Understanding**（理解） | C 因果の機序 | 「なぜ」の理解が正しくなければ、推論が構造的に間違う |
| **L2 Reasoning**（推論） | V 価値の優先順位 + I 関係者の意図 | 行動計画の方向性は、価値判断と実行環境（人間の意図）で決まる |
| **L3 Execution**（実行） | O 判断の補正 + I 関係者の意図 | 出力の最終調整は、受け手の文脈と直感的な妥当性で決まる |
| **L4 Reflection**（内省） | C 因果の機序 + T 変化の予兆 | 学習の方向性は、因果の正確さと環境変化の見通しで決まる |

### 3.1 カスケード障害 — 上流の抽出漏れが下流を壊す

例: agentic-ai-hr での連鎖

```
HR-001（Perception）で「M&A を検討中」を抽出できなかった
  → HR-003: 等級設計が将来の統合組織に合わない
  → HR-005: 報酬設計が統合後の市場水準と乖離する
  → HR-007: 導入計画が組織再編のタイムラインと衝突する
  → HR-008: Reflection が「なぜ制度が機能しないのか」の原因を特定できない
```

**上流のゲートほど、抽出漏れの損害が大きい。**

## 4. プロジェクト別 HITL ゲートの情報抽出要件

### 4.1 agentic-ai-hr（HR ライフサイクル）

#### HR-001: COMPANY_CONTEXT（ContextPerceptionAgent）

**センターピン**: 公式には語られない組織の力学 — 誰が実質的な意思決定者か、何が政治的にタブーか

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| F | 直近の組織変動（M&A 検討、リストラ計画、キーパーソンの離職リスク） | 短命な制度を設計する |
| I | 制度改革に対する各層の本音（推進派 vs. 抵抗派、労組の姿勢） | 導入時に想定外の抵抗に遭い頓挫する |
| T | 今後 6-12 ヶ月で予定されている事業構造の変化（IPO 準備、海外展開、事業売却） | 現在の組織前提で最適化した制度が数ヶ月で陳腐化する |
| V | 制度設計で絶対に守るべき文化的不可侵領域（例:「年功を完全否定する制度は受け入れられない」） | 文化と制度の衝突により定着しない |

#### HR-002: TALENT_MODEL（TalentModelingAgent）

**センターピン**: 公式な能力定義と、実際の昇進・高評価で重視される行動のギャップ

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| F | 「実際に高評価される人」が持っている特徴で、公式基準にないもの（例:「社内政治力」「顧客との飲み会」） | 形式上正しいが実効性のないコンピテンシーモデルが生成される |
| C | なぜ特定のコンピテンシーが自社で特に重要/不要か — 事業特性との因果（例:「規制産業なのでコンプライアンスが生命線」） | 汎用的だが差別化のないモデルになる |
| V | 相反するコンピテンシー間のトレードオフ（イノベーション vs. コンプライアンス、スピード vs. 品質） | 矛盾する期待を設定し、評価時に混乱する |

#### HR-003: GRADING_SYSTEM（GradingArchitectAgent）

**センターピン**: 組織の実態としての意思決定権限の境界線 — 公式の組織図との乖離

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| F | 現行の暗黙の等級感覚（「部長と課長の間にもう 1 段必要」「執行役員が多すぎて等級が形骸化」） | 現実と乖離した等級構造になり、格付け時に大量の例外処理が発生する |
| I | 昇格に対する経営層の本音（ポスト不足、人件費制約、「この人だけは特別扱いしたい」等） | 設計上は公平でも運用で歪む |
| T | 組織の成長/縮小見通し（3 年後に 2 倍の規模なら等級数を増やすべきか） | 現在最適だが将来破綻する構造になる |

#### HR-004: EVALUATION_SYSTEM（EvaluationArchitectAgent）

**センターピン**: 過去の評価運用で実際に発生した不公平感・紛争の具体事例

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| F | 現行評価で「暗黙に許容されている」バイアス（年功序列の残滓、部門間の甘辛差、男女格差） | 既存のバイアスを新制度に再生産する |
| C | 評価結果が実際の行動変容にどう結びついている/いないか（「良い評価でも昇給しないので意味がない」） | 評価制度が行動を変えるメカニズムを欠く |
| V | 差をつける評価 vs. 横並び文化のどちらを志向するか（明確な意志表明） | 制度と運用が矛盾し、評価者が板挟みになる |

#### HR-005: COMPENSATION_SYSTEM（CompensationArchitectAgent）

**センターピン**: 報酬に関する 3 つの究極の問いへの意志表明（※現設計で唯一適切に設計された抽出パターン）

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| V | 3 つの究極の問い（能力 vs. 職務、蓄積 vs. 洗替、階級 vs. 機能）**← 設計済み** | — |
| F | 現在の報酬に対する社内の不満の実態 — どの層が何に不満か（「エンジニアが市場水準との乖離に不満」等） | 最も緊急な問題を解決しない報酬体系になる |
| I | 予算制約の実態と、報酬に使える原資の上限感 | 理論上最適だが実行不可能な報酬設計になる |
| T | 採用市場の変動見通し（競合の報酬水準の変化、業界の人材需給） | 設計時点では競争力があるが 1 年後に陳腐化する |

#### HR-006: OUTPUT_CONFIRMATION（PolicyDocumentAgent）

**センターピン**: 従業員がこのドキュメントを読んだときに生じうる誤解・不安・反発の予測

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| I | 特に影響を受ける層（給与が下がる人、等級が下がる人）の反応予測 | 展開後に炎上し、制度の信頼性が毀損される |
| F | 過去の制度変更で起きたトラブルの具体事例（訴訟、大量離職、労組問題） | 同じ失敗を繰り返す |
| V | 透明性の程度 — 全情報開示 vs. 段階的開示（報酬テーブルの公開範囲等） | 情報の出し方を誤り、不要な混乱を招く |

#### HR-007: DEPLOYMENT_PLAN（DeploymentPlannerAgent）

**センターピン**: 導入を阻害しうる組織内の同時並行イベントと、各部門リーダーの変革受容度

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| F | 各部門リーダーの変革受容度（誰が協力的で誰が抵抗するか） | 抵抗勢力への対策なく展開し、部分的にしか定着しない |
| T | 同時進行中の他の大規模プロジェクト・繁忙期・組織再編の予定 | タイミングが衝突し、現場の許容量を超える |
| I | 経営層の「本当のデッドライン」— 外部制約（株主総会、期初、監査） | 実は動かせない期限があるのに柔軟なスケジュールを組んでしまう |

#### HR-008: POLICY_REVISION（ReflectionAgent）

**センターピン**: この四半期で経営の優先順位や事業環境に起きた変化のうち、制度に影響するもの

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| T | 経営戦略の方向転換や事業環境の変化で、制度の前提が崩れているもの | 過去のデータに最適化し続け、未来に適応できない |
| F | 制度運用の現場で起きている「運用の歪み」（形骸化、抜け道利用、評価者の負担） | KPI 上は問題ないが実態は機能不全 |
| C | データ上の制度 KPI 変動が、制度自体の問題か外部要因かの判断 | 外部要因を制度の問題と誤認し、不要な改定を行う |
| O | Reflection 提案のうち「理論上は正しいが今やるべきでない」ものの判別 | 組織の変革疲れを引き起こす |

### 4.2 neumann（経営管理サイクル）

#### MG-001: REPORT_QUALITY（ReportIntakeAgent）

**センターピン**: 品質指摘の却下理由 — なぜその「曖昧さ」が実は適切なのか、あるいは意図的なのか

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| C | 報告品質が低い構造的原因（時間不足？スキル不足？報告しにくい情報の政治的回避？） | 表層的な品質改善を繰り返すが、根本原因が解決しない |
| O | システムの検知が誤検知である場合の根拠（ドメイン特有の表現、意図的な抽象化） | 誤検知が学習データに混入し、検知精度が劣化する |
| F | 報告者が意図的に省略した情報の存在（「書けない事情がある」） | 重要なシグナルを見落とす |

#### MG-002: VARIANCE_ANALYSIS（VarianceAnalysisAgent）

**センターピン**: データに反映されていない直近 1-2 週間の顧客・競合・チーム内の変化イベント

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| F | 数値変動の裏にある「現場でしか知り得ない出来事」（キーパーソンの退職、大口顧客の不満表明、チーム内の対立） | 因果分析がデータ内の相関に閉じ、本当の原因を外す |
| C | システムの因果仮説に対する「実はこっちが本当の原因」という現場感覚 | 誤った因果モデルに基づいてアクションが策定される |
| I | 顧客・競合の動き — 公にはなっていないが感じている変化（「競合が値下げしたらしい」） | 外部要因を見落とし、内部要因に帰属してしまう |
| T | 来月以降に予見される変化（大型案件の動向、季節要因、法規制の変更） | 一時的な変動を構造的問題と誤認する、またはその逆 |

#### MG-003: ACTION_PLAN（ActionPlannerAgent）

**センターピン**: 各アクションの実行可能性に関する制約 — 担当者の実際の稼働状況と組織間の力関係

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| I | アサインされた担当者が実際にこのアクションを優先するか（本人の抱えている業務量、モチベーション） | 紙の上では完璧な計画が実行されない |
| F | 進行中の他プロジェクトとのリソース競合 | 実行リソースの二重計上が発生する |
| V | 5 つのアクション候補のうち「これだけは必ずやる」の選別（全部やるのが理想だが現実は無理） | 全てが中途半端に実行され、どれも効果を出さない |
| O | 期待効果の予測が楽観的/悲観的すぎるという感覚 | 過大な期待に基づいてリソースを配分し、効果測定時に裏切られる |

#### MG-004: REPORT_OUTPUT（ReportGeneratorAgent）

**センターピン**: レポートの受け手が最も気にしている論点と、避けるべきセンシティブな表現

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| I | 受け手（経営会議等）の今週の関心事と政治的文脈 | 重要な論点が埋もれ、レポートが「読まれたが活用されない」 |
| F | レポートに含まれる情報の機密性レベル（配布範囲との整合） | 機密情報が意図しない範囲に共有される |

#### MG-005: CAUSAL_MODEL_UPDATE（CycleReflectionAgent）

**センターピン**: 因果モデルの更新が事業環境の本質的変化を反映しているか、一時的ノイズか

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| C | 新しい因果リンクが「本当にそのメカニズムで繋がっている」かの確認 | 疑似相関を因果として学習し、以降の差異分析が体系的に間違う |
| O | モデルの確信度と自身の確信度のギャップ（「データ上は有意だが、これは偶然では？」） | 過学習した因果モデルが固定化する |
| T | この因果構造が今後も有効かの見通し（一時的な市場状況に依存していないか） | 環境変化後も古い因果モデルで判断し続ける |

### 4.3 agentic-ai-sales（レベニューサイクル）

#### RC-001: OUTREACH_PLAN（OutreachAgent）

**センターピン**: ターゲットリードに関する、CRM に入っていない人的情報 — 紹介者の存在、過去の非公式な接点、業界内の評判

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| F | リードの担当者の人柄、好みのコミュニケーションスタイル（「この人はメールを読まない」「電話を嫌がる」） | チャネル選択を誤り、第一印象を損なう |
| I | リード企業の購買検討ステータス（水面下で予算を確保している？競合と話している？） | タイミングを逃す、または時期尚早にアプローチして関係を損なう |
| T | リード企業の予算サイクル、組織変更、キーパーソンの異動予定 | 意思決定者が変わった後に前任者向けの提案をしてしまう |

#### RC-002: DEAL_STRATEGY（DealStrategyAgent）

**センターピン**: この商談で意思決定を左右するキーパーソンの「本当の関心事」 — 公式な RFP の要件と異なる場合がある

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| I | 顧客側チャンピオンの社内政治的立場（推進力があるか、敵がいるか、この案件での個人的利害） | チャンピオンの力を過大/過小評価し、戦略を誤る |
| F | 競合の提案状況（何社が参加、各社の強み、提示価格帯の推定） | 競合に対する差別化ポイントを外す |
| C | 同種の顧客で過去に成功/失敗した際の「決め手」と「落とし穴」（CRM の記録にない営業の暗黙知） | 過去の教訓を活かせず、同じ失敗を繰り返す |
| V | この商談での最低限獲得すべき条件と、譲歩可能な条件 | 交渉の幅を間違え、必要以上に譲歩するか、不要に強硬になる |

#### RC-003: PROPOSAL_REVIEW（ProposalAgent）

**センターピン**: 提案を受ける「意思決定者」が重視する判断基準 — 技術的合理性か、ROI か、リスク回避か、社内の政治的正当化か

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| I | 受け手が持っている先入観や懸念（前回の商談での反応、競合から聞いた情報） | 未対処の懸念を残したまま提案し、不採用になる |
| O | 資料の中で「これは刺さる/刺さらない」という営業経験に基づく直感 | 一般論的に正しい提案資料が、この顧客に刺さらない |
| F | 受け手がすでに知っている情報（重複説明は信頼を損なう） | 知っていることの繰り返しに時間を使い、核心に到達しない |

#### RC-004: PRICING_STRATEGY（PricingAgent）

**センターピン**: 顧客の実際の予算感と価格感度 — 公式な予算枠とは別の「本当の上限」

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| F | 競合の提示価格帯の推定（顧客からのほのめかし、業界情報） | 価格設定が市場実態と乖離する |
| I | 顧客の調達担当の交渉スタイル（値引きを前提とする文化か、提示価格が最終か） | 値引き余地の設計を誤る |
| V | この案件でのマージン vs. 市場シェアの重みづけ（戦略的に安く取るべきか） | 案件ごとの戦略的判断なく、一律の価格ロジックを適用する |
| T | 価格交渉の期限やタイムプレッシャーの存在（「今月中に決めないと予算が消える」） | 時間の制約を考慮しない交渉戦略になる |

#### RC-005: EXPANSION_PLAN（ExpansionAgent）

**センターピン**: 顧客の現在の満足度の「本音」と、拡大提案を受け入れる心理的準備状態

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| F | 顧客社内で起きている変化（部門拡大、新プロジェクト、担当者交代） | 機会を見逃す、またはタイミングを誤る |
| I | 顧客のキーパーソンのキャリア目標（サービス拡大が彼/彼女の功績になるか） | キーパーソンの動機を活用できない |
| O | ヘルススコアの数値と実感のギャップ（「数値は高いが、最近なんとなく温度感が下がっている」） | 数値に安心して顧客離反の兆候を見逃す |

#### RC-006: PRODUCT_FEEDBACK（ProductFeedbackAgent）

**センターピン**: フィードバック一覧の中で「これがないと本当に失注/解約する」ものと「あったら嬉しい」ものの境界線

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| C | 顧客が表面的に要望している機能の裏にある、本当に解決したい課題（「API を改善して」の裏に「業務自動化したい」がある） | 機能を実装しても顧客の課題が解決しない |
| I | プロダクトチームの実装余力と技術的制約（「これは 3 ヶ月かかる」「アーキテクチャ的に困難」） | 実行不可能な優先度を提案する |
| O | データ上の優先度と実際のビジネスインパクトのズレ（「件数は少ないが失う顧客の LTV が大きい」） | 件数に引きずられ、インパクトの大きい少数意見を見落とす |

#### RC-007: REFLECTION_REVIEW（RevenueCycleReflectionAgent）

**センターピン**: Win/Loss 分析の統計パターンに対する「直感的な納得感」と「違和感」の所在

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| C | 大型の受注/失注案件それぞれの「本当の決め手」 — CRM に入力されていない要因 | 勝ちパターン辞書が表層的な要因で構成され、再現性が低い |
| F | 営業チーム内の暗黙知（「最近こういう売り方が効く」という共有されていない感覚） | 個人の暗黙知が組織知に変換されない |
| T | 市場・競合環境の変化で、過去のパターンが今後も有効かの見通し | 過去の成功パターンに固執し、環境変化に適応できない |
| V | 次の四半期で営業組織としてフォーカスすべき方向性 | Reflection が戦略的方向性と無関係な最適化を続ける |

### 4.4 agentic-ai-analysis（データ分析サイクル）

#### DA-001: ANALYSIS_REQUEST（QuestionArchitectAgent）

**センターピン**: この分析結果がどの意思決定に・いつまでに・誰が使うのか — 分析の目的の明確な文脈

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| I | 分析を依頼した人が「実は期待している結論」の有無（確証バイアスの認識） | 期待に沿う結論に無意識に誘導される、または逆に期待と異なる結論が受け入れられない |
| V | 精度 vs. 速度のトレードオフ（「ざっくり今週中に」vs.「厳密に 1 ヶ月かけて」） | 過剰品質の分析に時間を費やし、意思決定のタイミングを逃す |
| T | この分析に関連する意思決定の期限（いつまでに何を決める必要があるか） | 期限を過ぎた分析が納品される |
| F | この問いについて既に試みた分析や得られている仮説 | ゼロから分析を始め、既知の知見を活かせない |

#### DA-002: DATA_QUALITY（DataQualityAgent）

**センターピン**: このデータの生成プロセスにおける既知の問題 — 「あのチームのデータ入力は信用できない」レベルの運用知識

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| F | データソースシステムの最近の変更（仕様変更、移行、既知のバグ） | 品質問題の根本原因を見誤り、対症療法的なルールが増殖する |
| C | 品質問題の根本原因 — 運用の問題か、設計の問題か（「手入力なので漏れが多い」vs.「API の仕様がおかしい」） | 根本原因を解決せず、同じ問題が永続する |

#### DA-003: MODEL_VALIDATION（StatisticalModelingAgent）

**センターピン**: 分析結果が現場の実感と合っているかの判断と、合っていない場合に「どこがどう違うか」

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| C | 「データ上はこうだが、実際のメカニズムはこうだ」というドメイン知識（例:「相関はあるが因果は逆」） | 統計的に有意だが因果的に誤ったモデルを採用する |
| F | モデルに含まれていない重要な変数（「あの要因を考慮していない」） | 交絡変数の見落としによる疑似因果を学習する |
| I | この結果を受け取る意思決定者がどう受け止めるかの予測（「これは受け入れられない」） | 正しいが政治的に実行不可能なインサイトを提示する |

#### DA-004: INSIGHT_REVIEW（InsightGeneratorAgent）

**センターピン**: アクション提案の組織的な実行可能性 — 「分析として正しいが実行できない」の判別

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| I | 提案されたアクションの実行に必要な人・予算・権限が確保できるか | 実行されないアクション提案が蓄積し、分析→アクション転換率が低下する |
| F | 同様の施策を過去に試みた経験の有無とその結果（「それ 2 年前にやって失敗した」） | 過去の失敗を繰り返す |
| V | 複数のアクション候補のうち、組織として優先すべき順序 | 分析的に最適だが組織の優先順位と合わないアクションが提案される |

#### DA-005: REPORT_OUTPUT（ReportGeneratorAgent）

**センターピン**: 受け手の現在の「信念体系」と、レポートがそれに与えるインパクトの予測

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| I | 受け手がこのレポートをどう使おうとしているか（意思決定の根拠？社内説得の材料？形式的な報告？） | 用途に合わない粒度・表現のレポートが生成される |
| F | 受け手のデータリテラシーと、誤解されやすい統計表現 | p 値や信頼区間の意味を誤解され、結論が歪められる |
| O | 結論の強さの調整（断定的にすべきか、留保付きにすべきか） | 過度に断定して信頼を失う、または過度に留保して意思決定に使えない |

#### DA-006: REFLECTION_REVIEW（AnalysisCycleReflectionAgent）

**センターピン**: 分析組織として次の四半期に注力すべき分析テーマの方向性

| 盲点 | 抽出すべき情報 | 未抽出時の失敗モード |
|---|---|---|
| C | 精度が低かった分析について、手法の問題か前提の問題かの判断 | 手法を変えても前提が間違っていれば改善しない |
| T | 事業環境の変化により、分析の優先領域が変わるかどうか | 過去の分析テーマに最適化し続け、新しい経営課題に対応できない |
| O | 学習内容の中で「過学習」の可能性 — 特定ケースに引きずられた過度な一般化 | パターンライブラリの品質が劣化する |

## 5. 横断分析

### 5.1 全 27 ゲートの盲点カバレッジ

各ゲートの抽出要件を盲点分類別に集計した結果。

| 盲点 | HR (8) | neumann (5) | sales (7) | analysis (6) | 合計 | カバー率 |
|---|---|---|---|---|---|---|
| **F** 未観測の事実 | 8/8 | 4/5 | 6/7 | 4/6 | **22/26** | 85% |
| **C** 因果の機序 | 3/8 | 4/5 | 2/7 | 4/6 | **13/26** | 50% |
| **I** 関係者の意図 | 5/8 | 3/5 | 7/7 | 4/6 | **19/26** | 73% |
| **V** 価値の優先順位 | 5/8 | 1/5 | 3/7 | 2/6 | **11/26** | 42% |
| **T** 変化の予兆 | 4/8 | 2/5 | 3/7 | 2/6 | **11/26** | 42% |
| **O** 判断の補正 | 1/8 | 3/5 | 3/7 | 3/6 | **10/26** | 38% |

**発見**:

1. **F（未観測の事実）は全プロジェクトで最重要の抽出対象。** ほぼ全ゲートで必要。これはシステムの知覚範囲の限界が最も普遍的な盲点であることを意味する
2. **V（価値の優先順位）と T（変化の予兆）は構造的に不足している。** これらは「聞かれなければ絶対に言わない」情報であり、最も能動的な抽出設計が必要
3. **O（判断の補正）は主に下流ゲートで必要。** Execution・Reflection 層のゲートに集中している。上流の F と C を十分に抽出できれば、O の必要性は減少する

### 5.2 プロジェクト別の抽出特性

| プロジェクト | 最重要盲点 | 理由 | 抽出の時間軸 |
|---|---|---|---|
| **agentic-ai-hr** | F + V | 制度設計は「現実の組織」と「あるべき姿」の両方を知る必要がある。V（価値の優先順位）は報酬の 3 つの究極の問いに象徴される | 深く・一回（設計時に集中的に抽出し、四半期で再検証） |
| **neumann** | F + C | 週次の差異分析は「何が起きたか」と「なぜ起きたか」の両方が必要 | 薄く・毎週（鮮度が命。1 週間前の情報は価値が急減する） |
| **agentic-ai-sales** | I + F | 商談は人間同士の交渉。関係者の意図と、CRM に入らない人的情報が勝敗を分ける | 深く・案件ごと（案件固有の情報が最も価値が高い） |
| **agentic-ai-analysis** | C + I | 統計モデルの妥当性はドメイン知識（因果の機序）でしか検証できない。実行可能性は関係者の意図で決まる | 中程度・分析ごと（分析の性質に応じて深さが変わる） |

### 5.3 現設計で唯一適切な抽出パターン

全 27 ゲートの中で、**HR-005 の「3 つの究極の問い」だけが V（価値の優先順位）の能動的抽出を正しく設計している。**

```
Q1: Competency（能力）vs. Role（職務）?
Q2: Accumulation（蓄積・安定性）vs. Wash-replacement（市場競争力）?
Q3: Status（階級制）vs. Function（機能制）?
```

このパターンが優れている理由:

1. **二項対立で提示する** — 「どうしたいですか？」ではなく「A と B のどちらに寄せますか？」と聞く
2. **抽象度が適切** — 個別の数字ではなく、設計哲学のレベルで意志を問う
3. **下流の全設計に波及する** — この 3 つの回答が報酬設計の全パラメータを方向づける

**他の 26 ゲートにも、同様の「構造化された意志の抽出パターン」が必要である。** それが次のステップの設計課題となる。

## 6. 設計上の含意

### 6.1 本ドキュメントが定義したもの

各ゲートで「何を引き出すべきか」（情報要件）を定義した。これにより、HITL ゲートの設計が「承認/却下」の判断装置から、**組織知能の盲点を埋める情報収集装置**へと転換する基盤ができた。

### 6.2 次に設計すべきもの

| 設計課題 | 内容 |
|---|---|
| **抽出パターンの設計** | 各抽出要件に対して、人間から情報を引き出すための具体的な質問パターン（二項対立、反例提示、仮説検証型等）を設計する |
| **入力の足場（Scaffolding）** | 人間が構造化された形で情報を提供できるテンプレート・選択肢・評価尺度を設計する |
| **完全性検証プロトコル** | 人間の入力が十分かを検証し、不足時に追加質問を行うプロトコルを設計する |
| **抽出情報の Memory 統合** | 抽出した情報を 3 構造の Memory（長期・作業・評価）にどう格納し、どう学習に活用するかを設計する |

### 6.3 根本的な設計原則の追加提案

design.md の 6 原則に加え、以下の原則を HITL 設計の指針として提案する。

> **Active Extraction（能動的抽出）**: システムは人間に「正しいですか？」と聞くのではなく、「あなたが知っていてシステムが知らないことを引き出す」ように設計される。人間は受動的な承認者ではなく、組織知能の感覚器官である。
