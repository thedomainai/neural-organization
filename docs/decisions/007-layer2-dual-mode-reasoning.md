# Layer 2 Reasoning の2モード設計：定型的推論と創発的推論の統合

## 問い

Layer 2 Reasoning において、「定型的推論（ステートマシン的な既知パターンの適用）」と「創発的推論（状況と目的から行動を都度導出）」をどう統合すべきか。

### 背景

- **現在の設計**: Layer 2 は「創発的推論」を設計目標としている（ワークフローに従わず、状況と目的から最適な行動を導出）
- **新たな洞察**: Openclaw の分析により、初期段階では「ステートマシン的な定型的推論」の方が信頼構築に適していることが判明
- **統合の必要性**: 実装上の堅牢性（定型的推論）と思想的先進性（創発的推論）を両立する設計が必要

## 論点

### 論点1: 境界基準 — 定型的推論と創発的推論の境界をどう定義すべきか

どのような条件が揃ったときに「定型的推論」で十分であり、どのような状況で「創発的推論」が必要になるのか。境界を明確にする判定基準は何であるべきか。

### 論点2: 遷移メカニズム — Trust Score との連動をどう設計すべきか

Neural Organization には Governance の Trust Score（0.0-1.0）があり、信頼の蓄積に応じてシステムの自律範囲が拡大する。この Trust Score と、2つの推論モードの使い分けをどう連動させるべきか。

### 論点3: パターンマッチング — 「過去の成功パターンに合致」の判定をどう実装すべきか

定型的推論の前提となる「現在の状況が過去の成功パターンに合致する」という判定は、どのようなロジック・データ構造で実現すべきか。

## 論点分解

### 論点1の分解

1. **判定方式の選択**: 条件列挙型 / スコアリング型 / 二段階判定型
2. **判定の責任主体**: システムが自動判定 / 人間が指定 / ハイブリッド
3. **不変原理との整合性**: 特に原理1（能力による間接参照）、原理2（計測）、原理4（コンテキスト予算）

### 論点2の分解

1. **Trust Score レベル別の推論モード配分**: Cold Start / Learning / Trusted / Highly Trusted での定型/創発の比率
2. **遷移の段階性**: 閾値での跳躍 / 連続的な調整
3. **領域別の非対称性**: CS 領域は Trusted、財務領域は Cold Start という状況への対応
4. **可逆性**: Trust Score 低下時のダウングレード設計

### 論点3の分解

1. **パターンの表現形式**: データ構造・スキーマ設計
2. **マッチングアルゴリズム**: ルールベース / セマンティック類似度 / ハイブリッド / 機械学習
3. **Memory との統合**: 長期記憶・作業記憶・評価記憶のどこにパターンを蓄積するか
4. **Reflection による学習**: パターンの発見・更新・検証・抽象化のフロー

## 仮説構築

### 論点1: 境界基準の設計案

#### 案A: 条件列挙型（ルールベース）

定型的推論で十分な条件を明示的に列挙し、いずれかを満たさない場合に創発的推論にフォールバック。

**条件例**:
1. 過去に同一カテゴリの判断を5回以上実行しており、成功率が80%以上
2. 状況の構成要素（エンティティ数、関連領域数）が3以下
3. 影響度が「小」または「中」
4. 不可逆性が「低」

**トレードオフ**:
- 利点: 判定ロジックが人間に説明しやすい。実装が単純。
- 欠点: 連続的な状況のグラデーションを二値で切るため、境界付近で不安定。

#### 案B: スコアリング型（連続値判定）

「定型度スコア」を算出し、閾値で分岐。

```yaml
routine_score =
  pattern_match_confidence * 0.35 +
  context_similarity * 0.25 +
  outcome_predictability * 0.20 +
  reversibility * 0.10 +
  trust_score_for_domain * 0.10

閾値:
  >= 0.75 → 定型的推論
  0.50-0.74 → ハイブリッド（定型+創発的検証）
  < 0.50 → 創発的推論
```

**トレードオフ**:
- 利点: 連続値で判定するため境界付近の不安定さが少ない。重みをデータ化すれば Evolution Engine で自己改善可能。
- 欠点: 初期の重み設定が困難（Cold Start 問題）。複合スコアの説明可能性がやや低い。

#### 案C: 二段階判定型（分類 + 確認）

軽量な分類器で「定型候補」「創発候補」を判定 → 定型候補にのみパターンマッチングを実行 → 信頼度で最終判定。

```
入力 → 分類器（定型候補 / 創発候補）
  → 定型候補 → パターンマッチング → 信頼度 >= 0.80 → 定型的推論
                                    → 信頼度 < 0.80 → 創発的推論
  → 創発候補 → 創発的推論を直接実行
```

**トレードオフ**:
- 利点: 軽量な分類で大半のケースを高速処理。創発的推論の計算コストを節約（原理4に適合）。
- 欠点: 分類器の精度が全体を左右する。分類器自体の学習データが初期に不足。

### 論点2: Trust Score との連動設計案

#### Trust Score レベル別の推論モード配分

| Trust Score レベル | 定型的推論 | 創発的推論 | ハイブリッド | 根拠 |
|-------------------|-----------|-----------|-------------|------|
| Cold Start (0.00-0.39) | 90% | 0% | 10% | 成功パターンの蓄積がないため、定型的推論 = 人間が定義したワークフロー。創発的推論は禁止（予測不能な行動が信頼を破壊） |
| Learning (0.40-0.69) | 60% | 15% | 25% | 評価記憶にデータが蓄積され始め、いくつかのパターンが検証済み。創発的推論を限定的に許可するが、必ず人間の承認を経由 |
| Trusted (0.70-0.89) | 40% | 35% | 25% | 長期記憶に十分なパターンが蓄積。創発的推論が自律的に実行される範囲が拡大 |
| Highly Trusted (0.90-1.00) | 20% | 60% | 20% | 創発的推論が主モード。定型的推論は定常業務にのみ使用 |

**重要な補足**: 上記の比率は静的な設定値ではなく、分類器による判定結果の「期待分布」。Trust Score が上がると分類器の閾値が緩和され、結果として創発的推論が選択される頻度が増加する。

```yaml
# 閾値の動的調整関数
pattern_match_confidence_for_routine = 0.60 + (trust_score * 0.33)

# trust_score=0.00 → 閾値0.60（低い → 定型に入りやすい）
# trust_score=0.50 → 閾値0.77
# trust_score=0.90 → 閾値0.90（高い → 定型は高確信時のみ）
```

#### 可逆性（ダウングレード）設計

Trust Score が下がった場合、推論モードは自動的に定型的推論側に戻る。

**ダウングレードトリガー**:
1. Trust Score が 0.10 以上低下した場合
2. 創発的推論の判断が3回連続で人間に覆された場合
3. Governor または Custodian による手動ダウングレード

**ヒステリシス**: 境界付近での頻繁な切り替え（チャタリング）を防止するため、復帰には「元の値 + 0.05」を要求。

### 論点3: パターンマッチングの実装設計案

#### パターン表現スキーマ

```yaml
reasoning_pattern:
  # 識別情報
  id: "rp_20250815_early_intervention_smb"
  version: 3
  created_at: "2025-08-15T10:30:00Z"

  # パターンの内容
  name: "SMB顧客の早期介入による解約防止"
  description: "健全性スコア70以下のSMB顧客に14日以内にCSMがアウトリーチ → 解約率40%低減"

  # 適用条件
  trigger:
    gap_type: "metric_decline"
    entity_type: "customer"
    conditions:
      - field: "customer.segment"
        operator: "in"
        value: ["SMB", "Mid-Market"]
      - field: "customer.health_score"
        operator: "<"
        value: 70

  # 文脈制約
  context_constraints:
    not_applicable_when:
      - "customer.contract_remaining_days < 30"
      - "customer.has_open_escalation == true"
    required_resources:
      - capability: "csm_outreach"
        min_availability: 1

  # 行動テンプレート
  action_template:
    primary_actions:
      - domain: "cs"
        action: "proactive_outreach"
        parameters:
          timing: "within_14_days"
          channel: "email_then_call"
        expected_contribution: "+3.0pt retention"

    escalation_to_emergent:
      - condition: "health_score < 50"
        reason: "深刻な状況では定型パターンでは不十分"

  # 信頼度メトリクス
  confidence:
    overall: 0.85
    sample_size: 47
    success_rate: 0.79
    last_n_success_rate: 0.83  # 直近10回の成功率

  # 鮮度管理
  freshness:
    last_applied: "2026-02-10T09:00:00Z"
    days_since_last_validation: 34
    staleness_threshold_days: 90
    status: "active"  # active / stale / deprecated
```

#### マッチングアルゴリズムの比較

| アルゴリズム | 精度 | 計算コスト | 説明可能性 | Cold Start 耐性 | 推奨度 |
|------------|------|-----------|------------|----------------|--------|
| ルールベース | 高（条件が明確な場合） | 極低 | 極高 | 高（人間が初期ルール定義可能） | Phase 1-2: 高 |
| セマンティック類似度 | 中-高 | 中 | 中 | 低（学習データ必要） | Phase 3-4: 高 |
| ハイブリッド（ルール + セマンティック） | 高 | 中 | 高 | 中 | 全Phase: 最高 |
| 機械学習（分類モデル） | 高（データあれば） | 高 | 低（ブラックボックス） | 極低 | 非推奨 |

**Phase別の使い分け**:
- Phase 1 (Awakening): ルールベースのみ（人間定義のIF-THENルール）
- Phase 2 (Symbiosis): ルールベース + セマンティック検索を補助的に導入
- Phase 3 (Metamorphosis): ハイブリッド（並列実行してマージ）
- Phase 4 (Emergence): セマンティック検索が主軸、ルールは事前フィルタ

#### Reflection による学習フロー

**フロー1: 新規パターンの発見**
- トリガー: 同一カテゴリの行動が3回以上成功した場合
- 処理: 共通構造を抽出してパターン候補を生成 → 評価記憶に保存
- 初期信頼度: 0.50（検証中）

**フロー2: 適用結果のフィードバック**
- トリガー: 定型的推論でパターンが適用された後、結果が判明
- 処理: `confidence`, `success_rate`, `last_n_success_rate` を更新
- 閾値判定: `confidence >= 0.70` かつ `sample_size >= 10` で長期記憶に昇格

**フロー3: パターンの検証・再検証**
- トリガー: `days_since_last_validation > staleness_threshold` または環境変化検出
- 処理: 前提条件が変わっていないか確認 → 必要に応じて更新 or deprecated

**フロー4: パターンの抽象化**
- トリガー: 同一構造の3つ以上のパターンが異なるセグメント/領域で成功
- 処理: より一般的な「原則」に昇華（例: SMB/Enterprise/Mid-Market の個別パターン → 「早期介入の原則」）

## 仮説の評価

### 論点1: 境界基準の比較

| 基準案 | 利点 | 欠点 | 不変原理との整合性 | 推奨度 |
|--------|------|------|-------------------|--------|
| A: 条件列挙型 | 説明しやすい、実装が単純 | 境界付近で不安定、条件の追加に人手が必要 | 原理5（設定のデータ化）: 条件を外部化すれば適合。原理2（計測）: 条件の有効性計測が別途必要 | 中 |
| B: スコアリング型 | 連続値で境界の不安定さが少ない、重みを自己改善可能 | 初期重み設定が困難、説明可能性がやや低い | 原理5: 強く適合。原理2: 各因子の計測が学習ループに直結 | 高 |
| C: 二段階判定型 | 計算コスト最適化、分類と解決の分離 | 分類器の精度が全体を左右、初期の学習データ不足 | 原理1（間接参照）: classify → resolve パターンに自然に適合。原理4（コンテキスト予算）: 最も適合 | 高 |

### 論点2: Trust Score 連動の妥当性検証

| 検証観点 | 評価 |
|---------|------|
| Phase 1-4 との整合性 | ✓ Phase 1 は定型的推論のみ（Reasoning ほぼ稼働せず）、Phase 4 は創発的推論が主モード。自然に対応 |
| 領域別の非対称遷移 | ✓ Trust Score が領域別に蓄積される既存設計と整合。CS 領域で Trusted でも財務領域では Cold Start が可能 |
| ダウングレードの安全性 | ✓ Trust Score 低下時に自動的に定型的推論に戻る設計により、システムの予測可能性が保全される |
| ヒステリシスの効果 | ✓ 境界付近でのチャタリング防止。ただし過度なヒステリシスは適応速度を下げるため 0.05 が適切 |

### 論点3: パターンマッチング実装の妥当性検証

| 検証観点 | 評価 |
|---------|------|
| Memory 3構造との整合性 | ✓ 検証済みパターン → 長期記憶、検証中 → 評価記憶、適用中 → 作業記憶。memory-design.md と完全に整合 |
| ハイブリッドアルゴリズムの複雑性 | △ ルール + セマンティックの両方を実装・維持するコストは高い。Phase 別に段階導入することで緩和 |
| Reflection フローの完全性 | ✓ 発見・更新・検証・抽象化の4フローで、パターンのライフサイクル全体をカバー |
| Cold Start 問題への対処 | ✓ Phase 1 ではルールベースのみ（人間定義）で稼働。セマンティック検索は Phase 2 以降に導入 |

## 結論

### 採用する設計

**論点1（境界基準）**: **案C（二段階判定型）をベースに、案Bのスコアリングを分類器の内部ロジックとして統合する**

理由:
- 不変原理1「能力による間接参照」の classify → resolve パターンに自然に適合
- 計算コスト最適化（原理4「コンテキストは有限資源」）
- 案Bの連続値判定の利点も内包可能

具体的な判定フロー:
1. **Step 1**: 軽量な分類器で「定型候補」「創発候補」を判定（創発トリガーに該当すれば即座に創発候補）
2. **Step 2**: 定型候補に対してのみパターンマッチングを実行（ルールベース + セマンティック）
3. **Step 3**: マッチング信頼度で最終判定
   - `>= 0.80` → 定型的推論
   - `0.60-0.79` → ハイブリッド
   - `< 0.60` → 創発的推論

**論点2（Trust Score 連動）**: **Trust Score に応じて分類器の閾値を動的調整する連続関数を採用**

```yaml
pattern_match_confidence_for_routine = 0.60 + (trust_score * 0.33)
```

Trust Score レベル別の期待モード配分:
- Cold Start (0.00-0.39): 定型90% / 創発0% / ハイブリッド10%
- Learning (0.40-0.69): 定型60% / 創発15% / ハイブリッド25%
- Trusted (0.70-0.89): 定型40% / 創発35% / ハイブリッド25%
- Highly Trusted (0.90-1.00): 定型20% / 創発60% / ハイブリッド20%

ダウングレード設計:
- Trust Score が 0.10 以上低下 → 閾値を即座に再計算
- 創発的推論が3回連続で覆される → 当該領域をハイブリッドモードに一時切り替え
- ヒステリシス: 復帰には「元の値 + 0.05」を要求

**論点3（パターンマッチング）**: **ハイブリッド（ルールベース + セマンティック）をPhase別に段階導入**

パターン表現スキーマ: 上記の `reasoning_pattern` YAML を採用

マッチングアルゴリズム:
- Phase 1: ルールベースのみ（人間定義のIF-THENルール）
- Phase 2: ルールベース + セマンティック検索（補助的）
- Phase 3-4: ハイブリッド（並列実行してマージ）

Reflection 学習フロー: フロー1-4（発見・更新・検証・抽象化）を採用

Memory 蓄積先:
- 検証済みパターン（confidence >= 0.70, sample_size >= 10）→ 長期記憶
- 検証中パターン → 評価記憶
- 適用中インスタンス → 作業記憶

### Phase 1-4 との対応

| Phase | 期間 | Trust Score | 推論モードの実装 | Layer 2 の主な活動 |
|-------|------|-------------|-----------------|-------------------|
| Phase 1: Awakening | 0-3ヶ月 | 0.00-0.39 | 人間定義のIF-THENルールのみ。Reasoning ほぼ稼働せず | 世界モデル構築が最優先。パターン蓄積なし |
| Phase 2: Symbiosis | 3-12ヶ月 | 0.40-0.69 | ルールベース + セマンティック検索導入。ハイブリッドで学習 | 「提案→承認→実行」パターン。成功パターン蓄積開始 |
| Phase 3: Metamorphosis | 12-36ヶ月 | 0.70-0.89 | パターン抽象化 + 領域別Trust Score連動 | 領域横断構成が本格稼働。創発的推論の比率増加 |
| Phase 4: Emergence | 36ヶ月以降 | 0.90-1.00 | 創発的推論が主モード。自己最適化 | 完全な「創発的な最適行動」。定型的推論は効率化のため残る |

### 不変原理との整合性の確認

| 不変原理 | 2モード設計との整合性 |
|---------|---------------------|
| 原理1: 能力による間接参照 | ✓ 二段階判定型は classify → resolve パターンそのもの。分類（定型 or 創発）と解決（パターン適用 or 推論）が分離 |
| 原理2: 計測なきシステムは収束しない | ✓ すべてのモード選択・パターンマッチング結果を Execution Trace に記録。Reflection の学習ループに直結 |
| 原理3: 優雅な劣化 | ✓ Trust Score 低下時のダウングレード設計により、システムは定型的推論に自動的に戻る（停止しない） |
| 原理4: コンテキストは有限資源 | ✓ 二段階判定型は軽量な分類で創発的推論の計算コストを節約。ハイブリッドモードの計算コスト正当化は残論点 |
| 原理5: 設定のデータ化 | ✓ パターンスキーマ、分類器の閾値、マッチングスコアの重みすべてがデータとして外部化。Evolution Engine で自己改善可能 |

### 4. 次の論点（論点2）への着手

## 残論点

### 残論点1: 定型的推論の行動テンプレートの粒度

`action_template` が詳細すぎると柔軟性を失い、粗すぎると定型的推論の意味がなくなる。現在のスキーマでは `primary_actions` と `optional_actions` の2段階だが、テンプレートのパラメータ化の深さ（どの変数を埋め込み可能にするか）について、実データに基づく検証が必要。

### 残論点2: ハイブリッドモードの計算コストの正当化

ハイブリッドモード（定型的推論+創発的検証）は、定型的推論のみと比較して計算コストがほぼ2倍。原理4「コンテキストは有限資源」との緊張がある。「ハイブリッドモードが創発的推論の精度向上にどれだけ寄与するか」を定量的に評価する仕組みが必要。

**提案**: 「ハイブリッドモードで創発的検証が定型的推論の結果を修正した割合」を追跡し、その割合が10%を下回った場合はハイブリッドモードを廃止して定型的推論に切り替える判断基準を設ける。

### 残論点3: パターンの鮮度と信頼度が矛盾する場合の優先順位

高い信頼度（sample_size=100, success_rate=0.90）だが長期間適用されていないパターン（staleness_threshold 超過）と、低い信頼度（sample_size=5, success_rate=0.80）だが直近で成功したパターンが同じ状況にマッチした場合の優先順位を定める必要がある。

現在のスコアリング式では freshness_score の重みが 0.15 と控えめだが、急速に変化する市場環境では鮮度の重みを引き上げるべきかもしれない。この重み自体を Evolution Engine の最適化対象にすることで解決可能。

### 残論点4: 「成功」の定義が領域によって異なる問題

パターンの `success_rate` を計算するには「何をもって成功とするか」の定義が必要。CS 領域では「顧客が解約しなかった」が成功、営業領域では「成約した」が成功。この成功定義をパターンスキーマに含めるか、領域別に外部定義するかの設計判断が残っている。

**推奨**: パターンスキーマ内に `success_criteria` フィールドを追加し、パターンごとに成功の定量基準を明示する。

### 残論点5: 人間が定型的推論を Override した場合の学習

Governor が「この状況では定型的推論ではなく創発的推論を使え」と指示した場合、その指示自体を学習データとして扱うべき。繰り返し Override される定型パターンは、trigger 条件の精緻化が必要か、パターン自体の deprecation が必要。

この「Override からの学習」を Reflection のフロー2（適用結果のフィードバック）に組み込む設計が必要。
