# [ARCHIVED — v2 設計で置換済み] v1.0 設計思想の v2.0 への統合

## 問い

v1.0（AX Company Architecture）が持つ設計思想のうち、v2.0（Neural Organization concept.md）の品質を高めるために統合すべきものは何か。また、統合すべきでないものは何か。

## 背景

Neural Organization は 2 つの世代の設計を持つ。

- **v1.0**（`neural-organization/docs/concept.md`）: 「記憶を持ち、戦略に基づいて動き、人と共に学ぶ組織」。5層スタック（Input → Memory → Processing → Evaluation → Output）、離散的エージェント（Strategy Agent, Execution Agent, AI Critic）、社内業務の効率化に焦点
- **v2.0**（`concept.md`）: AGI 時代の原理原則から再設計。5層認知パイプライン（Perception → Understanding → Reasoning → Execution → Reflection）+ 4横断的要素、流体的インテリジェンス、組織そのものの次の形態

v2.0 は issue-001 → 002 → 003 → 004 を経て v1.0 から根本的に再設計されたが、抽象化の過程で v1.0 が持っていた具体性・運用設計・ナラティブの力が薄まった部分がある。

## 論点

1. Memory の内部構造 — v2.0 の Memory は「各層の永続状態」と定義されているが、時間軸による構造化が欠落していないか
2. 品質保証の運用設計 — v1.0 の二重フィードバックループ（AI Critic → Human-in-the-Loop）に相当する運用原則が v2.0 に明示されているか
3. 学習メカニズムの具体性 — v2.0 の Reflection は「全層を対象とした自己参照的内省」と定義されているが、学習が具体的にどう起きるかの設計が十分か
4. Phase 1 のナラティブ — v2.0 のビジョンは射程が長いが、最初の顧客体験を一言で伝える力が v1.0 に劣っていないか
5. 人間-システム関係の双方向性 — v2.0 の設計原理は人間→システムの方向が強く、システム→人間の成長促進が設計されているか

## 論点分解

### 論点 1: Memory の時間軸構造

- v1.0 は記憶を 3 つの時間軸で明確に分離していた（Vector Database = 長期、Context Store = 短期、KPI Store = 評価基準）
- v2.0 は「各層が永続状態を持つ」と層別に分類しているが、時間軸の区別がない
- 「今この瞬間の会話文脈」と「10年かけて蓄積した組織の知見」はアクセスパターンも更新頻度も本質的に異なる
- 時間軸の構造化なしに Memory の実装設計は困難

### 論点 2: 品質保証の運用設計

- v1.0 は AI Critic（自動品質チェック）→ Human-in-the-Loop（人間の確認）という明確な二段階を持つ
- v2.0 では Execution の「品質保証」と Governance Gate が別々に定義されているが、両者の連携が暗黙的
- 「システムが先にフィルタリングし、人間は価値判断にのみ集中する」という原則が明示されていない
- この原則の欠如は、人間が些末な品質確認に時間を奪われる設計を許容してしまう

### 論点 3: 学習メカニズムの具体性

- v1.0 は具体例で学習を示していた（人間が「砕けた表現にして」→ 記憶層に「砕けた表現を好む」と追記）
- v2.0 の Reflection は「どの層が原因か」という自己参照的分析を定義しているが、学習の入力→変換→蓄積先の対応が不明確
- 特に「人間の修正行為そのものが最も価値の高い学習データである」という洞察が暗黙的
- Phase 2（Symbiosis）の「価値観の学習」がどの具体的メカニズムで駆動されるかが未設計

### 論点 4: Phase 1 のナラティブ

- v1.0 のタグライン「忘れない組織をつくる」は一瞬で問題と価値を伝える
- v2.0 の「組織そのものの次の形態」は知的に正しいが、最初の顧客が Phase 1 で体験する価値を一言で伝えられない
- issue-002 の評価でも「市場の理解しやすさ: 低い」「短期の売りやすさ: 低い」と自己認識済み
- ビジョンの射程を損なわずに、Phase 1 の具体的な問題解決力を伝える橋渡しナラティブが必要

### 論点 5: 人間-システム関係の双方向性

- v1.0 の「共に育つ」原則: 人間のフィードバック → AI が学習 → AI の能力向上 → 組織が強化 → より高度なフィードバック
- v2.0 の 5 設計原理は人間→システムの方向が中心（Intent over Instruction, Governance not Control 等）
- Agency Preservation は「人間の主体性を守る」防御的原則であり、「人間の主体性を高める」攻めの原則がない
- システムが人間の認知を拡張する方向の設計が欠落している

## 仮説構築

各論点について、v1.0 の要素を v2.0 に統合すべきか、統合すべきでないかを判断する。

### 統合すべき要素

| 要素 | v1.0 での形 | v2.0 への統合先 |
|---|---|---|
| Memory の時間軸構造 | Vector DB / Context Store / KPI Store | Memory（横断的要素）の内部設計として 3 構造を定義 |
| 二重フィルタリング原則 | AI Critic → Human-in-the-Loop | Governance の運用原則として明示 |
| 学習の具体的メカニズム | 人間の修正 → 記憶への追記 | Reflection の学習入力→変換→蓄積先の対応表 |
| 「忘れない組織」の問題定義力 | タグライン「忘れない組織をつくる」 | Phase 1（覚醒）の導入ナラティブ |
| 人間-AI 相互成長の循環 | 「共に育つ」原則 | 人間-インテリジェンス界面の第 6 設計原理 |

### 統合すべきでない要素

| 要素 | v1.0 での形 | 不採用の理由 |
|---|---|---|
| 社内データへのスコープ限定 | 入力は「社内の事実」に集中 | v2.0 の Perception は現実世界全体を対象とすべき。社内限定は人為的な制約 |
| Strategy / Execution Agent の分離 | 離散的エージェントとしての定義 | 流体的インテリジェンスの原則に反する。「エージェント」という境界は旧来の部門構造の写像 |
| 固定的なツール出力先リスト | Slack, CMS, GitHub, Mail 等の列挙 | v2.0 の Execution はケイパビリティを動的に拡張すべき。固定リストは実装の制約を概念設計に持ち込む |
| WHAT / HOW の二分法 | Strategy Agent = WHAT、Execution Agent = HOW | v2.0 の Reasoning は戦略と計画構成を一体として行う。分離は人工的な境界 |

## 仮説の評価

### 統合によって v2.0 が改善される点

| 評価軸 | 統合前の v2.0 | 統合後の v2.0 |
|---|---|---|
| Memory の実装可能性 | 「各層の永続状態」という抽象的定義のみ | 時間軸構造が加わり、実装設計への橋渡しが可能 |
| Governance の運用品質 | Gate の存在は定義されているが、人間の注意配分が未設計 | 二重フィルタリングにより人間の注意を最高価値の判断に集中させる |
| Reflection の具体性 | 「全層を対象とした自己参照的内省」という方向性のみ | 5 種類の学習入力と蓄積先の対応が明確になり、Phase 2 の駆動メカニズムが設計される |
| Phase 1 の伝達力 | 「鏡」としての体験が抽象的 | 「組織の記憶喪失」という切実な問題に具体的に応答する |
| 人間-システム関係の完全性 | 人間→システムの 5 原則 | 双方向の 6 原則。防御と攻めの両面が設計される |

### 統合によるリスク

| リスク | 評価 |
|---|---|
| v2.0 の抽象度が下がり、ビジョンの射程が短くなる | 低い。統合する要素はすべて v2.0 の枠組み内に位置づけられており、v1.0 の枠組みに引き戻すものではない |
| 概念の肥大化 | 中程度。concept.md の分量は増えるが、各追加は既存セクションの深化であり、新セクションの追加ではない |

## 結論

5 つの要素を v2.0 に統合し、4 つの要素を不採用とした。統合を起点にアーキテクチャ全体の具体化を実施し、最終的にドキュメントを概念編と設計編に分離した。

### 統合した変更（5 要素）

| # | 要素 | 概念編での位置（concept.md） | 設計編での位置（design.md） |
|---|---|---|---|
| 1 | **Memory の時間軸構造** | Section 2.6 Memory — 3 構造の定義と意義 | Section 2.3 — 技術スタック、選択的忘却、階層的抽象化 |
| 2 | **二重フィルタリング原則** | Section 2.6 Governance — 原則の定義とフロー図 | Section 1.4 — 品質保証の 4 段階チェックとして実装 |
| 3 | **学習の具体的メカニズム** | Section 2.5 Reflection — 5 種類の学習入力と蓄積先の対応表 | Section 1.5 — 各層を横断する学習の具体例 5 パターン |
| 4 | **Phase 1 の橋渡しナラティブ** | Section 3.1 — 「組織の記憶喪失」の問題定義 | （概念編のみ。設計編での具体化は未実施） |
| 5 | **Mutual Evolution 原則** | Section 4.1 — 第 6 設計原理として定義 | Section 3.1 — Interface の 5 モード内で具体的な対話パターンに反映 |

### 統合を起点としたアーキテクチャの具体化

v1.0 統合によって「抽象から具体へ」の方向性が確立され、アーキテクチャ全体の具体化を実施した。具体化の内容量がドキュメントの性質を変える規模に達したため、概念編（WHY + WHAT）と設計編（HOW の橋渡し）への分離を行った。

**各層の具体化（design.md Section 1）:**

| design.md セクション | 具体化の内容 | 設計上の意義 |
|---|---|---|
| 1.1 Perception | 接続ツール群の 7 カテゴリ（コミュニケーション、顧客関係、プロダクト開発、マーケティング、人事組織、ファイナンス、運用・インフラ） | 「現実世界への接続」が具体的なツール統合の設計問題であることを明示 |
| 1.2 Understanding | Customer エンティティモデル（基本属性・財務状態・健全性メトリクス・行動状態・サポート状態・関係性・シグナル・因果リンク） | 「世界モデル」がグラフ構造として実装可能であることを示した |
| 1.3 Reasoning | 領域横断構成の定量例（顧客維持率 88%→96%、4 領域の施策と期待効果） | 流体的インテリジェンスが具体的にどのような行動計画を生むかを数値で示した |
| 1.4 Execution | アーティファクト 4 カテゴリ、品質保証の 4 段階チェック（基準適合・論理整合性・安全性・最適化） | 二重フィルタリング原則の具体的な実装を設計した |
| 1.5 Reflection | 内省から学習への具体例 5 パターン（盲点発見、因果モデル修正、行動構成最適化、暗黙の価値観学習、因果チェーン精緻化） | 学習が単一層に閉じず複数層を横断することを実例で示した |

**横断的要素の具体化（design.md Section 2）:**

| design.md セクション | 具体化の内容 | 設計上の意義 |
|---|---|---|
| 2.1 Purpose | 4 要素（Why/Where/How/Boundary）の具体例。トレードオフ重み付け（顧客満足 70% vs 短期収益 30%）、Boundary の例 | Purpose が Reasoning にどう作用するかを構造化した |

### 不採用とした要素と理由

- 社内スコープ限定、離散的エージェント、WHAT/HOW 二分法 — いずれも v2.0 の設計原則（流体的インテリジェンス、全領域横断、動的ケイパビリティ）と矛盾するため不採用
- 固定的なツール出力先リスト — 不採用としたが、Perception の接続ツール群は v2.0 の枠組み（「能動的にどのソースに接続すべきかを判断する」）内で具体例として採用した。v1.0 が Output に固定ツールリストを置いていたのに対し、v2.0 では Perception の入力側に 7 カテゴリとして構造化した点が異なる

## 残論点

- **3 層構造の運用ルール**: concept.md → design.md → docs/philosophy/ の 3 層が確立されたが、「どの粒度の内容をどの層に書くか」の判断基準がまだ暗黙的。新しい設計内容を追加する際に、層の責務を維持するためのガイドラインが必要
- **ナラティブの二層構造**: 「忘れない組織」（Phase 1 の具体的価値）と「組織の次の形態」（長期ビジョン）をどう一貫したストーリーとして市場に伝えるか。Pitch Deck やホワイトペーパーでの表現設計が必要
- **定量パラメータの妥当性**: 信頼スコアの重み（成功率 0.30 等）、優先度管理の重み（緊急性 0.30 等）、トレードオフの重み（顧客満足 70% 等）は仮の値である。これらをどう検証・キャリブレーションするか
- **具体例の一般性**: design.md の具体例は SaaS 企業の CS/営業シナリオに偏っている。異なる業種・規模の組織に対してアーキテクチャの汎用性を示す追加の例が必要か
- **concept.md と design.md の相互参照**: 現在は冒頭の 1 行で相互リンクしているが、各セクション内での参照（「詳細は design.md Section X.X を参照」等）を追加すべきか。読者の導線設計
