# Neural Organization 5 層アーキテクチャ — フレームワークリファレンス

> 本ドキュメントは Neural Organization の 5 層アーキテクチャの構造・各層の責務・横断的要素・層間インタラクションを定義する、フレームワークのマスターリファレンスである。個別プロダクトの設計はこのフレームワークに準拠し、各層をドメイン固有のエージェント・機能にマッピングする。
>
> 思想・概念の詳細は [concept.md](concept.md) を、各層と横断的要素の詳細設計は [design.md](design.md) を参照。

## 1. アーキテクチャ全体像

Neural Organization は、現実世界からアウトプットが生まれるまでの変換パイプラインを **5 つのレイヤー** で構成する。各レイヤーは不可分な認知変換に対応し、明確な入力と出力を持つ。横断的要素として **Purpose（目的）**・**Governance（統治）**・**Memory（記憶）**・**Orchestration（統率）** が全層に作用する。

```
 Purpose ──────────────────────────────────── Governance
 (WHY: 意志の注入)     │                      (WHO: 統治)
                       │                        │
                       ▼                        │
┌──────────────────────────────────────────────────────────┐
│  Layer 0: Perception（知覚）                               │
│  現実世界 → シグナル                                       │
├──────────────────────────────────────────────────────────┤
│  Layer 1: Understanding（理解）                            │
│  シグナル → 世界モデル                                     │
├──────────────────────────────────────────────────────────┤
│  Layer 2: Reasoning（推論）       ◄── Purpose + Gov Gate  │
│  世界モデル × 目的 → 行動計画                               │
├──────────────────────────────────────────────────────────┤
│  Layer 3: Execution（実行）       ◄── Governance Gate     │
│  行動計画 → アウトプット                                    │
├──────────────────────────────────────────────────────────┤
│  Layer 4: Reflection（内省）                               │
│  アウトプット → 学習 → 全層更新                              │
└──────────────────────────────────────────────────────────┘
       ▲                                  ▲
  Memory（記憶）                   Orchestration（統率）
  WHAT: 全層の永続状態               HOW: 層間フロー制御
```

## 2. 各層の責務

### 2.1 Layer 0: Perception（知覚）

**変換**: 現実世界 → シグナル

現実世界との接続を確立し、流入するデータからシグナルを検出する。「何に接続すべきか」という能動的判断と、「何が意味あるシグナルか」というフィルタリングの両方を担う。

| 要素 | 内容 |
|---|---|
| 能動的接続 | どのデータソースに接続すべきかをシステム自身が判断し、接続を要求する |
| 受動的接続 | 外部システムからのプッシュ（Webhook, イベントストリーム）を受信する |
| 正規化 | 多様なフォーマットを共通表現に変換する |
| シグナル検出 | ノイズから意味ある変化・異常・トレンド・イベントを検出する |
| 注意配分 | 最も重要なシグナルにシステムの注意を集中させる |

**永続状態**: 接続レジストリ、ソースカタログ、注意モデル、シグナルパターン

### 2.2 Layer 1: Understanding（理解）

**変換**: シグナル → 世界モデル

検出されたシグナルを統合し、現実の因果モデルを構築・維持する。データベースではなく「生きた世界モデル」。

| 要素 | 内容 |
|---|---|
| エンティティモデル | 対象ドメインの実体とその関係性を構造化する |
| 因果モデル | 「X すれば Y が起こる、なぜなら Z だから」という因果構造を構築する |
| 予測モデル | 現在の状態から将来を確率的に予測する |
| 反実仮想モデル | 「もし X していたら Y だっただろう」という反事実的推論を行う |

**永続状態**: 世界モデル（知識グラフ + 因果モデル + 予測モデル）

### 2.3 Layer 2: Reasoning（推論）

**変換**: 世界モデル × 目的 → 行動計画

世界モデル（Layer 1）と人間が注入した目的（Purpose）を接続し、何をすべきかを導出する。ワークフローに従うのではなく、状態と目的から最適な行動を都度導出する「創発的推論」。

| 要素 | 内容 |
|---|---|
| ギャップ分析 | 現在の世界モデルと目的の間の乖離を特定する |
| 機会の特定 | 乖離を埋めるために取りうる行動の選択肢を生成する |
| トレードオフ評価 | 選択肢間のトレードオフを多軸で評価する |
| 判断 | 最適な行動を選択する。または Governance にエスカレーションする |
| 領域横断構成 | 複数領域を横断する統合的な行動計画を構成する |

**永続状態**: 判断履歴、戦略パターン、調整パターン

**Governance Gate**: 判断の重要度・不可逆性・不確実性が閾値を超える場合、人間の承認を要求する。

### 2.4 Layer 3: Execution（実行）

**変換**: 行動計画 → アウトプット

行動計画を実際に実行し、現実世界にアウトプットを生成する。

| 要素 | 内容 |
|---|---|
| アーティファクト生成 | コンテンツ、コード、分析レポート、提案書、コミュニケーション等を生成する |
| 外部システム操作 | メール送信、データベース更新、デプロイ、API 呼び出し等を実行する |
| 品質保証 | 出力がブランドガイドライン・品質基準・法的要件を満たしているかを検証する |
| デリバリー | アウトプットを適切な受信者に、適切なタイミングで、適切なチャネルで届ける |

**永続状態**: ケイパビリティレジストリ、実行ログ、デリバリーベストプラクティス

**Governance Gate**: 不可逆性の高いアクション（外部への送信、資金移動等）は人間の承認を要求する。

**品質保証の 4 段階チェック** (design.md §1.4):

| 段階 | チェック内容 | 不合格時の処理 |
|---|---|---|
| 1. 基準適合 | フォーマット規定、用語統一 | 自動修正して再生成 |
| 2. 論理整合性 | 事実との整合性、数値の正確性 | 問題箇所を特定して再生成 |
| 3. 安全性 | 法的リスク、機密情報の漏洩 | 即座にブロック、Governance にエスカレーション |
| 4. 最適化 | 受け手に合わせた表現・チャネル | 最適化案を適用して再生成 |

### 2.5 Layer 4: Reflection（内省）

**変換**: アウトプット + 観測された結果 → 学習シグナル → 全層の更新

アウトプットが現実世界に与えた影響を観測し、システム全体を改善する。単なるフィードバックではなく、全層を対象とした自己参照的な内省。

| 内省の対象 | 問い |
|---|---|
| → Perception | 接続していないデータソースが必要ではなかったか。重要なシグナルを見落としていなかったか |
| → Understanding | 世界モデルの因果構造は正しかったか |
| → Reasoning | 判断の前提は正しかったか。行動の構成は最適だったか |
| → Execution | 出力の品質は十分だったか |

**学習の入力と Memory への蓄積:**

| 学習入力 | 学習の変換 | Memory への蓄積先 |
|---|---|---|
| 人間が提案を修正した | 修正パターンから暗黙の価値観・品質基準を抽出する | 長期記憶（判断基準・組織の「らしさ」） |
| 人間が提案を却下した | 却下理由から「何を許容しないか」の境界を学習する | 長期記憶（Governance の暗黙ルール） |
| アウトプットが期待した結果を生んだ | 行動と結果の因果関係を世界モデルに統合する | 評価記憶（成功パターン） |
| アウトプットが期待した結果を生まなかった | 原因層を特定し、該当層の永続状態を修正する | 各層の永続状態 |
| 外部環境が予測と乖離した | 世界モデルの因果構造を修正する | Layer 1 の世界モデル |

**人間の修正行為そのものが最も価値の高い学習データである。** この蓄積が、システムが組織固有の価値観を学習し、自律性を高めていくプロセスを駆動する。

**永続状態**: 学習履歴、モデル評価、改善提案

## 3. 横断的要素

### 3.1 Purpose（目的）

人間が組織の意志を注入する界面。指示（Instruction）ではなく意志（Intent）を受け取る。

**意志の 4 要素:**

| 要素 | 内容 |
|---|---|
| Why（存在目的） | この組織はなぜ存在するのか |
| Where（方向） | どこへ向かうのか。定量的な Outcome |
| How（価値観） | 何を重視し、何を許容しないか。トレードオフの判断基準 |
| Boundary（制約） | 予算上限、法的制約、倫理的境界 |

Purpose は主に Layer 2（Reasoning）に作用し、世界モデルと目的の間の行動計画導出を方向づける。

### 3.2 Governance（統治）

人間がシステムの行動を統治する仕組み。「承認フロー」ではなく「組織知能の立憲的統治」。

**Governance Gate:**
- **Layer 2 Gate**: 判断の重要度・不可逆性・不確実性が閾値を超える場合、人間の承認を要求する
- **Layer 3 Gate**: 不可逆性の高いアクション（外部への送信等）は人間の承認を要求する

**二重フィルタリング原則:**

```
行動計画/アウトプット → システム自己評価（品質・整合性・基準適合）
  → パスしたもののみ → Governance Gate（人間の判断）
```

人間は「品質の確認」ではなく「価値判断・倫理判断・戦略判断」に注意を集中する。

**信頼スコアモデル:**

```
Trust Score = 成功率 × 0.30 + 結果品質 × 0.35 + 整合性 × 0.20 + リスク管理 × 0.15
```

| レベル | Trust Score | 自律範囲 |
|---|---|---|
| Cold Start | 0.00 - 0.39 | 全判断に承認が必要 |
| Learning | 0.40 - 0.69 | 定型的な判断は自律実行 |
| Trusted | 0.70 - 0.89 | 大半の判断は自律実行 |
| Highly Trusted | 0.90 - 1.00 | 戦略的判断を含む広い自律 |

信頼は**領域別**に蓄積される。ある領域で Highly Trusted でも、別の領域では Cold Start の可能性がある。

### 3.3 Memory（記憶）

Memory は独立したレイヤーではなく、全層にまたがるインフラストラクチャ。各レイヤーが自身の永続状態を持ち、それらの総体が「組織の記憶」を構成する。

**3 構造の記憶:**

| 構造 | 時間軸 | 内容 | 更新頻度 |
|---|---|---|---|
| 長期記憶（Institutional Memory） | 年〜永続 | 組織の成功・失敗パターン、暗黙の価値観、「組織らしさ」を規定する判断基準 | 低頻度・高価値 |
| 作業記憶（Working Memory） | 秒〜日 | 進行中のタスクの状態、現在の会話文脈、直近の判断の連鎖 | 高頻度・揮発的 |
| 評価記憶（Evaluative Memory） | 週〜月 | 目標と現状のギャップ、KPI の推移、行動と結果の対応関係 | 定期的・累積的 |

**技術スタックの推奨:**

| 記憶構造 | 技術スタック | 選択理由 |
|---|---|---|
| 長期記憶 | PostgreSQL + ベクトル DB | エンティティ関係の保持 + セマンティック検索 |
| 作業記憶 | Redis | 高速アクセス。揮発的だが重要な文脈は長期記憶に昇格 |
| 評価記憶 | PostgreSQL（時系列） | KPI 推移と行動-結果の対応を時系列で蓄積 |

**選択的忘却:**
- 作業記憶: タスク完了後、要約されて長期記憶に昇格するか、忘却される
- 評価記憶: 一定期間後、個別データは統計的パターンに集約される
- 長期記憶: 原則として忘却しない。否定された知見は信頼度を下げて保持する

### 3.4 Orchestration（統率）

層間の実行フローを統率する機構。各層は「何を変換するか」を定義するが、「いつ・どの順序で・どのリソースで実行するか」は Orchestration が決定する。

**フロー制御の 4 パターン:**

| パターン | 発動条件 | 説明 |
|---|---|---|
| 並列実行 | 独立した複数タスクが同時に発生 | 相互に依存しないタスクを同時に処理する |
| ループバック | 後段の層が前段の修正を要求 | 情報不足等を検出した場合に前段へ差し戻す |
| スキップ | 中間層が不要な場合 | 既知のパターンに合致する場合に中間層を省略する |
| 条件分岐 | 判断の結果に応じてフローが分岐 | Governance Gate の結果に応じて次の処理先を決定する |

**優先度管理:**

```
Priority Score = 緊急性 × 0.30 + 影響度 × 0.35 + 依存関係 × 0.20 + リソース効率 × 0.15
```

**トリガーパターン:**

| パターン | 説明 |
|---|---|
| サイクル駆動 | 定期的なサイクル（週次・月次・四半期）で処理を起動する |
| イベント駆動 | 外部イベント（ステージ変化、異常検知等）をトリガーとして起動する |
| リクエスト駆動 | 人間または他システムからのリクエストで起動する |
| アラート駆動 | 閾値超過や異常検知をトリガーとして即時起動する |

## 4. 層間インタラクション

### 4.1 基本フロー

標準的なデータフローは L0 → L1 → L2 → L3 → L4 の順方向だが、Reflection（L4）からの学習シグナルは全層にフィードバックされる。

```
L0 (Perception)
  │ シグナル
  ▼
L1 (Understanding)
  │ 世界モデル
  ▼
L2 (Reasoning)  ◄── Purpose  ◄── Governance Gate
  │ 行動計画
  ▼
L3 (Execution)  ◄── Governance Gate
  │ アウトプット
  ▼
L4 (Reflection)
  │ 学習シグナル
  ├──→ L0 更新（接続先・シグナルパターン）
  ├──→ L1 更新（世界モデル・因果モデル）
  ├──→ L2 更新（判断基準・戦略パターン）
  ├──→ L3 更新（品質基準・デリバリーパターン）
  └──→ Memory 更新（長期記憶・評価記憶）
```

### 4.2 HITL（Human-in-the-Loop）インタラクション

各プロダクトは、ドメイン固有の Governance Gate（HITL ゲート）を定義する。ゲートの共通設計原則:

- 人間の注意は有限のリソースであり、最もレバレッジの高い判断にのみ消費されるべきである
- Trust Score に応じてゲートの閾値を動的に調整する
- 不可逆性の高いアクションほど、HITL を必要とする閾値が低い（＝人間の承認を得やすい）

### 4.3 人間の参与形態

| 参与形態 | 役割 |
|---|---|
| Governor（統治者） | 組織の存在理由を定義する。Purpose と Values の源泉 |
| Sensemaker（意味付与者） | データが捉えられない質的現実をシステムに翻訳する |
| Creator（創造者） | まだ存在しないものを構想する |
| Connector（接続者） | 人間同士の信頼・共感・関係性を構築する |
| Custodian（守護者） | システムの行動が価値観・規範と整合しているかを監視する |

## 5. プロダクト設計への適用ガイド

### 5.1 マッピングの手順

各プロダクトは、以下の手順でドメイン固有の設計を作成する。

1. **Purpose の定義**: そのドメインにおける意志の 4 要素（Why / Where / How / Boundary）を具体化する
2. **各層のマッピング**: L0〜L4 の各層で、ドメイン固有のデータソース・分析・推論・成果物・学習を定義する
3. **エージェント設計**: 各層の責務を遂行するエージェント群を設計する
4. **Memory 設計**: 3 構造の記憶に、ドメイン固有のデータモデルをマッピングする
5. **Governance 設計**: ドメイン固有の HITL ゲートと Trust Score の測定方法を定義する
6. **Orchestration 設計**: ドメイン固有のトリガーパターンとワークフローを定義する

### 5.2 共有記憶基盤

複数プロダクトが連携する場合、共有記憶基盤を通じてデータを交換する。各プロダクトは以下を明示的に定義する。

- **書き込み**: 他プロダクトに提供するデータと、その消費者・活用例
- **読み出し**: 他プロダクトから取得するデータと、その提供元・活用例

## 6. 設計原理

このフレームワークを貫く 6 つの設計原理:

| 原理 | 内容 |
|---|---|
| Intent over Instruction | 人間は指示ではなく意志を伝える |
| Governance, not Control | 人間は制御ではなく統治する |
| Ambient Presence | システムは環境として常に存在する |
| Radical Transparency | あらゆる推論・判断の根拠が確認可能 |
| Agency Preservation | 人間の主体性を保全する |
| Mutual Evolution | 人間とシステムが相互に高め合う |
