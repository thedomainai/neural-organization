# Agentic AI 構築フレームワーク — Neural Organization 準拠

> 本ドキュメントは、Neural Organization の思想に準拠した Agentic AI を新規に構築するための**再利用可能なフレームワーク**を定義する。agentic-ai-hr（HR）、agentic-ai-sales（レベニュー）、agentic-ai-analysis（データ分析）の 3 プロジェクトの設計から共通パターンを抽象化し、任意のビジネスドメインに適用できる形に一般化した。

## 1. フレームワークの全体像

Neural Organization 準拠の Agentic AI は、以下の 8 つの設計ステップで構築される。

```
Step 1: サイクル定義
  「どの業務サイクルの全工程を最適化するか」

Step 2: 5 層マッピング
  「サイクルの各工程を L0-L4 にどう配置するか」

Step 3: エージェント設計
  「各工程をどのエージェントが担うか」

Step 4: Knowledge Layer 設計
  「ドメイン固有の普遍的フレームワークは何か」

Step 5: Memory 設計
  「3 構造（長期・作業・評価）に何を蓄積するか」

Step 6: Governance 設計
  「HITL ゲートと Trust Score をどう適用するか」

Step 7: 共有記憶インターフェース設計
  「他の Agentic AI と何を読み書きするか」

Step 8: 実装ロードマップ
  「どの順序で段階的に構築するか」
```

**各ステップに対して、テンプレートと 3 プロジェクトの具体例を示す。**

## 2. Step 1: サイクル定義

### 2.1 原則

> **ツールではなくサイクルを設計する。**
> v1 の各プロジェクトに共通していた問題は、業務サイクルの「一工程のツール」にとどまっていたこと。v2 では業務サイクルの全工程をカバーする。

### 2.2 サイクル定義のテンプレート

```
1. ドメインのビジネスサイクルを定義する
   - 工程を洗い出す（最低 4 工程）
   - 工程間の依存関係を明確にする
   - サイクルが「閉じる」こと（最後の工程が最初に戻る）を確認する

2. 各工程の最適化目標を定義する
   - 何を最大化/最小化したいか
   - 測定可能な KPI を設定する

3. v1 の機能をサイクルの中に再配置する
   - 既存機能は消えない。サイクルの一工程として位置づけ直す
```

### 2.3 3 プロジェクトの具体例

| プロジェクト | サイクル名 | 工程数 | v1 の位置づけ |
|---|---|---|---|
| **agentic-ai-hr** | HR ライフサイクル | 4 工程（設計→導入→運用→進化） | v1 の 5 段階ワークフロー = 設計工程 |
| **agentic-ai-sales** | レベニューサイクル | 7 工程（リード→商談化→提案→受注→CS→アップセル→FB） | v1 の SPIN 生成 = 提案工程 |
| **agentic-ai-analysis** | データ分析サイクル | 9 工程（問い→データ→品質→探索→検証→インサイト→レポート→意思決定→効果測定） | 新規プロジェクト |

### 2.4 共通パターン

3 プロジェクトに共通するサイクルの構造的特徴：

```
┌─────────────────────────────────────────────────┐
│             共通サイクル構造                        │
│                                                   │
│  [入力受付] → [理解・構造化] → [分析・推論]        │
│      → [実行・出力] → [効果測定 → 入力に戻る]     │
│                                                   │
│  ※ 全サイクルに Reflection（効果測定→学習）が存在  │
│  ※ サイクルの「閉じ」が学習ループを駆動する       │
└─────────────────────────────────────────────────┘
```

## 3. Step 2: 5 層マッピング

### 3.1 原則

> **サイクルの各工程を 5 層に配置する。すべての層がカバーされていること。特に L4 Reflection は必須。**

### 3.2 マッピングテンプレート

```
L0 Perception（知覚）:
  - サイクルの入力データは何か
  - どのデータソースに接続するか
  - 他の Agentic AI からの入力は何か

L1 Understanding（理解）:
  - 入力データをどう構造化するか
  - パターン・異常・トレンドをどう検知するか
  - ドメイン固有のメタデータは何か

L2 Reasoning（推論）:
  - 構造化されたデータからどんな判断を導出するか
  - 選択肢間のトレードオフをどう評価するか
  - Purpose（経営方針）とどう接続するか

L3 Execution（実行）:
  - どんなアーティファクトを生成するか
  - 品質チェック（4 段階）をどう適用するか
  - デリバリー先とチャネルは何か

L4 Reflection（内省）:
  - アウトプットの効果をどう測定するか
  - HITL 修正パターンから何を学ぶか
  - 各層のどの永続状態を更新するか
```

### 3.3 3 プロジェクトの L4 Reflection 比較

| プロジェクト | Reflection Agent | 学習する対象 | Memory 更新先 |
|---|---|---|---|
| **agentic-ai-hr** | ReflectionAgent | HITL 修正パターン → 暗黙の価値観、運用結果 → 制度改定提案 | 長期: 組織の価値観、評価: 制度 KPI |
| **agentic-ai-sales** | RevenueCycleReflectionAgent | Win/Loss パターン、予測モデルの精度、HITL 修正 | 長期: 勝ち/負け辞書、評価: 受注率推移 |
| **agentic-ai-analysis** | AnalysisCycleReflectionAgent | 分析精度、手法有効性、品質ルール精度、問いの質 | 長期: 分析パターン + 品質ルール、評価: 精度推移 |

**共通パターン**: Reflection Agent は (1) アウトプットの効果測定、(2) HITL 修正パターンの分析、(3) 予測/モデルの精度追跡の 3 つを必ず行う。

## 4. Step 3: エージェント設計

### 4.1 原則

> **1 エージェント = 1 つの認知変換。Reflection Agent は必ず 1 つ以上存在する。**

### 4.2 エージェント設計テンプレート

各エージェントは以下のテンプレートで定義する。

```
#### Agent N: [DomainConcept]Agent（[対応する工程] — [対応する層]）

役割: [1 文で役割を定義]

入力:
  ● [前段エージェントの出力]
  ● [外部データ]
  ● [連携] [他 Agentic AI からのデータ]
  ● Memory: [参照する記憶]

処理:
  1. [処理ステップ 1]:
     [具体的なロジックまたは数式]
  2. [処理ステップ 2]:
     ...

出力:
  ● [出力の名前]:
    - [出力の構成要素]

HITL Gate: [Gate ID]
  [承認者] の問い: 「[承認時の判断ポイント]」
  ※ [Trust Score に応じた自動パスの条件]
```

### 4.3 命名規則

| 要素 | 規則 | 例 |
|---|---|---|
| Agent 名 | `[DomainConcept]Agent` | `QuestionArchitectAgent`, `LeadIntelligenceAgent` |
| 出力名 | `[出力の性質を表す名詞]` | `AnalysisDesign`, `DealStrategy`, `GradingSystem` |
| Gate ID | `[プロジェクト略称]-[連番]` | `DA-001`, `RC-003`, `MG-002` |

### 4.4 エージェント数の目安

| サイクル工程数 | エージェント数の目安 | 根拠 |
|---|---|---|
| 4 工程 | 5-7 エージェント | 各工程 1-2 + Reflection 1 |
| 7 工程 | 8-12 エージェント | 各工程 1-2 + Reflection 1 + 横断支援 1-2 |
| 9 工程 | 7-10 エージェント | 類似工程を統合 + Reflection 1 |

### 4.5 Reflection Agent の設計パターン

3 プロジェクトの Reflection Agent に共通する**5 つの学習機能**：

```
1. アウトプット効果測定
   - 予測効果 vs 実際の効果の比較
   - 効果の乖離が大きい場合、原因層を特定

2. HITL 修正パターン分析
   - 人間の修正行為から暗黙の価値観・品質基準を抽出
   - 「毎回同じ修正をしている」→ 自動化候補として学習

3. 予測/モデルの精度追跡
   - ドメイン固有の予測モデルの精度を時系列で追跡
   - 精度低下時にモデルの再学習をトリガー

4. パターン辞書の更新
   - 成功パターン/失敗パターンの自動抽出
   - 長期記憶への蓄積

5. サイクル全体のメタ分析
   - 「N サイクル連続で同じ問題」→ 構造的問題の示唆
   - 「アクション完了率が低い」→ アクション設計の問題
   - 「Governor が毎回要因を追加」→ Perception の盲点
```

## 5. Step 4: Knowledge Layer 設計

### 5.1 原則

> **ドメイン固有の普遍的フレームワークを構造化データとして格納し、エージェントが参照する。企業固有の値はパラメータ化して外出しする。**

### 5.2 Knowledge Layer 設計テンプレート

```
1. ドメインの普遍的フレームワークを特定する
   - どの企業にも適用できる構造・原則・パターン
   - 業界のベストプラクティス

2. 企業固有の要素を分離する
   - 金額、日程、固有名詞 → パラメータ化
   - 企業の方針・文化 → Purpose Interface で Governor が設定

3. 構造化データ形式で格納する
   - YAML / JSON で knowledge/ ディレクトリに格納
   - バージョン管理 + 変更履歴

4. エージェントからの参照パスを定義する
   - どのエージェントがどのフレームワークを参照するか
```

### 5.3 3 プロジェクトの Knowledge Layer

| プロジェクト | 普遍的フレームワーク | 企業固有（パラメータ化） |
|---|---|---|
| **agentic-ai-hr** | コンピテンシー・トリアド、スコープベース等級（8 段階）、コンピテンシー進化マトリクス、3 ゲート評価、報酬の 3 つの究極の問い、4 フェーズ運用体系 | 具体的年俸テーブル、役職手当額、評価サイクル日程 |
| **agentic-ai-sales** | SPIN / BANT / MEDDIC フレームワーク、ICP スコアリング重み、顧客健全性スコア計算式、アップセルの 5 次元（タイミング×場所×商材×価格×表現） | 自社商材情報、価格テーブル、ブランドガイドライン |
| **agentic-ai-analysis** | 分析パターンライブラリ（手法×問いの種類の対応表）、データ品質ルール辞書、統計検定の選択フレームワーク | データソース接続情報、社内用語定義 |

### 5.4 Purpose Interface パターン

3 プロジェクトに共通する、Governor に「意志」を問う設計パターン：

| プロジェクト | Purpose Interface | 問う内容 |
|---|---|---|
| **agentic-ai-hr** | 3 つの究極の問い | Q1: 能力 vs 職務、Q2: 蓄積型 vs 洗い替え型、Q3: 階級 vs 機能 |
| **agentic-ai-sales** | GTM 戦略定義 | ICP 定義、価値提案、チャネル戦略 |
| **agentic-ai-analysis** | 分析優先度方針 | どの KPI に分析リソースを集中するか |

**共通パターン**: 指示（Instruction）ではなく意志（Intent）を受け取る。Governor が「方針」を設定し、エージェントが「詳細」を自律的に導出する。

## 6. Step 5: Memory 設計

### 6.1 原則

> **3 構造（長期・作業・評価）は不変。内容はドメイン固有。**

### 6.2 Memory 設計テンプレート

```
長期記憶（Institutional Memory）:
  用途: 組織が蓄積すべき永続的知見
  技術: PostgreSQL + ベクトル DB
  内容:
    - [ドメイン固有のパターン辞書]
    - [フレームワーク/Knowledge Layer]
    - [暗黙の価値観（HITL 修正から学習）]

作業記憶（Working Memory）:
  用途: 進行中のタスクの状態
  技術: Redis
  内容:
    - [進行中のワークフロー状態]
    - [エージェント間の中間出力]
    - [HITL 待ち状態]
  TTL: 7 日 → セッション終了時に長期記憶への昇格判定

評価記憶（Evaluative Memory）:
  用途: KPI 推移と効果測定
  技術: PostgreSQL（時系列）
  内容:
    - [ドメイン固有の KPI 推移]
    - [予測精度推移]
    - [Trust Score 推移]
```

### 6.3 3 プロジェクトの Memory 内容比較

| 記憶構造 | agentic-ai-hr | agentic-ai-sales | agentic-ai-analysis |
|---|---|---|---|
| **長期** | 6 フレームワーク + 組織の暗黙の価値観 + 制度設計パターン | 勝ち/負けパターン辞書 + ICP + 顧客エンティティ | 分析パターンライブラリ + 品質ルール辞書 + データカタログ |
| **作業** | 進行中の設計セッション + HITL 待ち | 進行中の商談文脈 + 提案書の生成状態 | 進行中の分析コンテキスト + 中間データセット |
| **評価** | 離職率推移 + 評価分布 + 昇格率 | 受注率推移 + 解約率 + チャネル ROAS | 予測精度推移 + 分析→アクション転換率 |

### 6.4 共通 PostgreSQL スキーマパターン

全プロジェクトに共通する HITL 修正履歴テーブル：

```sql
CREATE TABLE hitl_modifications (
    id UUID PRIMARY KEY,
    context_id UUID NOT NULL,         -- 企業ID / 分析ID 等
    workflow_id UUID NOT NULL,
    gate_id VARCHAR(50) NOT NULL,
    agent_type VARCHAR(50) NOT NULL,
    original_output JSONB NOT NULL,
    modified_output JSONB,
    modification_type VARCHAR(50) NOT NULL,  -- 'approved' | 'modified' | 'rejected'
    feedback TEXT,
    decided_by VARCHAR(100),
    decided_at TIMESTAMP NOT NULL
);
```

## 7. Step 6: Governance 設計

### 7.1 原則

> **二重フィルタリング原則: システム自己評価 → Governance Gate。Trust Score で動的にゲートを制御する。**

### 7.2 HITL ゲート設計テンプレート

```
| Gate ID | ゲート名 | エージェント | 承認者 | 不可逆性 |
|---|---|---|---|---|
| [略称]-001 | [ゲート名] | Agent N | [Governor/Custodian/Sensemaker] | [低/中/高] |
```

**不可逆性の判断基準**：

| レベル | 基準 | 例 |
|---|---|---|
| **低** | 社内のみで影響。修正が容易 | 分析設計、戦略立案、制度設計 |
| **中** | 対外的な影響あり。撤回に一定のコスト | レポート配信、アウトリーチ、制度説明 |
| **高** | 対外的な影響大。撤回困難 | 価格コミット、提案資料提示、人事通達 |

### 7.3 Trust Score 設計テンプレート

4 因子の重み（0.30 + 0.35 + 0.20 + 0.15 = 1.00）は全プロジェクト共通。測定方法はドメイン固有。

```
Trust Score = 成功率 × 0.30 + 結果品質 × 0.35 + 整合性 × 0.20 + リスク管理 × 0.15

成功率: [HITL で修正なしに承認された割合]
結果品質: [ドメイン固有の品質指標]
整合性: [Purpose との整合度]
リスク管理: [リスクを適切にエスカレーションした割合]
```

### 7.4 動的ゲート制御の共通パターン

4 レベルのゲート制御は全プロジェクトで同じ構造を持つ：

| Trust Score | 自動パスの対象 | 承認が必要な対象 |
|---|---|---|
| Cold Start (0.00-0.39) | なし | 全ゲート |
| Learning (0.40-0.69) | 定型的・社内完結のゲート | 判断を伴うゲート + 対外的ゲート |
| Trusted (0.70-0.89) | 大半のゲート | 高不可逆性ゲート + 価値判断ゲート |
| Highly Trusted (0.90+) | ほぼ全ゲート | 前例のない判断のみ |

### 7.5 3 プロジェクトの承認者パターン

| 承認者 | 判断の性質 | 対応するゲートの例 |
|---|---|---|
| **Governor**（経営層） | 戦略的判断、方針判断 | アクション計画承認、報酬制度承認、価格戦略承認 |
| **Sensemaker** | 定性的補足、現場感の注入 | 差異分析の要因確認、モデル検証の実感チェック |
| **Custodian**（守護者） | 安全性、公平性、法令遵守 | レポート出力の確認、データ品質ルール承認、制度文書確認 |

## 8. Step 7: 共有記憶インターフェース設計

### 8.1 原則

> **各 Agentic AI は独立して動作可能であること。共有記憶はオプショナルな接続であり、接続により価値が増幅する。**

### 8.2 インターフェース設計テンプレート

```
書き込み（他プロジェクトに提供するデータ）:
| データ | 消費者 | 活用例 |
|---|---|---|
| [自プロジェクトの出力データ] | [消費するプロジェクト] | [具体的な活用方法] |

読み出し（他プロジェクトから取得するデータ）:
| データ | 提供元 | 活用例 |
|---|---|---|
| [取得するデータ] | [提供するプロジェクト] | [自プロジェクトでの活用方法] |

REST API:
  GET  /api/v2/[resource]/{context_id}   [読み出しエンドポイント]
  POST /api/v2/events/[event-name]        [イベント発行エンドポイント]
```

### 8.3 接続パターンの全体マップ

4 つの Agentic AI の接続関係：

```
┌────────────────────────────────────────────────────────────────┐
│                       共有記憶基盤                                │
│                                                                │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐               │
│  │ neumann  │←───→│agentic-  │←───→│agentic-  │               │
│  │(経営管理)│     │ai-hr     │     │ai-sales  │               │
│  └────┬─────┘     └────┬─────┘     └────┬─────┘               │
│       │                │                │                      │
│       │    ┌───────────┴────────────────┘                      │
│       │    │                                                   │
│       ▼    ▼                                                   │
│  ┌──────────────┐                                              │
│  │ agentic-ai-  │ ← 全プロジェクトの深掘り分析を担当           │
│  │ analysis     │ → 分析結果を各プロジェクトに返却              │
│  │(データ分析)  │                                              │
│  └──────────────┘                                              │
│       ▲                                                        │
│       │                                                        │
│  ┌──────────────┐                                              │
│  │ ai-executive │ → 市場データの供給                            │
│  │ -dashboard   │ ← 全プロジェクトの KPI を統合                 │
│  └──────────────┘                                              │
└────────────────────────────────────────────────────────────────┘
```

**agentic-ai-analysis の特殊な位置づけ**: 他の 3 プロジェクトが「自分のサイクルの中で簡易な分析を行う」のに対し、agentic-ai-analysis は「統計的に厳密な深掘り分析」を提供する。他のプロジェクトの Reflection Agent が検出した「もっと深く調べるべき問い」が、agentic-ai-analysis への自動リクエストとなる。

### 8.4 自動連携パターン

プロジェクト間の自動連携には 3 つのパターンがある：

| パターン | トリガー | 例 |
|---|---|---|
| **異常検知→分析依頼** | あるプロジェクトの Agent が異常を検知 | neumann の KPI 異常 → analysis に分析リクエスト |
| **データ参照** | あるプロジェクトの Agent が他プロジェクトのデータを参照 | agentic-ai-sales の ProposalAgent が agentic-ai-hr の等級データを参照 |
| **Reflection 連鎖** | Reflection Agent の学習が他プロジェクトに波及 | neumann の Reflection → agentic-ai-sales の戦略更新 → agentic-ai-hr の制度改定 |

## 9. Step 8: 実装ロードマップ

### 9.1 原則

> **コアから外側へ。Reflection は後回しにしない。**

### 9.2 ロードマップ設計テンプレート

```
Phase 1: コアサイクルの最小構成（3-4 週）
  - サイクルの中核工程を担うエージェント 2-3 体
  - DB 導入（Memory の永続化基盤）
  - v1 の既存機能がここに再配置される

Phase 2: 分析/推論の強化（3-4 週）
  - L2 Reasoning のエージェント
  - Knowledge Layer の構築
  - Memory の長期記憶実装

Phase 3: 実行とインサイト（2-3 週）
  - L3 Execution のエージェント
  - 品質チェック（4 段階）の実装
  - アウトプットの多様化

Phase 4: Reflection（2-3 週）
  - Reflection Agent の実装
  - 効果測定の仕組み
  - パターン辞書の自動更新

Phase 5: 共有記憶基盤接続（2 週）
  - 他プロジェクトとの接続
  - 自動連携パターンの実装
```

### 9.3 Phase 順序の根拠

```
                    Phase 1        Phase 2        Phase 3
                    コア           分析強化       実行
                   ┌─────┐       ┌─────┐       ┌─────┐
                   │ L0  │       │     │       │     │
                   │ L1  │  →    │ L2  │  →    │ L3  │
                   │(基礎)│      │(推論)│      │(出力)│
                   └─────┘       └─────┘       └─────┘
                                                  │
                    Phase 4                        │
                    Reflection                     │
                   ┌─────┐                         │
                   │ L4  │  ←──────────────────────┘
                   │(学習)│    L3 の出力がないと Reflection できない
                   └─────┘
                      │
                    Phase 5
                    接続
                   ┌─────┐
                   │共有  │
                   │記憶  │
                   └─────┘
```

**Reflection を Phase 4 に置く理由**: L4 Reflection は L3 Execution の出力結果を観測して学習する。出力がなければ学習もできない。ただし、**Phase 4 を省略しない**ことが極めて重要。Reflection のないシステムは「生成して終わり」の v1 に逆戻りする。

## 10. 品質チェックリスト

新規 Agentic AI の設計が Neural Organization に準拠しているかを確認するチェックリスト。

### 10.1 必須要件（全項目が Yes であること）

| # | 確認項目 | 確認の観点 |
|---|---|---|
| C1 | サイクルが閉じているか | 最後の工程（効果測定）が最初の工程に戻るか |
| C2 | 5 層すべてがカバーされているか | L0-L4 に対応するエージェントが存在するか |
| C3 | Reflection Agent が存在するか | 学習ループが設計されているか |
| C4 | Memory の 3 構造が定義されているか | 長期・作業・評価のそれぞれに内容が定義されているか |
| C5 | HITL ゲートが設計されているか | 人間の承認フローが存在するか |
| C6 | Trust Score が定義されているか | 4 因子のドメイン固有測定方法が定義されているか |
| C7 | 動的ゲート制御が設計されているか | Trust Score に応じてゲートが動的に変化するか |
| C8 | 共有記憶インターフェースが定義されているか | 読み書きするデータが明記されているか |
| C9 | Purpose Interface が存在するか | Governor が「意志」を注入する界面があるか |
| C10 | 品質チェック（4 段階）が適用されているか | L3 Execution の出力が品質検査を通過するか |

### 10.2 設計原則への適合確認

| 原則 | 確認の問い |
|---|---|
| **Intent over Instruction** | エージェントは指示ではなく意志を受け取っているか？ Governor に「方針」を問い、「詳細」は自律的に導出しているか？ |
| **Governance not Control** | 人間は逐一の操作ではなく、原則の設定と例外の処理を行っているか？ |
| **Ambient Presence** | システムは「開いて使う」ものではなく、サイクルに沿って自律的に動作しているか？ |
| **Radical Transparency** | 全エージェントの判断根拠が開示されているか？ 設計根拠が明示されているか？ |
| **Agency Preservation** | 人間がいつでもシステムの判断を覆せるか？ HITL ゲートが適切に配置されているか？ |
| **Mutual Evolution** | HITL 修正が学習データとして蓄積されるか？ システムの学習が人間の判断能力を高めているか？ |

## 11. 適用ガイドライン

### 11.1 新規ドメインへの適用手順

```
1. ドメインのビジネスサイクルを定義する（Step 1）
   - ドメインエキスパートへのヒアリング
   - 現行プロセスの棚卸し
   - サイクルの閉じ方（効果測定→次のサイクル）を確認

2. 既存のツール/システムを棚卸しする
   - v1 に相当する既存機能を特定
   - サイクルのどの工程に位置づけられるか

3. 5 層マッピングとエージェント設計（Step 2-3）
   - 各工程を L0-L4 に配置
   - エージェントを定義（テンプレート適用）

4. Knowledge Layer を構築する（Step 4）
   - ドメイン固有のフレームワーク/ベストプラクティスを特定
   - 普遍的 vs 企業固有を分離

5. Memory と Governance を設計する（Step 5-6）
   - 3 構造の内容を定義
   - HITL ゲートと Trust Score を設計

6. 共有記憶インターフェースを設計する（Step 7）
   - 既存の Agentic AI との接続点を特定

7. 実装ロードマップを策定する（Step 8）
   - コアから外側へ
   - Reflection を省略しない

8. 品質チェックリストで検証する（§10）
   - 全必須要件が Yes であること
   - 6 原則への適合を確認
```

### 11.2 このフレームワークが適用可能なドメインの例

| ドメイン | サイクル | 工程の例 |
|---|---|---|
| プロダクト開発 | 開発サイクル | 要件定義→設計→実装→テスト→リリース→ユーザーフィードバック→改善 |
| 法務 | 契約管理サイクル | 契約要件→ドラフト→レビュー→交渉→締結→履行管理→更新/解約 |
| 財務 | 財務管理サイクル | 予算策定→執行→モニタリング→決算→分析→次期計画 |
| マーケティング | マーケサイクル | 戦略策定→キャンペーン設計→実行→効果測定→最適化 |
| カスタマーサポート | サポートサイクル | チケット受付→分類→解決→エスカレーション→ナレッジ化→予防策 |
| 採用 | 採用サイクル | 要件定義→母集団形成→選考→オファー→オンボーディング→定着評価 |
